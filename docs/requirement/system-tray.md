# 系统托盘需求说明

| 项目 | 内容 |
| --- | --- |
| 功能名称 | 系统托盘（System Tray） |
| 产品 | SpaceAssistant |
| 状态 | 待实现 |
| 关联模块 | `electron/main.ts`、`electron/menu.ts`、`electron/windowRef.ts` |

## 1. 背景与目标

SpaceAssistant 当前在用户关闭主窗口后会直接退出应用（Windows / Linux 上由 `window-all-closed` 触发 `app.quit()`）。用户希望应用能在后台常驻，通过系统托盘快速唤起，并在无界面时降低资源占用。

**目标：**

1. 应用启动后自动在系统托盘显示图标。
2. 用户关闭主窗口时，应用不退出，托盘图标保持可见。
3. 主窗口关闭后，应用进入低占用后台状态（相比窗口打开时更省资源）。
4. 用户可通过托盘图标重新打开主窗口，或从托盘菜单彻底退出。

## 2. 现状（实现前）

| 行为 | 当前实现 |
| --- | --- |
| 启动 | `app.whenReady()` 后创建主窗口，无托盘 |
| 关闭主窗口（Windows / Linux） | 触发 `window-all-closed` → `app.quit()`，进程结束 |
| 关闭主窗口（macOS） | 窗口关闭但进程保留（已有 `activate` 重建窗口逻辑） |
| 菜单退出 | Windows：`文件 → 退出`（Ctrl+Q）调用 `app.quit()` |
| 托盘 | 无 |

## 3. 用户故事

1. **后台常驻**：作为日常助手用户，我希望关闭主窗口后应用仍在托盘运行，以便稍后继续对话而不必重新启动。
2. **快速唤起**：我希望双击托盘图标即可恢复主窗口。
3. **明确退出**：我希望在托盘右键菜单中明确选择「退出」，以彻底结束应用并释放资源。
4. **资源友好**：主窗口关闭后，我不希望 GPU / 内存仍维持与前台窗口相同的占用水平。

## 4. 功能需求

### 4.1 托盘图标创建

| ID | 需求 | 优先级 |
| --- | --- | --- |
| TRAY-01 | 应用完成 `app.whenReady()` 且主进程初始化就绪后，创建系统托盘图标 | P0 |
| TRAY-02 | 托盘图标在应用整个生命周期内仅创建一次（单例），避免重复图标 | P0 |
| TRAY-03 | 托盘图标 tooltip（悬停提示）显示应用名称：`SpaceAssistant` | P1 |
| TRAY-04 | 托盘图标资源按平台区分：<br>• Windows：`.ico`<br>• macOS：`.png` 或 `.icns`（遵循 Electron / electron-builder 惯例）<br>• Linux：`.png` | P0 |
| TRAY-05 | 开发模式（`app.isPackaged === false`）与打包模式均显示托盘，行为一致 | P0 |

### 4.2 主窗口关闭行为

| ID | 需求 | 优先级 |
| --- | --- | --- |
| TRAY-10 | 用户点击主窗口关闭按钮（或等效关闭操作）时，**不退出应用**，仅关闭 / 隐藏主窗口界面 | P0 |
| TRAY-11 | 主窗口关闭后，托盘图标必须仍然可见 | P0 |
| TRAY-12 | 主窗口关闭后，应用进入后台模式，资源占用应低于窗口前台展示时（见 §5.2） | P0 |
| TRAY-13 | 修改现有 `window-all-closed` 逻辑：在启用托盘后，**不得**因「所有窗口已关闭」而自动 `app.quit()`（Windows / Linux 均需调整） | P0 |
| TRAY-14 | 若主窗口已被关闭 / 销毁，再次打开时应恢复用户可继续使用的界面（会话状态由现有 Redux + 数据库机制保证，不应因关窗而清空） | P0 |

**推荐实现策略（供开发参考，写入验收标准）：**

- 默认方案：拦截 `close` 事件，`preventDefault()` 后调用 `win.hide()`，保留 `BrowserWindow` 实例以便快速恢复。
- 若实测内存仍偏高，可在 `hide` 后对 renderer 启用更强节流，或在隐藏一段时间后销毁窗口、下次从托盘重建（需保证重建逻辑与 `createMainWindow()` 一致）。

### 4.3 托盘交互 — 打开主窗口

| ID | 需求 | 优先级 |
| --- | --- | --- |
| TRAY-20 | **双击**托盘图标：打开 / 显示主窗口 | P0 |
| TRAY-21 | 若主窗口已存在但被隐藏：显示并聚焦（`show()` + `focus()`） | P0 |
| TRAY-22 | 若主窗口已被销毁：按现有 `createMainWindow()` 流程重新创建并加载 | P0 |
| TRAY-23 | 打开主窗口后，窗口应置于前台并获得焦点（在 OS 允许范围内） | P0 |

### 4.4 托盘右键菜单

| ID | 需求 | 优先级 |
| --- | --- | --- |
| TRAY-30 | 右键点击托盘图标弹出上下文菜单 | P0 |
| TRAY-31 | 菜单第一项：**打开主窗口** — 行为与双击托盘一致（TRAY-20 ~ TRAY-23） | P0 |
| TRAY-32 | 在第一项与最后一项之间插入**分隔线**（`type: 'separator'`） | P0 |
| TRAY-33 | 菜单最后一项：**退出** — 彻底退出应用 | P0 |
| TRAY-34 | 菜单文案使用简体中文（与产品 UI 语言一致） | P0 |

**菜单结构示意：**

```
打开主窗口
──────────
退出
```

### 4.5 退出行为

| ID | 需求 | 优先级 |
| --- | --- | --- |
| TRAY-40 | 托盘菜单「退出」、应用菜单「文件 → 退出」（Windows Ctrl+Q）、macOS 标准 Quit 均应触发**同一套**退出流程 | P0 |
| TRAY-41 | 退出前执行现有清理：`appDb?.flushSave()`（已有 `before-quit` 钩子） | P0 |
| TRAY-42 | 退出时销毁托盘图标，避免托盘残留「幽灵图标」 | P0 |
| TRAY-43 | 退出时销毁所有 `BrowserWindow`，随后调用 `app.quit()` | P0 |
| TRAY-44 | 退出后进程完全结束，不保留后台驻留 | P0 |

**实现注意：** 需区分「用户关窗（后台驻留）」与「用户主动退出」。可通过 `app.isQuitting` 标志位，在用户选择退出时设为 `true`，允许 `close` 事件正常关闭窗口并触发 quit。

## 5. 非功能需求

### 5.1 平台与兼容性

| ID | 需求 |
| --- | --- |
| NFR-01 | 必须支持 Windows 10+（主要开发与测试平台） |
| NFR-02 | 应支持 macOS、Linux（Electron 跨平台目标）；macOS 需与现有 `activate` 行为协调 |
| NFR-03 | Linux 下若桌面环境无系统托盘（极少数），应优雅降级：关窗行为可回退为直接退出，或记录日志；具体策略实现阶段确认 |

### 5.2 资源占用

| ID | 需求 |
| --- | --- |
| NFR-10 | 主窗口隐藏后，renderer 进程应受益于 Electron 默认后台节流（`backgroundThrottling`） |
| NFR-11 | 不要求实现精确的内存阈值，但验收时需能观察到：关窗后 CPU / GPU 占用明显低于窗口前台展示时 |
| NFR-12 | 托盘本身仅保留主进程中的 `Tray` 实例与菜单，不应额外启动 renderer |

### 5.3 可靠性与生命周期

| ID | 需求 |
| --- | --- |
| NFR-20 | 托盘模块初始化失败（如图标文件缺失）时，应记录错误日志；Windows 上可降级为无托盘但保留原有关窗即退出行为，避免应用无法使用 |
| NFR-21 | 应用仅允许一个托盘图标实例 |
| NFR-22 | 二次启动（若未来支持单实例锁）不应重复创建托盘；当前版本至少保证单次进程内不重复 |

## 6. 与现有功能的关系

### 6.1 应用菜单

| 场景 | 期望行为 |
| --- | --- |
| Windows `文件 → 退出` | 彻底退出（TRAY-40） |
| macOS `SpaceAssistant → 退出` | 彻底退出 |
| macOS 点击窗口关闭按钮 | 隐藏窗口，不退出（与 Windows 对齐） |
| `查看 → 开发者工具` | 仅在有主窗口时可用；无窗口时菜单项可保持现有逻辑（无窗口则 no-op） |

### 6.2 数据与会话

- 关窗隐藏 **不应** 触发数据库写入或会话丢失（除非用户主动退出）。
- 现有 `SessionBackupManager`、流式聊天 IPC 在后台的行为：实现阶段需确认进行中的流式响应在关窗后是否继续；**本需求不强制暂停流式任务**，但若关窗导致 renderer 卸载，需在技术设计中说明影响。

### 6.3 代码落点（实现指引）

| 职责 | 建议位置 |
| --- | --- |
| 托盘创建、菜单、事件 | 新建 `electron/tray.ts`（或等价模块） |
| 关窗拦截与 `isQuitting` | `electron/main.ts` 或 `electron/tray.ts` |
| 窗口显示 / 隐藏 / 重建 | 复用 `createMainWindow()`、`getMainWindow()` / `setMainWindow()` |
| 托盘图标资源 | `build/` 或 `resources/` 目录（与 electron-builder 配置对齐） |

## 8. 验收标准

以下场景全部通过即视为功能完成：

1. **启动**：运行 `npm run dev` 或打包应用，系统托盘出现 SpaceAssistant 图标。
2. **关窗驻留**：打开主窗口后点击关闭，主窗口消失，托盘图标仍在，进程未退出（任务管理器 / 活动监视器中可验证）。
3. **资源**：关窗后等待数秒，CPU 占用应低于窗口在前台时（主观对比或简单采样即可）。
4. **双击唤起**：双击托盘图标，主窗口重新出现且可正常操作（会话列表、聊天等功能正常）。
5. **菜单唤起**：右键 →「打开主窗口」，行为与双击一致。
6. **分隔线**：右键菜单中「打开主窗口」与「退出」之间存在分隔线。
7. **彻底退出**：右键 →「退出」，托盘图标消失，进程结束；再次启动应用为全新进程。
8. **菜单退出一致**：Windows 下 Ctrl+Q 或菜单退出，与托盘「退出」行为一致。
9. **重复打开**：连续多次关窗 → 托盘打开，窗口均能正常显示，无多窗口堆叠（仅一个主窗口）。
10. **打包验证**：Windows 安装包（`npm run pack:win`）环境下行为与开发模式一致。
