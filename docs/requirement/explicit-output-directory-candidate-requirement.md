# Agent 工作产物管理：项目变更、工作包与会话草稿区 — MVP 细化需求

## 1. 文档定位

本文重构现有“文件写入目录规范”功能。它不再试图按扩展名重排所有新文件，而是管理 Agent 在不同任务中产生的工作产物，解决以下问题：

- 软件开发任务中，正式源码、测试、配置和 migration 不能被误藏；
- 分析任务中，报告、SQL、脚本、数据口径需要能被用户核对和提交；
- 写作任务中，草稿、研究资料、摘录、引用与成稿需要持续可见、可迭代；
- 真正的一次性缓存、下载分片和失败尝试不应散落在项目根目录；
- 用户明确指定的输出文件或目录必须按字面语义生效。

本期采用三类用户可理解的容器：

```text
项目变更集      → 正式修改项目
工作包          → 围绕一个交付结果持续工作、核对和交付
会话草稿区      → 可删除的一次性执行材料
```

“目录规范”的目的由“整理扩展名”调整为：让每一份 Agent 文件有正确的归属、可解释的路径和符合用户预期的保留方式。

## 2. 核心原则

1. **所有用户逐项指定的路径按字面生效**：无论它是项目文件、主成果、支撑材料还是资料附件，合法的精确文件或目录路径均优先于产品目录规则；不得追加扩展名分类目录或强制移入材料目录。
2. **工作成果优先于文件格式**：`.py`、`.sql`、`.csv`、`.md` 仅表示格式，不能单独决定文件是正式项目文件、交付材料还是临时文件。
3. **项目任务保持项目正确性**：实现、修复、重构、补测试、更新配置、migration 等任务的新正式文件写入项目结构。
4. **可核对材料不应被隐藏**：支撑报告结论、写作论述或数据口径的 SQL、脚本、资料和摘录属于工作包，默认对用户可见。
5. **只有纯临时材料进入草稿区**：缓存、下载分片、无价值调试输出、失败尝试和不可复用的执行输入才进入会话草稿区；未建立工作包的待决定研究资料是唯一受独立保护的暂存例外。
6. **用户掌握保留决策**：资料是否随最终稿保留、某个草稿文件是否提升为支撑材料或项目文件，必须由用户可见、可操作。
7. **安全是底线，不是分类规则**：所有项目文件、工作包文件和草稿文件均在 `workDir` 内；路径安全校验不替代文件归属判断。
8. **不重复确认**：合法、唯一、明确的用户输出位置直接采用；当前会话已关联成果的持续修改不重复确认。仅在位置缺失、歧义、首次可能覆盖无关文件或安全问题时询问。
9. **成果关联而非全会话误继承**：会话可以记住已识别成果及其路径，但新成果不得自动继承另一成果的输出位置，除非用户明确设为后续默认。

## 3. 三类容器

| 容器 | 解决的问题 | 典型内容 | 默认位置 |
|---|---|---|---|
| 项目变更集 | 改变软件项目最终状态 | 源码、测试、配置、migration | 项目已有结构 |
| 工作包 | 交付、审阅、写作与研究的可核对成果 | 主文档、草稿、SQL、脚本、引用笔记、资料 | 用户指定输出位置及其同级材料目录 |
| 会话草稿区 | 不值得长期保留的执行材料 | 缓存、分片、失败调试、一次性输入 | `workDir/.spaceassistant/runs/<session-id>/` |

一个会话可同时拥有多个项目变更和多个工作包。例如，用户要求“实现功能并把设计评审写到 `docs/review`”时，源码属于项目变更，评审文档与口径 SQL 属于工作包，临时验证脚本属于草稿区。

## 4. 文件归属模型

不再使用单一“中间文件”概念。每个新文件由“容器 + 材料类型”共同描述：主成果、支撑材料、研究资料或草稿文件；主成果还可处于草稿、定稿或工作中状态。具体数据结构由后续技术设计确定。

| 归属 | 判定条件 | 例子 |
|---|---|---|
| `project` | 构成项目最终状态，或用户明确要求改项目 | `src/auth.ts`、`tests/auth.test.ts`、migration |
| `package + primary` | 用户要阅读、迭代或交付的主成果 | `review.md`、`draft.md`、最终 CSV |
| `package + supporting` | 用户可能用来核对、复用或随主成果提交的材料 | `analysis.sql`、`analyze.py`、outline、citation-notes |
| `package + reference` | 用户要求获取、可能需要核对的原始资料及其来源信息 | 行业报告 PDF、网页快照、来源索引 |
| `scratch` | 不构成项目或工作包价值的短期材料 | 下载分片、失败输出、临时缓存 |

Agent 可以声明建议归属和用途，但最终归属由用户意图、项目最终状态和安全规则决定：

```text
用户对该文件的精确路径或明确保留要求
  > 用户是否要求该文件成为当前项目最终状态的一部分
    > 项目变更目标与已有项目结构
    > 与工作包主成果的明确关联
      > Agent 声明
        > 默认规则
```

### 4.1 归属纠正与继续编辑

系统可以对低风险、可逆的文件做默认归属，但不得把归属判断藏在请求完成后：

- 每次新建文件的工具结果紧邻显示其容器、用途和最终路径；
- 用户意图和项目惯例足以唯一确定归属时，系统直接采用默认归属，并在紧邻工具结果中展示；例如开发任务的新源码/测试默认属于项目变更，明确的一次性验证脚本默认属于草稿区，不逐文件询问；
- 只有存在两个以上合理归属，且不同归属会对用户产生重要差异（显式路径、是否参与提交、是否可能被清理或交付完整性）时，才在首次写入前询问；
- 同一请求中用途一致的一组文件可由一次归属选择覆盖，不得对每个文件重复询问；例如“后续可能用于运维的检查脚本”在项目 `scripts/` 与会话草稿区均合理时，询问一次其保留方式；
- 用户可以将草稿文件改为项目文件、工作包支撑材料或资料附件；纠正后系统记录该文件的**唯一规范位置**，同一成果后续读写默认继续该位置；
- 若用户想保留两个副本，必须明确选择“复制并继续编辑原文件”或“复制并切换后续编辑到新文件”，完成摘要显示两个位置及当前编辑对象；
- 不允许因“保留到工作包”默认留下两个同名且后续编辑对象不明的版本。

## 5. 项目变更集

### 5.1 判定

当用户目标包括实现功能、修复缺陷、重构、补测试、更新构建或配置、新增 migration、正式脚本或项目资源时，系统建立项目变更上下文。

在此上下文中，为达成目标新建的源码、测试、配置、migration 和正式资源默认为 `project`。用户不需要逐文件声明。

| 用户要求 | 文件 | 归属 |
|---|---|---|
| “实现登录并补测试” | `src/auth/login.ts` | 项目变更集 |
| 同上 | `tests/auth/login.test.ts` | 项目变更集 |
| 同上，临时验证 token | `check_tokens.py` | 会话草稿区 |
| “新增数据库 migration” | `migrations/002_add_user.sql` | 项目变更集 |

### 5.2 行为

- 编辑或覆盖已有项目文件时保持原路径，继续使用既有覆盖确认；
- 正式新建项目文件不得被扩展名映射重定向；
- 任务完成摘要将项目变更列为交付结果的一部分；
- 如果用户同时要求导出副本，例如“实现后把 API 清单写到 `docs/api.md`”，副本属于工作包主成果，不复制或移动正式源码。

### 5.3 仓库正式文档与独立交付的区分

文档、脚本、数据和图片是否属于项目变更，不由“评审”“报告”等名称决定，而由用户是否要求改变当前项目的最终状态决定：

| 用户意图 | 归属 |
|---|---|
| “更新仓库的 `docs/api.md`” | 项目变更集 |
| “在 `docs/review/` 新增本项目评审并提交到仓库” | 项目变更集 |
| “分析该仓库，并把报告导出到 `reports/review.md`” | 工作包主成果，除非用户表达纳入项目 |
| “导出一份 CSV 给我核对” | 工作包主成果 |

当归属会改变 Git 提交范围、清理风险或完成摘要，而用户意图仍不足以判断时，系统在首次写入前询问“作为项目文件保留”还是“作为独立工作包交付”，不能擅自选择。

## 6. 工作包

### 6.1 定义

工作包是围绕一个用户可阅读、可核对、可逐步完善的成果建立的可见容器。适用于：

- 数据分析报告、评审文档、调查结论；
- 研究 → 草稿 → 多轮修改 → 成稿的写作；
- 带 SQL、脚本、数据口径或附件的交付；
- 用户明确要求保留资料、过程依据或引用的任务。

工作包不是隐藏的 Agent 内部目录；它位于用户指定的输出位置附近，用户可在普通文件树中浏览和管理。

### 6.2 主成果与材料目录

用户指定精确主文件时：

```text
docs/review/database-review.md
```

当存在未指定路径的材料时，默认工作包结构为：

```text
docs/review/
├── database-review.md                 # package + primary
└── database-review.materials/
    ├── supporting/
    │   ├── queries/
    │   │   └── active-users.sql
    │   ├── scripts/
    │   │   └── analyze.py
    │   └── notes/
    │       └── methodology.md
    └── references/
        ├── industry-survey.pdf
        └── sources.json
```

用户只指定目录时：

```text
docs/review/
```

系统先确定主成果文件名，再使用其文件名推导同级 `.materials/` 目录。例如主成果为 `database-review.md`，材料目录为 `database-review.materials/`。

材料目录是**默认建议位置**，不是对用户路径的限制：

- 用户逐项指定 `queries/analysis.sql`、`appendix/source.pdf` 等合法路径时，文件必须写入这些精确路径，不强制移入 `.materials/`；
- 用户只说“附上 SQL”“保留调研报告”而未指定材料路径时，才使用 `.materials/`；
- 简单交付只有一两个未指定材料时，允许直接平铺于 `<primary>.materials/`，不强制创建 `supporting/`、`references/` 两层；只有多种材料或同名风险时才使用分类子目录；
- 材料目录命名必须稳定、可预测且可被用户重命名；重命名或移动主成果时，产品不得静默移动材料目录，而应提示是否同步移动并更新关联。

### 6.3 支撑材料

以下文件在未指定路径时默认应归入工作包材料目录，前提是它们直接支撑主成果且用户可能需要核对、复用或提交：

- 报告采用的 SQL、分析脚本、数据口径；
- 写作提纲、摘录、引用笔记；
- 可复现结论所需的小型派生数据；
- 用户明确说“附上”“保留”“一并提交”的文件。

不得因为扩展名而自动把所有脚本或数据放入工作包。例如失败调试脚本、无用尝试和下载缓存仍属于草稿区。

### 6.4 研究资料与引用

用户要求检索或获取报告 PDF、网页资料、数据集时，资料是 `reference`，而不是默认垃圾文件。

每项资料至少需要保留标题、来源链接、获取时间、本地位置（如有）、访问/许可说明（如有）以及其关联的主成果；具体存储格式由技术设计确定。

规则：

- 用户明确指定资料文件或目录路径时，按字面路径保存，不要求额外建立工作包；
- 用户明确要求下载、保留或核对资料但未指定路径时，若当前已有明确关联的工作包，保存到该工作包的默认资料位置，并展示最终路径；
- 用户明确要求下载、保留或核对资料但未指定路径、且当前没有关联工作包时，必须让用户选择：**选择长期保存位置**、**先暂存在本会话研究资料**、或**取消下载**。选择长期保存位置后按用户选定路径保存；选择暂存后创建待决定资料并遵循其生命周期；取消后不产生本地文件；
- 用户仅要求“检索后总结”或短摘要、未要求本地文件、保留、核对原件或持续研究时，检索所需内部缓存不登记为用户工作产物，不写入 `workDir`，不创建待决定资料，也不触发草稿区 Git 选择；结果中的来源链接仍正常展示；
- Agent 判断某份资料将作为后续持续任务的可核对依据、但用户未明确要求本地资料时，必须在创建前以一次轻量提示说明“将在当前工作目录暂存该资料，便于后续核对”，并给出创建或不创建的选择；
- 用户拒绝创建待决定资料时，若任务可在不保留本地原件的前提下完成，继续完成并仅返回来源链接/摘要；若某个后续步骤确实依赖该本地文件，明确说明受影响的步骤并等待用户决定，不得无解释中止整个请求；
- 用户创建工作包或选择“保留资料”时，可将待决定资料复制/移动进入工作包并建立关联；用户未建立工作包时，资料继续留在会话资料清单，直到用户明确清理或会话草稿区被清理；
- 最终成稿不自动携带所有 PDF。用户可选择“只保留引用链接”“保留本地资料”“随稿交付资料”；
- 资料的下载和保存必须遵循现有网络、版权、访问控制和文件写入确认规则；产品不因归入工作包而暗示拥有再分发权。

待决定资料的生命周期必须明确：

- 取消主成果写入不会删除待决定资料；取消提示需说明其数量和位置；
- 会话结束本身不自动删除待决定资料；MVP 不承诺自动过期；
- 用户清理草稿区时，待决定资料单独列出，必须选择“保留并跳过”或“连同资料删除”，不得被无提示的草稿清理删除；
- 用户删除会话时，显示待决定资料数量并确认是否一并删除；
- 这些资料的存储、来源和清理状态必须能从“本会话研究资料”入口查看。

### 6.5 写作工作包

写作不是一次性输出时，主成果可处于 `draft` 阶段。用户可在原路径持续要求“继续完善”“改写第三节”“根据资料补充引用”，无需每轮重新选择输出目录。

示例：

```text
docs/writing/ai-industry/
├── draft.md
├── draft.materials/
│   ├── supporting/
│   │   ├── outline.md
│   │   └── citation-notes.md
│   └── references/
│       ├── industry-survey.pdf
│       └── sources.json
└── final.md                             # 用户明确要求定稿时创建
```

- 草稿与成稿是用户可见主成果，不属于草稿区；
- 用户将 `draft.md` 指定为最终文件时，直接在该路径定稿，不强制创建 `final.md`；
- 用户未指定工作包位置、但请求持续调研和写作时，系统应在首次需要保存草稿或长期保留资料时询问工作包位置；仅检索或短摘要不触发该询问；
- 在用户选择位置前，**已取得用户同意而创建的待决定资料**通过“本会话研究资料”入口可见，用户随后建立工作包时可选择保留其中资料。普通检索的内部缓存不进入该入口；待决定资料遵循 §6.4 的清理规则。

## 7. 会话草稿区

### 7.1 位置与用途

```text
.spaceassistant/
└── runs/
    └── <session-id>/
        ├── scripts/
        ├── data/
        ├── pending-references/
        ├── cache/
        └── index.json
```

- 位置为 `workDir/.spaceassistant/runs/<session-id>`，保持所有 Agent 文件在用户工作目录内、可从文件系统找到和清理；
- 默认仅存放 `scratch`；唯一例外是尚未建立工作包的 **待决定资料**，它们位于 `pending-references/`，遵循 §6.4 的独立可见性与清理规则；
- 主草稿、用户明确要求保留的脚本和已关联工作包的资料不得默认放入此处；
- 普通文件树默认折叠/隐藏该目录，用户可随时显示；聊天区或详情区提供“本会话草稿文件”和“本会话研究资料”入口；
- 会话活跃时不自动清理；MVP 支持单文件和整会话手动清理。整会话清理必须把待决定资料列为单独选择。

### 7.2 Git 处理

在 Git 工作区创建草稿区文件前，若 `.gitignore` 尚未覆盖 `.spaceassistant/runs/` 且该工作区没有已保存的用户选择，显示一次选择：

1. **将 SpaceAssistant 草稿区加入 `.gitignore`（推荐）**；
2. **继续但不忽略**；
3. **取消本次草稿文件创建**。

规则：

- 用户确认加入时，产品仅追加精确规则 `.spaceassistant/runs/`，通过正常文件写入确认展示 `.gitignore` 修改；
- 用户选择继续但不忽略时，提示这些文件会出现在 Git 未跟踪文件中；该选择可在当前工作区设置中修改；
- 选择以工作区为范围保存，不因新会话重复打断；仅当用户在设置中重置选择，或 `.gitignore` 相关规则发生变化时重新询问；
- 用户取消时，不创建草稿文件，不回退写入项目根目录；
- 产品不得静默修改 `.gitignore`、项目配置或任何需提交文件；
- 工作包和项目变更是用户有意保留的内容，不能因为草稿区忽略规则而被忽略。

### 7.3 清理与运行中保护

- 系统在 Agent 当前请求中正在读取、编辑、执行或作为后续步骤依赖的草稿文件上显示“使用中”；
- 用户尝试删除使用中文件时，不执行删除，并说明“该文件正在被本次请求使用，完成或取消后再清理”；
- 使用结束后恢复可清理状态；
- 整会话清理同样跳过使用中文件，并列出跳过原因；
- 待决定资料不随普通草稿清理删除，遵循 §6.4 的单独选择。

## 8. 输出意图与确认

### 8.1 成果级路径关联与精确意图

会话保存“已识别成果 → 规范路径”的关联，用于持续编辑同一成果；它不是把最近目录自动施加给会话中的所有后续成果。关联至少能区分成果、文件/目录路径和用户指定的消息；具体数据结构由技术设计确定。用户只有明确说“后续都输出到……”时，才建立可跨新成果使用的默认目录。

- “继续”“按这个方向写”“改写第三节”等指令匹配已识别成果时，持续使用其关联路径，不重复进行目录或覆盖确认；
- 当前消息中新的明确路径只更新其指向的成果；它不覆盖会话中无关成果的路径；
- 用户明确说“后续都输出到 `reports/final`”时，才更新跨新成果使用的默认目录；
- 对已有成果说“把这份草稿改到 `reports/final.md`”时，系统必须说明并让用户选择：**移动并继续编辑新位置**、**另存为并继续编辑新副本**、或**只为未来新成果设置默认位置**；
- “需要”“继续”等简短答复不清空已识别成果的关联；
- 一个消息中的两个精确文件路径分别绑定对应主成果，例如报告写 `docs/report.md`、明细写 `data/result.csv`；
- 多个目录级位置无法关联主成果时，按主成果逐个询问，不能任意选择一个作为全会话默认；
- 会话结束或删除时，成果关联和跨成果默认均失效。

### 8.2 字面路径规则

- 用户明确指定 `docs/review/review.md`，主成果精确写入该路径；
- 用户明确指定 `docs/review`，主成果直接写入该目录；
- 用户明确指定 `queries/analysis.sql` 或 `appendix/source.pdf` 时，支撑材料或资料附件精确写入该路径；
- 显式文件或目录不追加扩展名映射目录；
- Agent 建议路径不能覆盖用户显式路径；
- 用户只指定目录时，主成果文件名由用户要求、项目约定或 Agent 建议确定，并在写入结果中展示；
- 材料目录只在存在未指定路径的 `supporting` 或 `reference` 时创建，打开确认或仅保存主成果不得预创建空目录。

### 8.3 路径类型判定

对于用户指定的路径，系统按以下顺序确定它表示文件还是目录：

1. 用户明确说“文件”“目录”“文件夹”“保存为”等，且与目标已存在类型一致时，按表达直接采用；
2. 用户明确表达的类型与目标已存在类型冲突时，提示冲突并要求用户改路径或改类型；不得把目录当文件覆盖，也不得把文件当目录写入；
3. 用户未明确类型且目标已存在时，按文件系统实际类型确定；
4. 不存在且输入以路径分隔符结尾（如 `reports/final/` 或 `reports\\final\\`）时，按目录处理；尾随分隔符是用户路径语义的一部分；
5. 其他不存在路径中，用户上下文与项目约定能唯一指向一种类型时直接采用；例如“把报告输出到 `reports` 目录”是目录，“生成无扩展名文件 `LICENSE`”是文件；
6. 仅当不存在路径且文件、目录两种解释都合理时，询问“将 `reports/final` 作为文件还是目录？”。不得仅凭扩展名、名称中是否带点或 Agent 猜测决定，更不得先创建再要求用户纠正。

带点的目录名（如 `reports.v2`）不能仅凭点号视为文件；无扩展名文件（如 `LICENSE`、`Makefile`）不能仅因没有扩展名视为目录。

### 8.4 何时询问

不询问目录选择或覆盖的条件：合法、唯一、工作目录内的显式文件或目录，且目标是当前会话已关联成果的持续修改，或用户本轮明确要求修改该已有文件。

仅在以下情况询问：

- 用户未指定主成果或工作包位置；
- 多个目录无法关联；
- 用户明确要求下载/保留资料但没有显式路径或关联工作包；此时按 §6.4 选择长期保存位置、会话暂存或取消；
- 路径格式错误或位于工作目录外；
- 首次可能覆盖与当前成果无关的已有文件；
- 目录级输出自动命名后发生同名冲突；
- 多个主成果同名或用户意图冲突。

目录级输出发生同名冲突时，系统必须展示拟用文件名，并让用户选择覆盖、改名、改目录或取消；不得静默改名或覆盖。

安全授权和目录选择是不同语义。若产品基线要求写入授权，应由既有写入确认承载，不能用目录选择重复询问。

## 9. 安全、跨平台与执行语义

### 9.1 路径边界

- `workDir` 是项目变更、工作包和会话草稿区的硬边界；MVP 不支持写出该范围；
- 草稿区文件只能由系统分配到当前会话目录；用户普通路径输入、Agent 声明和渲染进程回传不能任意指定草稿区路径；
- 未指定路径的工作包材料必须位于主成果同级 `.materials/` 目录内；用户逐项指定的材料路径按 §8.2 执行，但仍必须位于 `workDir` 内；
- 所有已存在路径解析 symlink/junction 后检查真实路径；不存在目录向上检查最近已存在父目录；
- 禁止用字符串前缀判断边界，必须使用规范化路径、`path.relative` 和真实路径校验；
- 工作目录外路径不自动改写为工作目录内同名路径；显示原因并要求用户选择工作目录内位置。

### 9.2 Windows 与 POSIX

- Windows 支持盘符路径、反斜杠和正斜杠；防止盘符切换、UNC、设备路径、junction、保留设备名与尾随点/空格别名；
- POSIX 支持绝对和相对路径；防止 symlink 越界；
- 异平台绝对路径不能证明在当前 `workDir` 内时拒绝，不擅自转换；
- 身份比较使用当前平台规范化后的真实路径，展示采用产品统一格式。

### 9.3 执行不变性

文件归属不能降低 Agent 完成任务的能力：

1. 工具结果返回容器、材料类型、原请求路径、最终路径和原因；
2. Agent 后续上下文使用最终路径；
3. 后续 `read_file`、`edit_file`、脚本执行和引用能够访问当前会话草稿文件及工作包材料；
4. 草稿脚本位于 `.spaceassistant/runs` 时，执行 cwd 仍按任务约定设为 `workDir`，保证 `./data/source.csv` 等项目相对资源语义不变；
5. 多个草稿或支撑文件互相引用时，系统必须提供一致的最终路径；
6. 重定向失败时明确失败，不回退写入项目根目录；
7. 会话切换后，其他会话不能误用当前会话草稿区。

## 10. 用户交互

### 10.1 本会话工作产物

聊天区或详情区展示：

```text
本会话工作产物
├── 项目变更（3）
├── 工作包（2）
│   ├── 数据库评审
│   │   ├── database-review.md
│   │   ├── 2 个支撑材料
│   │   └── 1 个研究资料
│   └── 行业文章草稿
└── 草稿文件（4）
```

- 项目变更和主成果默认突出展示；
- 工作包显示主成果、材料数量、位置和草稿/定稿状态；
- 研究资料可打开、查看来源信息、选择保留到工作包；
- 草稿文件默认折叠，但可打开、查看路径、保留到工作包或转为项目文件；
- 用户纠正归属时必须明确后续编辑对象：选择移动/切换后，后续 Agent 编辑新位置；选择复制/保留原件后，用户明确选择继续编辑原件或副本。完成摘要持续显示规范位置。

### 10.2 完成摘要

每次 Agent 请求完成时必须列出：

- 新增或修改的项目文件；
- 新建或更新的工作包主成果及精确路径；
- 新增的支撑材料与研究资料数量；
- 草稿文件数量和查看入口。

写作任务不能只显示“完成”，还应让用户明确当前修改的是草稿还是定稿。

### 10.3 取消

用户取消主成果写入或工作包位置选择时：

- 不创建或覆盖主成果；
- 不把主成果降级写入草稿区；
- 已下载、可核对的研究资料保留在“本会话研究资料”入口，用户可选择清理或稍后建立工作包；
- 显示本会话已经产生的草稿文件和研究资料数量；
- 明确提示：“本次仅取消保存该主成果；已完成的项目变更、本会话研究资料和已保留材料仍在原位置。”；
- 不自动删除项目变更、工作包材料或用户已打开/保留的资料。

- **Git 与清理**：三种 Git 选择、工作区级记忆、精确 `.gitignore` 修改、草稿查看/纠正归属/删除、使用中保护和幂等清理。
- **路径安全**：不存在目录、穿越、真实路径、symlink、junction、Windows 特殊路径和异平台输入。
- **执行不变性**：项目相对资源、材料引用、草稿脚本执行、会话隔离、最终路径回传。
- **端到端**：开发任务同时新增源码、测试和临时脚本；数据库分析产出报告与 SQL；调研写作下载 PDF、迭代草稿并定稿。

## 14. MVP 成功指标

试运行前须为以下事件建立可观测记录；数值阈值在灰度方案中确定，不在本需求中预设：

| 指标 | 分子 / 分母 | 最低观测口径 |
|---|---|---|
| 未归属辅助文件率 | 被写入项目根且未被判为 `project`、工作包或草稿区的文件数 / Agent 新建辅助文件数 | 记录原始请求路径、最终路径、归属与原因 |
| 显式路径改写率 | 用户显式路径与最终路径不一致的文件数 / 显式路径文件数 | 目标为 0，安全拒绝不计为改写 |
| 正式文件误隔离率 | 后续被用户或系统纠正为项目文件的草稿文件数 / 草稿区新建文件数 | 记录初始归属、纠正归属及触发来源 |
| 产物可发现性 | 用户从完成摘要或产物入口成功打开目标文件的次数 / 产物打开尝试次数 | 区分主成果、材料、待决定资料 |
| 重定向执行失败率 | 可归因于路径重定向的执行失败数 / 涉及重定向的工具执行数 | 记录工具、cwd、原路径、最终路径和错误类别 |
| 清理误伤率 | 清理操作误影响非草稿文件的次数 / 草稿清理操作次数 | 目标为 0 |

此外，Git 忽略选择必须记录为工作区级状态，以验证用户不会因新会话被重复打断。

## 15. 已确定的关键决策

1. 目录规范的目标是管理工作产物归属，不是按扩展名重排所有新文件。
2. 容器分为项目变更集、工作包和会话草稿区；仓库正式文档是否属于项目变更取决于用户是否要求改变项目最终状态。
3. 工作包可包含主成果、支撑材料和研究资料；写作草稿是用户可见主成果，不是临时文件。
4. 用户逐项指定的主成果、材料和资料附件路径均按字面生效；`.materials/` 只服务未指定路径的材料。
5. 草稿区位于工作目录内 `.spaceassistant/runs/<session-id>`；待决定研究资料可在其中暂存，但具有独立可见性和清理确认，不能被当作普通缓存删除。
6. 成果路径按成果关联持续继承，不把最近输出目录自动施加给会话中新增成果；持续编辑同一成果不重复确认。
7. 用户纠正归属后必须有唯一规范继续编辑位置；复制副本时由用户选择后续编辑对象。
8. 资料是否保留本地副本或随稿交付由用户决定；产品记录最小来源信息，但不默认暗示再分发权。
9. 归属仅在存在多个合理选择且会带来重要用户差异时询问；意图明确时采用默认归属并展示结果。
10. 简单检索的内部缓存不是用户工作产物，不写入工作目录、不触发 Git 交互；待决定资料只在用户要求或用户同意持续核对暂存时创建。
11. 明确下载资料但尚无路径或工作包时，由用户选择长期保存、会话暂存或取消；系统不得静默决定。
12. MVP 优先实现正确归属、可见性、路径安全和手动清理；复杂生命周期、自动移动和自动清理后置。
