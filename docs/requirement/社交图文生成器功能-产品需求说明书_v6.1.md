# 第〇部分：为什么做这个产品

---

## 0. 目标用户与使用场景

### 0.1 核心用户画像

| 画像 | 描述 | 月产出 | 核心痛点 | 对本产品的期望 |
|------|------|--------|----------|---------------|
| **P1 · 精品内容创作者** | 小红书/公众号的知识博主，注重内容深度和视觉品质。如旅行攻略作者、职场教练、读书博主 | 4-8 篇/月 | 每篇花 1-2 小时在 Canva/稿定设计手动排版；设计师外包成本高（¥200-500/篇）；改稿来回沟通耗时 | 「我把文章写好，排版的事自动搞定，而且看起来像设计师做的」 |
| **P2 · 品牌内容运营** | 企业新媒体运营，负责品牌账号日常更新，需要维持统一视觉调性 | 15-30 篇/月 | 品牌视觉一致性难以在多人协作中保持；模板复用性差；Canva 模板同质化严重看不出品牌辨识度 | 「输入文案出图，风格统一，配色一键切，能批量产出」 |
| **P3 · 效率矩阵号** | 运营多个账号的矩阵玩家，追求产出效率，内容标准化程度高 | 30-60 篇/月 | 手动排版是产能瓶颈；需要同时在多平台发布不同尺寸；现有工具的「一键生成」结果需要大量修改 | 「批量、多尺寸、少修改」 |
| **P4 · 设计师 / 模板创作者** | 有设计能力，想将自己设计的排版风格打包分发给他人使用（v2.0 生态目标） | — | 现有工具不支持模板的标准化打包和分发；设计稿到可用模板之间需要前端开发配合 | 「把设计稿变成可分发、可版本管理的排版组件」 |

> **v1.0 聚焦 P1 和 P2。** P3 的需求（批量、多尺寸）在架构上预留但不优先实现交互。P4 是 v2.0 生态战略目标。

### 0.2 用户当前的替代方案

| 替代方案 | 典型工具 | 优势 | 劣势 | 本产品的切入点 |
|----------|---------|------|------|---------------|
| 手动排版 | Canva、稿定设计、美图秀秀 | 控制力强、结果可控 | 耗时 30-90 分钟/篇；设计能力要求高；跨平台不同尺寸需重做 | 用 AI 消除「手动排列组合」的时间 |
| 外包设计师 | 兼职/全职设计师 | 品质保证、定制化 | ¥200-500/篇；沟通成本高；改稿周期长 | 提供「设计师级」的默认品质，用户只需微调 |
| AI 图文工具 | Gamma.app、美图 AI PPT | 真的快（几秒出图） | 结果不可控；无法精确指定风格细节；不支持品牌色/字体定制 | 提供「可控的自动排版」——风格可预设、版式可编排、文案可逐字修改 |
| 纯手动 HTML/CSS | 前端自写 | 完全自由 | 门槛极高；每次改内容等于重构 | 把 HTML/CSS 能力封装成组件化体系，降低至选择级别 |

### 0.3 使用场景与频率

| 场景 | 频率 | 典型行为 | 产品关键需求 |
|------|------|----------|-------------|
| 周更博主的常规发文 | 1-2 次/周 | 写好的文章 → 粘贴 → 选风格 → 生成 → 微调 → 导出 | 流程短、默认效果好、首版满意率高 |
| 日更矩阵号的批量产出 | 1-2 次/天 | 多篇文章排队处理，每篇微调量要尽可能少 | 批量生成、序列模板、一键导出 |
| 品牌活动的密集发布 | 集中 3-5 篇/天 | 同一风格下产出不同内容，保持品牌调性统一 | 风格克隆、配色一致锁定、多尺寸同时导出 |
| 风格探索期 | 低频（每月几次） | 浏览不同风格，尝试新风格，下载社区风格 | 风格预览流畅、切换成本低、社区内容丰富 |

### 0.4 竞品差异化定位

本产品与竞品的本质差异不在于「AI 生成图文」，而在于**排版组件的正交可替换性**：

| 维度 | Canva / 稿定设计 | Gamma.app | 本产品 |
|------|-----------------|-----------|--------|
| 排版方式 | 用户手动拖拽 | AI 一次性生成，不可拆解 | **AI 选择版式序列，用户可逐页替换版式** |
| 风格切换 | 换模板 = 全部推倒重来 | 不支持精确风格定义 | **换风格 = 换 CSS 实现；若 Contract 兼容，版式序列可迁移** |
| 品牌定制 | 上传品牌 kit，手动套用 | 不支持 | **风格包 = 品牌视觉系统；一次配置，所有内容自动套用** |
| 组件生态 | 模板市场（封闭，平台审核） | 无 | **Contract 驱动的开放生态：设计师定义风格，社区贡献版式，用户混搭** |
| 核心壁垒 | 素材库规模 + 协作功能 | AI 生成速度 | **语义理解（Skill）+ 排版组件的正交可替换性（产品架构）** |

> **一句话差异化：** Canva 让你「设计」图文，本产品让你「定义」排版系统——然后 AI 替你用它。

# 第一部分：产品框架

---

## 1. 第一性原理：这个产品到底解决什么问题

### 1.1 用户要什么

> **用户有一段内容（文章、攻略、测评、教程……），想把它变成一组好看的小红书图文或公众号封面，不用自己设计排版。**

这就是全部。所有功能都从这个原点生长出来。

### 1.2 拆解到底层，只需回答三个问题

```
❶ 长什么样？      →  视觉呈现（Style）
❷ 每页放什么？    →  智能分页与填内容（Skill / AI）
❸ 怎么拿出来？    →  渲染成图并导出（Engine）
```

整个产品的架构就是对这三个问题的回答。

### 1.3 一个实例贯穿始终

```
输入：一篇 2000 字的马尔代夫旅行攻略 + 8 张照片
期望：一组 7 张小红书图文

过程：
  1. 用户选「电子杂志风」风格（回答「长什么样」）
  2. AI 拆解文章为 7 页，选好每页的版式，
     填入用户文字和照片（回答「每页放什么」）
  3. 渲染引擎截图出 7 张 PNG，校验无问题（回答「怎么拿出来」）
  4. 用户预览、微调、导出到桌面
```

## 2. 三个核心系统

| 系统 | 回答的问题 | 核心工作 | 执行者 |
|------|-----------|----------|--------|
| **视觉呈现系统** | 长什么样？ | 定义排版语言、颜色、版式 | 产品管理组件，设计师创作组件 |
| **智能排版引擎** | 每页放什么？ | 理解内容、拆页、选版式、填文案 | Skill（AI） |
| **渲染与输出系统** | 怎么拿出来？ | HTML → PNG、校验、导出 | 产品 |

这三个系统是正交的：视觉组件可以独立设计、Skill 可以独立升级、渲染引擎可以独立优化。它们通过两个接口连接：

- **Skill 输出协议**：AI 排好的页面以结构化 JSON 交给渲染引擎
- **Class Contract**：版式组件引用视觉组件中定义的 CSS class

> 三个核心系统的详细架构、组件层级关系和机制细节见 **《渲染组件体系框架说明 v6.1》**。

## 3. 产品与 Skill 的分工边界

这是整个架构最重要的设计决策。

### 3.1 一句话

> **产品做一切确定性的工具活；Skill 只做「理解语义 + 选择版式 + 填入内容」这一件事。**

### 3.2 分工表

| | 产品（确定性机器） | Skill（AI 语义推理） |
|---|---|---|
| **负责** | 渲染截图、存储组件、切换风格/配色、执行校验、管理图片、导出 PNG、项目管理、HTML 预览面板推送 | 理解文章、拆解故事、从当前风格的版式库中选择序列、填入文案、必要时压缩文案 |
| **不负责** | 理解文章语义、选择版式、决定拆页数 | 渲染截图、管理文件、搜索图片、导出文件 |
| **比喻** | 相机机身 + 暗房 | 摄影师 |

### 3.3 编排 Skill 与排版 Skill 的拆分

v1.0 聊天场景中存在两个 Skill：

```
┌─────────────────────────────────────────────────┐
│                  编排 Skill                      │
│  （对话主持人，仅在聊天场景中存在）                 │
│                                                 │
│  职责：                                          │
│  • 管理对话节奏（逐步收集用户选择）                 │
│  • 展示风格 / 配色 / 版式选项                     │
│  • 调用排版 Skill 进行生成                       │
│  • 向右侧面板推送 HTML 内容                      │
│  • 处理用户微调指令                              │
│  • 触发导出                                      │
│                                                 │
│  不负责：理解文章、选择版式、填入文案               │
└───────────────┬─────────────────────────────────┘
                │ 调用（Skill 间通信）
                ▼
┌─────────────────────────────────────────────────┐
│                  排版 Skill                      │
│  （可被多种场景复用：聊天窗口 / 产品 UI / API）     │
│                                                 │
│  职责：                                          │
│  • 理解内容语义、判断内容结构类型                   │
│  • 从当前风格的版式库中选择序列                    │
│  • 将用户文案填入版式占位符                       │
│  • 压缩文案以适应硬限制                           │
│  • 输出结构化 Poster 数组                        │
│                                                 │
│  不负责：收集用户偏好、展示结果、处理微调、导出     │
└─────────────────────────────────────────────────┘
```

**拆分的理由：**

| # | 理由 | 说明 |
|---|------|------|
| 1 | **职责正交** | 编排 Skill 做「对话管理 + 交互引导」，排版 Skill 做「内容理解 + 版式选择 + 文案填入」 |
| 2 | **复用性** | 排版 Skill 是通用能力——将来完整产品 UI 出现时，仍然调用同一个排版 Skill |
| 3 | **可独立迭代** | 对话体验和核心推理能力可以各自独立优化 |
| 4 | **接口清晰** | 两者之间通过同一份数据契约（F1-F8）和 Poster 数组协议通信 |

### 3.4 两个关键接口

#### 3.4.1 产品层 → Skill（输入）

```
产品层 ──(提供)──→ Skill
  ├── 风格注册表（有哪些风格、每个风格下有哪些版式 / 配色 / Contract）
  ├── 用户内容（F1-F8 数据契约，见附录 A）
  └── 用户约束（硬性指令）

Skill ──(产出)──→ 产品层
  └── Poster 数组（结构化 JSON，含每页 HTML + 元数据 + 图片分配映射）
```

#### 3.4.2 编排 Skill → 排版 Skill 输入协议

编排 Skill 调用排版 Skill 时，传递以下结构化数据：

```jsonc
// 编排 Skill → 排版 Skill 输入
{
  "input": {
    "body": "string (Markdown)",          // F2 — 用户原始正文
    "images": [ImageAsset],               // F3 — 图片素材列表
    "tone": "travel",                     // F4 — 内容调性（可选）
    "metadata": {},                       // F7 — 补充元数据（可选）
    "constraints": "string"               // F8 — 硬性约束（可选，微调阶段可为用户修改后的文案）
  },
  "context": {
    "styleId": "editorial-magazine",      // F5 — 视觉风格 ID
    "targetSpecs": ["xhs"],               // F1 — 目标输出规格
    "themeId": "midnight-ink",            // F6 — 当前配色（生成前为默认值）
    "styleRegistry": {                    // 当前风格的版式库摘要
      "recipes": [
        {
          "id": "M01",
          "pageType": "cover",
          "description": "杂志封面，大标题 + 主图 + 底部要点条",
          "contentLimits": { "title": { "maxLines": 2, "maxCharsPerLine": 8 } }
        }
        // 元数据摘要，不含完整 HTML
      ]
    }
  }
}
```

> **说明：** `styleRegistry` 仅包含每个 Recipe 的元数据摘要（id、pageType、description、contentLimits），不包含完整 HTML。编排 Skill 不需要理解 HTML——它只需要传递给排版 Skill，排版 Skill 根据摘要选择版式序列。完整 HTML 由产品层在排版 Skill 确定版式后，从本地 Recipe 库中读取并填充。

## 4. 端到端工作流

### 4.1 核心假设：右侧 HTML 预览面板

v1.0 引入一个关键交互基础设施：**聊天窗口右侧的 HTML 预览面板**，可以实时渲染 HTML（非 PNG 截图），用户在其中直接看到排版效果。

HTML 渲染和 PNG 截图有本质区别：

| | HTML 渲染 | PNG 截图 |
|---|---|---|
| 速度 | 瞬时（< 100ms） | 每页 1-3 秒 |
| 成本 | 几乎为零 | 每次触发完整的浏览器截图管线 |
| 适用场景 | 「看看效果」「比较一下」「随便调调」 | 「确认最终结果」「导出」 |

引入 HTML 预览面板意味着：**「看」和「确认」被解耦了。** 用户可以随意「看」——换文字、换版式、换图片，所有变化在右侧面板中瞬时呈现。只有用户说「就是它了」的时候，才触发 PNG 渲染。

### 4.2 重组后的完整流程

基于 HTML 预览面板，Theme 和约束被后置到生成后的微调阶段（理由见 §4.3）：

```
① 用户 @ 文档 → 编排 Skill 识别意图
② 选平台（输出规格）
③ 选风格（轮播卡片，A/B/C/D/E）
④ 粘贴图片（可选，但明确引导。用户传图则排版 Skill 优先使用）
⑤ 调用排版 Skill 生成 → HTML 推送到右侧面板
⑥ 用户在面板中看到结果 → 配色切换、文案约束、换版式、改文字全部在此阶段
    ——所见即所得，所有调整都是自然语言 + 面板即时反馈
⑦ 确认 → PNG 渲染 → 导出
```

| | 生成前（聊天） | 生成后（面板 + 聊天） |
|---|---|---|
| 平台 | ✅ | |
| 风格 | ✅（轮播卡片） | |
| 图片 | ✅（⊕ 或 Cmd+V） | 可继续追加 |
| 配色 | 不收集，用默认 | ✅ 面板实时切换 |
| 约束（保留标题/密度/紧凑）| 不收集 | ✅ 自然语言 + 面板即时反馈 |
| 改文字 / 换版式 / 换图片 | | ✅ 双通道微调 |

### 4.3 为什么 Theme 和约束后置

Theme 和约束有三个共同特征：

1. **都是可选的**——Theme 有默认值，约束不填也能生成
2. **都依赖视觉参照物**——看着效果调，比凭空想象高效得多
3. **都不影响排版 Skill 的版式选择逻辑**——只影响渲染参数和文案处理，不需要排版 Skill 重新推理

这相当于：不要在用户还没看到菜单时问「你要几分熟」。生成前用默认 Theme 渲染，用户在面板上看到结果后再调整。

## 5. 功能模块总览

| 模块 | 一句话 | 详细说明 |
|------|--------|----------|
| **内容输入** | 用户素材 → 系统可消费数据结构 | 支持 @ Markdown 文档、粘贴正文、上传图片。数据契约 F1-F8（见附录 A） |
| **风格选择** | 轮播卡片式风格浏览与选择 | 5 个内置风格，A/B/C/D/E 标注，一条消息内可翻页浏览 |
| **排版生成** | AI 理解内容 → 输出结构化页面 | 排版 Skill 拆页、选版式、填文案，输出 Poster 数组 |
| **HTML 预览面板** | 右侧实时渲染 HTML | 瞬时推送、逐页浏览、配色实时切换、并排对比 |
| **微调交互** | 聊天下指令 + 面板看效果 | 改文字、换版式、换图片、批量修改——全部即时反馈 |
| **校验系统** | 自动检测问题，面板标注 + 聊天提醒 | 溢出/密度/字体一致性等 7 条规则 |
| **PNG 渲染与导出** | 确认后渲染 → 导出到桌面 | 仅用户确认后才触发 PNG 渲染管线 |
| **项目与历史** | 项目保存、版本管理 | 相对路径可移植、历史版本回退（项目数据模型见附录 D） |

# 第二部分：业务流程与用户交互

## 6. 双通道交互模型：聊天 + 预览面板

### 6.1 核心设计原则

> **聊天承载「决策」，面板承载「感知」。**
>
> 编排 Skill 在聊天中做的事：呈现选项 → 解释差异 → 征求确认 → 宣布结果。
> 面板做的事：把任何一个决策的视觉后果，在用户做下一个决策之前，就已经展示清楚。

```
┌─────────────────────┐   ┌──────────────────────────┐
│   聊天窗口（左侧）    │   │   HTML 预览面板（右侧）    │
│                     │   │                          │
│  · 自然语言指令      │   │  · 实时 HTML 渲染         │
│  · 选项选择          │   │  · 逐页浏览 / 缩放       │
│  · 确认 / 提交       │   │  · 并排对比两个版式       │
│  · 最终 PNG 缩略图   │   │  · 内联文字编辑（可选）   │
│                     │   │                          │
│  「控制面」          │   │  「视图面」               │
└─────────────────────┘   └──────────────────────────┘
```

### 6.2 信息分流原则

| 信息类型 | 放在聊天 | 放在面板 |
|---------|---------|---------|
| 「换成 M07 了」这个事实 | ✅ 一句话确认 | ❌ |
| M07 装了自己的内容长什么样 | ❌ | ✅ 即时 HTML |
| 「标题可能溢出」这个警告 | ✅ 一句话提醒 | ✅ 红色虚线标注 |
| 配色切换效果 | ✅ 一句话确认 | ✅ 面板即时重绘 |
| 最终的 PNG 全精度确认 | ✅ 缩略图 | ❌（面板只做 HTML） |
| 两个版式的对比 | ❌ | ✅ 并排渲染 |
| 校验结果清单 | ✅ 摘要消息 | ✅ 逐页叠加标注 |

### 6.2.1 「当前页」的权威来源

双通道模型中，聊天和面板可能指向不同的页面。以下规则确定「当前页」的权威来源：

| 场景 | 权威来源 | 行为 |
|------|----------|------|
| 聊天中明确提及页面编号（「第 3 页标题改成 XXX」） | **聊天指令** | 面板强制切换到第 3 页展示修改结果 |
| 用户模糊指令（「这一页换个版式」） | **面板当前展示页** | 「这一页」以面板当前显示的页面为准 |
| 用户在面板中手动翻页（未伴随聊天指令） | **面板** | 聊天中「当前页」概念不更新，直到下一条页面级指令 |
| 编排 Skill 主动推送（如「全部完成」） | **聊天指令** | 面板默认展示第 1 页（封面） |

> **核心规则：聊天下达的页面级指令（改文字、换版式、换图片）强制面板切换到对应页面。用户模糊指令「这一页」以面板当前展示页为准。**

### 6.3 交互节奏

| 交互阶段 | 视图载体 | 渲染方式 | 速度 |
|---------|---------|---------|------|
| **探索性调整**（「试试这个」「改成那个」） | 右侧 HTML 面板 | HTML 即时推送 | < 100ms |
| **确认性提交**（「就这样」「渲染吧」） | 聊天中的 PNG 缩略图 | 完整截图管线 | 1-3 秒/页 |
| **最终导出** | 桌面文件 | 全精度 PNG | 批量 |

## 7. v1.0 场景：Agent 聊天窗口完整流程

### 7.1 场景描述

用户在 Agent 聊天窗口中，选中一篇 Markdown 格式的文档作为正文输入，编排 Skill 以对话方式逐步引导用户完成风格选择、生成、预览、微调、导出全流程。聊天窗口右侧存在 HTML 预览面板，实时展示排版效果。

### 7.2 为什么这个场景适合 Agent 聊天窗口

| 维度 | 说明 |
|------|------|
| **输入自然** | 用户不需要学习「正文输入区」「风格选择器」等 UI 概念——只需把已有的 Markdown 文档丢进来，剩下的交给对话 |
| **逐步引导** | 每一步都是一问一答，用户的认知负担低，不会面对一屏幕的表单字段不知所措 |
| **微调直觉** | 「第 3 页标题改成 XXX」比在预览窗口中双击编辑更符合聊天场景下的表达习惯 |
| **可中断可恢复** | 对话天然支持暂停和恢复——用户可以先选风格，过一会儿再继续 |
| **即时视觉反馈** | 右侧面板让每次调整的效果立即可见，消除等待焦虑 |

## 8. 生成前：信息收集阶段

### 8.1 对话节奏原则

> **编排 Skill 的对话节奏应该是「用户给多少信息，我就少问多少问题」，而不是固定流程的逐步确认。**

| 场景 | 编排 Skill 行为 |
|------|----------------|
| 用户明确说「小红书 + 杂志风 + 午夜墨」 | 识别用户已一次性命中多个参数，跳过中间确认：「好的，小红书 + 杂志风 + 午夜墨。还有图片吗？没有我就直接开始了。」 |
| 用户粘贴图片时附带说明「封面用这张」 | 识别文本附言中的意图，直接建立 slot 映射，省掉一轮确认 |
| 用户说「和上次一样」 | 读取上一项目的 Style/Theme/输出规格配置，一句话确认：「上次用的是电子杂志风 + 午夜墨 + 小红书，这次一样吗？」 |

#### 8.1.1 最小可生成条件

编排 Skill 决定「现在可以调用排版 Skill 了」的判定规则。以下条件**全部满足**时方可调用：

| 条件 | 字段 | 规则 |
|------|------|------|
| ✅ 正文存在 | F2 `body` | `body` 存在且 ≥ 100 字符（软下限：低于 100 字符允许继续但提示「内容较短，可能只生成 1-2 页」） |
| ✅ 输出规格已确定 | F1 `targetSpecs` | 用户明确选择 或 默认使用 `xhs`（小红书） |
| ✅ 风格已确定 | F5 `stylePreference` | 用户明确选择 或 默认使用最近使用的风格；若无历史记录（首次使用），编排 Skill 必须引导用户从轮播卡片中选择，不可自行假设默认风格 |
| ✅ 图片非必填 | F3 `images` | 无图片时可生成纯文字排版；编排 Skill 在收集阶段应主动引导「有图片的话可以传给我」 |

> **询问上限：** 若条件不满足，编排 Skill 应主动询问缺失的关键字段。编排 Skill 最多询问 **2 轮**（避免陷入无限追问），2 轮后若仍有未满足条件：若缺失风格（首次用户），编排 Skill 仍须引导选择；其他可默认字段（如输出规格默认小红书）使用默认值继续。

### 8.2 风格选择：轮播卡片

风格选择的核心挑战：用户需要看到多张样本图才能判断「这个风格对不对味」。但单个风格自身的展示信息密度已经很高（名字 + 一句话描述 + 2-3 张样本图），5 个风格不可能挤在同一条聊天消息中。

**解决方案：轮播卡片。** 一条消息内展示一个风格的完整信息（名字 + 一句话描述 + 2-3 张典型页面样本），卡片下方有圆点指示器，用户点击圆点或左右滑动切换到其他风格。5 个风格 = 1 张可翻页的卡片。每个风格标注字母（A/B/C/D/E），用户回复字母即可选择。

```
编排 Skill: 想用什么视觉风格？

          ┌─────────────────────────────┐
          │  [封面样本] [内页样本]        │
          │  A · 电子杂志风               │
          │  文艺、杂志感，适合游记和深度内容 │
          │  ● ○ ○ ○ ○                  │
          └─────────────────────────────┘

User: A
```

每翻一页，卡片中的样本图和描述切换。用户可以在同一位置连续浏览，不产生额外消息。

> **配色不在此时收集。** 配色本质是「微调参数」——生成后用默认配色渲染，用户在面板上看到结果后，面板顶部提供配色下拉选择器实时切换。详见 §9.5。

### 8.3 图片上传

**实际可行的图片上传流程（桌面 Agent）：**

| 入口 | 操作 | 适合场景 |
|------|------|---------|
| **⊕ 按钮** | 聊天输入框旁的通用附件按钮 → 弹出文件选择器 → 多选图片 | 用户还没打开 Finder；需要一次选多张 |
| **Cmd+C → Cmd+V** | 用户在 Finder 中选中图片 → 复制 → 回到 Agent 粘贴 | 用户正在浏览照片文件夹，自然连贯 |

> 「拖拽上传」在全屏三栏桌面 Agent 场景下不可行——用户看不到 Finder，无法拖拽。⊕ 按钮和粘贴是桌面聊天应用的通用基建，不应在聊天界面额外增加「选择图片」按钮。

**图片上传后的编排 Skill 行为：**

```
编排 Skill: 收到 5 张图。
           [缩略图列表]
           封面我优先用「hero-maldives.jpg」，其他的按内容自动分配。
           有特别想用在某页的吗？
```

图片到 slot 的映射信号优先级（详见附录 A F3 `ImageAsset`）：

| 优先级 | 信号 | 说明 |
|--------|------|------|
| 1 | **文件名关键词匹配** | 文件名含「封面」「hero」「cover」→ 优先匹配封面 slot |
| 2 | **排版 Skill AI 推理** | 看图理解内容 × 文章结构上下文 |
| 3 | **uploadIndex 上传顺序** | 用户粘贴的第 1 张大概率是最重要的图（封面候选） |

**同优先级内的冲突解决规则：**

| 冲突场景 | 解决规则 |
|----------|----------|
| 两张图文件名都含「封面」 | 取 `uploadIndex` 较小的（先上传的优先） |
| 一张含「hero」，另一张含「封面」 | 「封面」精确匹配优先于「hero」部分匹配（关键词精确匹配 > 部分匹配） |
| AI 推理同时命中多个候选 slot | 取置信度最高的；若置信度相同，按 uploadIndex 顺序分配 |
| 生成后用户新增图片 | 新图片进入未分配池；**除非用户明确指令，否则不重新分配已有映射** |

> **重要认知：** v1.0 的图片映射不需要在生成前做到完美；它只需要做到「减少校正次数」。文件名关键词 + 上传顺序这两个零成本信号，可以把随机分配的校正率拉低到可接受范围。用户真正表达图片意图的场景是生成后——「封面换成这张」「第 3 页的图用图 2」。

## 9. 生成后：预览与微调阶段

### 9.0 微调指令分类

在进入微调子场景之前，必须区分用户指令的**处理方式**——并非所有「微调」都是编排 Skill 自己能完成的：

| 类别 | 示例 | 处理方式 | 是否触发排版 Skill |
|------|------|----------|-------------------|
| **纯文本替换** | 「标题改成 XXX」「删第二段」 | 直接替换 Recipe HTML 中对应占位符内容，面板即时更新 | **否** |
| **样式微调** | 「字大一点」「行距加大」「颜色深一点」 | **P2 · v1.0 暂不实现。** v1.0 微调仅支持纯文本替换 + 结构变更（换 Recipe）。v1.x 待 Template 的 CSS 变量标记「用户可调」标签后，编排 Skill 可在 CSS 变量层面调整 | **否**（v1.x） |
| **结构变更** | 「紧凑一点」「换一个更密的版式」 | 在同类型 Recipe 中选择 `contentLimits` 更严格的变体；若用户表达的是「换一个版式」则触发 Recipe 替换 | **否**（查找 Recipe） |
| **语义重排** | 「这段拆成两页」「这两页合并」「重新分配图片」 | 需要排版 Skill 重新推理内容结构和版式选择 | **是** |

> **编排 Skill 的责任：** 识别用户指令属于哪个类别。纯文本替换可以立即响应（< 100ms）；结构变更在同一风格的版式库中查找即可，无需 AI；语义重排需要调用排版 Skill（需提示用户等待）。**样式微调（「字大一点」「行距加大」等）v1.0 暂不实现——详见 §9.0 分类表中的标注。**

### 9.1 生成完成：首版展示

排版 Skill 返回后，编排 Skill 将 HTML 推送到右侧面板，聊天中展示摘要：

```
编排 Skill: 全部完成 ✨ 7 页排版好了，右边翻翻看 ↓

          [封面]   [图文]   [图文]   [清单]
          [数据]   [引语]   [收束]

          封面标题我稍微压缩了一下（原文 18 字 → 12 字），你看看行不行？
```

- 全部页面 HTML 推送到右侧面板，用户可逐页浏览
- 聊天中展示缩略图网格，每张标注页面类型
- 若有 warnings，以温和语气在消息末尾附上
- 若全部 PASS（无 warnings），值得一句轻量的肯定：「排版很规整，没有检测到任何问题 👍」

### 9.2 文字修改：边说边看

有 HTML 预览面板后，文字修改的体验从「盲改 → 等图」变成「边说边看」：

```
User: 第 3 页标题改成「2026 AI 趋势全解析」
编排 Skill: 已修改 → [面板即时更新第 3 页 HTML]
          标题略长，右边能看到溢出效果。还行吗？
User: 再短一点
编排 Skill: [面板即时更新为「AI 趋势全解析」]
User: 还是太长，「AI 趋势」吧
编排 Skill: [面板即时更新]
User: 这个好，就这样
编排 Skill: (此时才触发 PNG 渲染，发最终缩略图到聊天)
```

**编排 Skill 的行为规则：**

| 场景 | 编排 Skill 行为 |
|------|----------------|
| 用户发出文字修改指令 | 立即推送新 HTML 到面板，聊天中仅回复「已更新 →」或「看看右边」，不渲染 PNG |
| 同一页连续修改 | 每次即时推 HTML，不附加任何聊天消息（避免刷屏），直到用户说「就这样」才触发 PNG |
| 用户说「可以了」「就这样」「渲染吧」 | 触发 PNG 渲染，发缩略图到聊天，将最终 HTML 标记为「已确认」 |
| 修改后触发溢出 | 面板 HTML 正常渲染（包括溢出部分 + 视觉警告色），编排 Skill 在聊天中附一句：「标题略长，右侧可看到溢出效果」 |

### 9.3 换版式：即时换装

用户在聊天中选择候选版式 → 面板即时展示**填入了自己实际内容**的 HTML 版本。这消除了「缩略图用的是示例内容，和我的文字不搭」的问题。

```
User: 第 4 页换个版式
编排 Skill: 当前是「清单 M05」。同类型的可选版式：
          [M07] — 条目间距大，适合 3-4 项短清单
          [M11] — 紧凑网格，适合 5-8 项长清单
          点一下就能在右边看到你的内容装进去的效果。
User: M07
编排 Skill: [面板即时切换到 M07 HTML，填入用户实际内容]
          已预览 M07。效果如何？
User: 不太好，试试 M11
编排 Skill: [面板即时切换到 M11 HTML]
User: 这个不错，就它了
编排 Skill: (触发 PNG 渲染，发缩略图)
```

**并排对比：**

```
User: M07 和 M11 对比看看
编排 Skill: [面板进入并排对比模式，左侧 M07，右侧 M11，内容相同]
          左边 M07 · 右边 M11。选哪个？
User: M11
编排 Skill: [面板退出对比模式，全屏展示 M11]
```

### 9.4 批量修改

有 HTML 预览面板后，批量修改变得极其自然：

```
User: 改几个地方：
      第 3 页标题 →「2026 AI 趋势」
      第 5 页正文删第二段
      第 7 页行动号召 →「立即订阅」

编排 Skill: 已全部应用 → [面板自动切到第 3 页，用户可以逐页检查]
          3 处修改已在右侧预览。要逐个看看，还是全部确认？
User: 逐页看看
编排 Skill: [面板依次切到第 3 页 → 第 5 页 → 第 7 页]
User: 第 5 页的第二段好像还在？
编排 Skill: 抱歉漏了 → [面板更新第 5 页]
User: 好了，全部确认
编排 Skill: (批量渲染 3 页 PNG，发缩略图)
```

### 9.5 配色切换：面板直接操作

生成前用默认配色渲染。用户在面板上看到结果后，**面板顶部提供一个配色下拉选择器**，列出当前风格的所有可用配色（名字 + 色条预览），选中后即时注入对应 CSS 变量并重绘全部页面。

```
面板右上角：
  ┌──────────────────┐
  │ ██ 午夜墨    ▾   │
  │ ██ 暖纸经典      │
  │ ██ 薄荷晨光      │
  │ ██ 琥珀金        │
  └──────────────────┘
```

这不需要经过聊天——配色切换是纯视图操作，不触发排版 Skill、不改变版式序列、不影响文案。编排 Skill 不需要就配色切换发任何消息，除非用户主动在聊天中讨论配色：

```
User: 还有哪些配色？
编排 Skill: 这个风格有 4 套，右边面板顶部可以切换。
```

### 9.6 换图片

```
User: 封面换这张 (Cmd+V 粘贴一张新图片)
编排 Skill: [面板即时更新封面图片]
          已替换。看看右边？
User: 第 3 页的图换成图 2
编排 Skill: [面板即时更新第 3 页]
User: 可以了，渲染吧
编排 Skill: (触发 PNG 渲染)
```

### 9.7 全局重新生成

```
User: 整体换成瑞士网格重新生成
编排 Skill: 换成「瑞士网格」风格，之前的版式选择会全部清除。确认吗？
User: 确认
编排 Skill: (调用排版 Skill，传入新 styleId)
          正在理解文章结构…
          全部完成，共 7 页 ↓ [面板更新为全新 7 页 HTML]
```

#### 9.7.1 全局重新生成与用户手动修改的冲突处理

全局重新生成（换风格或触发排版 Skill 重新推理）时，用户此前的手动修改按以下策略处理：

| 修改类型 | 处理策略 | 说明 |
|----------|----------|------|
| **文字手动修改**（改过的标题、删过的段落） | **回写至 F2 body / F8 constraints**，排版 Skill 基于修改后的数据重新推理 | 用户的文字修改是「对内容的修正」，应该保留 |
| **版式手动替换**（换过的 Recipe） | **清除** | 全局重新生成的本质就是「重新选择版式」，保留旧选择没有意义 |
| **图片手动替换**（换过的 slot 图片） | **保留映射关系**，排版 Skill 在此基础上重新分配其余图片 | 用户显式指定的图片映射是「编辑意图」，不应丢失 |
| **配色手动切换** | **保留**（若新旧风格有同名配色则迁移，否则使用新风格默认配色） | 配色是用户审美偏好 |

> **编排 Skill 提示文案：** 「标题和正文的修改会保留，版式选择将重新进行。配色和图片映射也会尽量保留。」

## 10. 错误处理与恢复

### 10.1 核心思路

HTML 预览面板让「失败」从通知变成可操作的起点。纯聊天模式下，错误是一段文字——用户读完，然后想「那我该怎么办」。有了面板，**成功的页面已经在面板上可见**，失败的页面可以是一张简洁的占位卡，用户不用打字就能触发修复。

### 10.2 partial 状态：面板展示已有成果

排版 Skill 返回 `status: partial`（如 4 页成功 + 第 5 页失败）。

**面板：展示已有成果 + 简洁标注缺失**

```
面板中：
  [第 1 页 HTML]   [第 2 页 HTML]   [第 3 页 HTML]   [第 4 页 HTML]

  ┌─────────────────────────────┐
  │  第 5 页                     │
  │  ⚠️ 当前风格缺少此类型版式    │
  └─────────────────────────────┘
```

没有按钮。用户翻看前 4 页看到的是真实内容、排版效果——知道系统能跑出来好东西。

**聊天：同时呈现两条路径，说明各自代价**

风格是全局的，不存在「只给第 5 页换一个风格」的操作。两条路径都有成本：

```
编排 Skill: 4 页完成了 ✓ 第 5 页（数据展示）在当前风格下没有合适的版式。

          [跳过这页，导出 4 张]  →  丢一页，但不影响已完成的

          [换瑞士网格]  →  完整覆盖全部类型，但前 4 页也需要重新生成

          你怎么选？
```

| 用户选择 | 编排 Skill 行为 | 面板反应 |
|---------|----------------|---------|
| 跳过这页 | 移除第 5 页 → 确认导出 | 面板收缩为 4 页 |
| 换瑞士网格 | 确认 → 切换风格 → 重新调用排版 Skill | 面板更新为全新 5 页 |

### 10.3 failed 状态：聊天承载错误信息

排版 Skill 返回 `status: failed`（全部失败），没有产出任何 HTML——面板没有内容可渲染。

**在此状态下，不应在面板上构建交互式错误诊断界面。** 原因：

1. 实现成本与触发频率不成比例——为一种低频边缘情况构建带按钮的交互面板
2. 违反双通道原则——`failed` 时没有视觉结果可感知，错误信息本质上是「决策」信息，天然属于聊天通道
3. 效果并不更好——一段简洁的聊天消息比一个复杂的错误面板更直接

**正确做法：** 面板保持空置或显示简短状态提示。编排 Skill 根据排版 Skill 返回的 `analysis` 字段生成精准的聊天消息：

```
编排 Skill: 「电子杂志风」缺两种版式（对比和数据），所以这次没能生成。
          你的文章是「对比型」结构，换瑞士网格就能完整覆盖。
          要试试吗？
```

### 10.4 渲染失败：面板天然是降级方案

当 PNG 渲染失败时，面板渲染的是 HTML（不需要截图管线），它与 PNG 渲染走的是两条不同的技术路径。

| 失败场景 | 面板行为 | 聊天行为 |
|---------|---------|---------|
| 字体加载超时 → PNG 失败 | HTML 面板仍可渲染（用系统回退字体），内容可查看 | 「第 3 页渲染异常（可能是字体问题）。右边 HTML 版本可以看内容，已重试中。」 |
| 浏览器引擎崩溃 | 面板也失效 | 如实告知：「渲染出了点问题，建议重启应用。」 |
| 单页 PNG 失败，其余正常 | 面板中该页正常展示 HTML，其他页 PNG 缩略图正常 | 只通知失败页，不干扰其余页 |

**静默重试策略：**

```
PNG 渲染失败
  → 面板立即展示该页 HTML（用户不被打断）
    → 后台自动重试 PNG 渲染（不通知用户）
      → 成功 → 聊天中静默更新缩略图
      → 失败 → 第二次仍失败才在聊天中提示，但面板上始终可以看到 HTML 版本
```

### 10.5 三种状态的对话文案对照

| 排版 Skill 返回 | 编排 Skill 聊天文案 | 面板行为 |
|----------------|-------------------|---------|
| `success` | 「全部完成 ✨ 7 页在右边，翻翻看。」 | 全部页面 HTML 渲染 |
| `success` + 2 warnings | 「7 页完成。封面标题略长，第 4 页密度偏低——右边标出来了。」 | 正常 HTML + 对应页面叠加视觉标注 |
| `partial` | 「4 页完成了 ✓ 第 5 页在当前风格下没有合适版式。[跳过，导出 4 张] / [换瑞士网格]」 | 成功页正常 HTML + 缺失页简洁标注（无按钮） |
| `failed`（内容太短） | 「这篇文章比较短（XX 字），拆不成多页。要不要试单张封面 + 摘要卡？」 | 面板保持空置或简短状态提示 |
| `failed`（风格缺版式） | 「电子杂志风缺两种版式（对比和数据）。你的文章是『对比型』结构，换瑞士网格就能全覆盖。试试？」 | 面板保持空置或简短状态提示 |
| `failed`（AI 异常） | 「出了点意外，可能是文章结构比较特殊。换个风格或手动选版式试试？」 | 面板保持空置或简短状态提示 |

## 11. 导出与完成

### 11.1 导出流程

```
User: 整体确认，全部渲染
编排 Skill: (全部受影响页批量渲染)
          全部确认 ↓
          [7 张缩略图网格]
User: 导出
编排 Skill: (检查校验结果：无 FAIL)
          已导出 ✓ 7 张图都在桌面的「马尔代夫攻略」文件夹里。
          发布顺利 🚀
```

### 11.1.1 导出前置检查清单

导出前自动执行以下检查，全部通过后执行批量导出：

| 检查项 | 内容 | 不通过时的行为 |
|--------|------|---------------|
| ① 版本一致性 | 对比各页的「最后确认版本」与「面板当前版本」——若不一致，先触发增量 PNG 渲染 | 自动触发渲染，完成后继续导出 |
| ② 校验结果 | 执行全部校验规则（R1-R7）——若有 FAIL，在聊天中列出并询问 | 用户确认后仍可导出（FAIL 不阻止导出，但明确告知风险） |
| ③ 资源可访问性 | 确认所有图片文件可访问、输出目录可写 | 失败则提示具体原因（如「桌面不可写，请选择其他导出位置」） |
| ④ 输出完整性 | 确认所有页面均已渲染为 PNG（无缺失页） | 缺失页自动触发渲染后继续 |

### 11.2 导出交互

| 用户指令示例 | 编排 Skill 行为 |
|-------------|----------------|
| 「全部导出」或「导出到桌面」 | 批量导出 → 反馈路径和文件数量 |
| 「只导出第 1 张和第 5 张」 | 单页导出 ×2 → 反馈文件名 |
| 「导出为 JPEG 质量 80%」 | 切换格式和质量参数导出 |
| 「导出到 /Users/xxx/工作报告/」 | 指定路径导出 |
| 导出时存在 FAIL | 先提示，用户确认后才导出 |

## 13. 用户语言与系统语言的映射

### 13.1 用户不需要理解架构

| 系统术语 | 用户语言（编排 Skill 发言中使用的词） |

- Contract、Recipe ID（除非用户主动要求）

## 14. 预览面板交互设计

### 14.1 面板能力需求

| # | 能力 | 优先级 | 说明 |
|---|------|--------|------|
| PNL-01 | 接收 HTML 推送 | P0 | 编排 Skill 通过产品层接口向面板推送任意页的 HTML 内容 |
| PNL-02 | 逐页切换 | P0 | 上一页/下一页导航，响应编排 Skill 的「切到第 N 页」指令 |
| PNL-03 | 缩放控制 | P1 | 50% / 100% / 适应面板宽度 |
| PNL-04 | 并排对比 | P1 | 两栏模式，左右同时渲染两个 HTML（如同一页的两个版式） |
| PNL-05 | 高亮当前修改区域 | P1 | 编排 Skill 推送 HTML 时附带「高亮 selector」，面板在新渲染的 HTML 上做短暂高亮闪烁 |
| PNL-06 | 内联文字编辑 | P2 | 用户在面板中双击文字区域直接编辑，修改回传给编排 Skill |
| PNL-07 | 溢出/校验视觉标注 | P1 | 面板叠加半透明警告层（如溢出区域红色虚线边框） |
| PNL-08 | 推送时保留面板状态 | P1 | 切换页面或推送新 HTML 时，保留用户的缩放比例和滚动位置 |
| PNL-09 | 配色实时切换 | P0 | 面板接收配色 ID → 注入对应 CSS 变量 → 即时重绘，不触发排版 Skill |

> PNL-04（并排对比）和 PNL-05（高亮变化区域）是「让面板从被动视图升级为主动比较工具」的关键能力，强烈建议 v1.0 包含。

#### 14.1.1 面板推送消息格式（编排 Skill → 产品层 → 面板）

编排 Skill 通过产品层向面板推送 HTML 时，使用以下消息格式：

```jsonc
// 编排 Skill → 产品层 → HTML 预览面板
{
  "action": "render",              // "render" | "highlight" | "compare" | "navigate"
  "posters": [                     // 全部页面（增量时可只含变更页）
    {
      "index": 0,
      "html": "<section>...</section>",
      "highlightSelector": ".h-xl" // 可选，面板在此元素上做短暂高亮闪烁
    }
  ],
  "activeIndex": 0,                // 面板应展示的页码（从 0 开始）
  "themeId": "midnight-ink"        // 当前配色 ID
}
```

| action 值 | 含义 | posters 内容 | 面板行为 |
|-----------|------|-------------|---------|
| `"render"` | 渲染页面 | 全部或增量页面 | 替换/更新指定页面的 HTML，切换到 activeIndex |
| `"highlight"` | 高亮区域 | 仅含需要高亮的页面 | 在指定 selector 上做短暂高亮闪烁（~1.5 秒） |
| `"compare"` | 并排对比 | 恰好 2 个 poster（同一页的不同版式） | 进入并排对比模式。**防御性约束：** `action: "compare"` 时 `posters` 数组长度必须为 2；若不为 2，面板忽略此消息并在日志中记录 |
| `"navigate"` | 仅切换页码 | 可为空 | 面板切换到 activeIndex，不改变 HTML 内容 |

### 14.2 面板不承载的能力

- **不承载复杂错误诊断界面**（failed 状态下保持空置，错误信息由聊天承载）
- **不承载确认对话框**（确认操作由聊天承载）
- **不承载文件管理**（导出路径、文件命名由聊天承载）
- **不承载图片上传入口**（图片上传通过聊天 ⊕ 或 Cmd+V）

## 15. 情感化设计

### 15.1 愉悦时刻

| 节点 | 编排 Skill 表达 |
|------|----------------|
| 首次生成完成 | 「全部完成 ✨ 7 页排版好了，看看右边 ↓」 |
| 全部 PASS（无 warnings） | 「排版很规整，没有检测到任何问题 👍」 |
| 用户连续微调 5 次以上 | 「这组图文打磨得很细致了 👀 准备导出了吗？」 |
| 导出完成 | 「已导出 ✓ 7 张图都在桌面的『马尔代夫攻略』文件夹里。发布顺利 🚀」 |
| 用户说「和上次一样」且编排 Skill 正确识别 | 「记得！上次是电子杂志风 + 午夜墨 + 小红书。这次一样吗？」 |

### 15.2 可恢复的挫折

当用户修改后不满意时，编排 Skill 主动缓解挫折感：

```
User: 还是换回之前那个版式吧

编排 Skill: 换回 M05 了 ↓
          [恢复后的缩略图]
          其实 M05 在这个内容上排版效果挺好的。要微调哪里，还是就这样？
```

### 15.3 语气规范

| 场景 | 语气 | 示例 |
|------|------|------|
| 正常引导 | 温和、有信息量 | 「好的。想用什么视觉风格？」 |
| 发现小问题 | 轻描淡写、不给压力 | 「封面标题我稍微压缩了一下，你看看行不行？」 |
| 严重问题 | 明确但不吓人 | 「检测到 1 个严重问题：第 2 页内容可能溢出。建议处理一下再导出。」 |
| 全部失败 | 诚实、给出路 | 「电子杂志风缺两种版式。你的文章换瑞士网格就能完整覆盖。试试？」 |
| 用户确认破坏性操作 | 清晰说明代价 | 「换风格会清除当前 7 页的版式选择。确认吗？」 |
| 导出成功 | 温暖、有成就感 | 「已导出 ✓ 发布顺利 🚀」 |

## 16. 渐进式披露与学习曲线

### 16.1 能力层级

v1.0 首次使用时仅暴露核心流程（选平台 → 选风格 → 传图片 → 生成 → 微调 → 导出）。高级功能（版式收藏/禁用、手动编排、序列模板、自定义配色）按用户行为信号自动解锁：

| 用户行为信号 | 编排 Skill 主动介绍 |
|-------------|-------------------|
| 连续 3 次手动换掉同一类型的版式 | 「我注意到你经常换掉清单页的版式。你可以把喜欢的版式标记为『收藏』——下次我会优先用收藏的版式。要不要设置一下？」 |
| 连续 5 次使用同一套设置 | 「你已经连续 5 次用杂志风 + 午夜墨了。要不要设为你的默认风格？」 |
| 手动编辑了每页的文案 | 「看起来你对文字控制有比较高的要求。要不要试试『手动编排模式』——你可以直接拖拽版式，自己决定每页用什么布局？」 |

### 16.2 首次使用引导

用户首次 @ 文档触发图文生成时，编排 Skill 以欢迎语气简要介绍能力边界：

```
编排 Skill: 嗨！我可以帮你把文章排成小红书图文或公众号封面。
          你只要告诉我：发到哪个平台、喜欢什么风格。
          我们现在开始？
```

不暴露功能清单、不展示设置项、不介绍高级概念。

## 17. 移动端 UI 适配原则（架构预留）

> 本节描述的是 **UI 层面**的适配原则，不要求 v1.0 实现，但架构上预留适配空间。
>
> 真正的架构层面预留（数据协议、Skill 上下文、渲染适配）见下方 17.2。

### 17.1 桌面与移动的关键差异

| 桌面聊天 | 移动聊天 | 影响 |
|---------|---------|------|
| Cmd+V 粘贴图片 | 从相册选择或拍照 | 需要明确的「添加图片」入口 |
| 并排展示 5 张风格卡片 | 屏幕宽度仅容纳 1-2 张 | 必须采用轮播或堆叠布局 |
| 缩略图网格 7 张可看清 | 7 张小图在手机上无法看清细节 | 需要默认展示 2 列布局，并提供「查看大图」入口 |
| 导出到桌面路径 | 导出到相册或文件 App | 导出路径选择逻辑完全不同 |
| 右侧 HTML 预览面板 | 面板需切换为全屏或半屏 overlay | 双通道模型需适配单屏 |

### 17.2 架构预留项（v1.0 不实现，但数据协议应预留）

| 预留项 | 说明 |
|--------|------|
| **数据协议 `projectId` 顶层字段** | 所有协议（Skill 输入/输出、面板推送）的顶层均预留 `projectId` 字段，为多项目并发做准备 |
| **Skill 上下文感知** | 编排 Skill 的对话逻辑无需感知「当前是桌面还是移动」——双通道模型统一了指令语义 |
| **渲染适配** | HTML 预览面板在移动端使用与桌面相同的 HTML，通过 CSS `@media` 或 `viewport` 适配小屏（面板变为全屏 overlay） |
| **输出规格扩展** | `targetSpecs` 枚举预留移动端特有规格（如 `xhs_mobile`），当前仅 `xhs` 和 `wechat_pair` |

### 17.3 UI 适配原则

- 所有图片消息同时提供缩略图和「查看大图」入口
- 风格选择在移动端自动降级为 2 列或单列轮播
- 导出路径在移动端默认为相册（小红书图文）或文件 App（公众号封面）
- 预览面板在移动端切换为可展开/收起的底部抽屉

## 18. MVP 范围与优先级

### 18.1 优先级定义

| 标记 | 含义 | 说明 |
|------|------|------|
| **P0** | Must — MVP 必须 | v1.0 必须上线。缺失则产品不可用。 |

### 18.2 MVP 最小闭环（P0 汇总）

用户从输入内容到导出第一组图文的最小闭环：

| 步骤 | 所需功能 |
|------|----------|
| ① 输入与引导 | 用户 @ Markdown 文档 → 编排 Skill 识别意图；对话中收集：平台、风格、图片 |
| ② 风格选择 | 轮播卡片（A/B/C/D/E），用户回复字母选择 |
| ③ 排版生成 | 编排 Skill 将 F1-F8 传给排版 Skill → 排版 Skill 输出结构化 Poster 数组 |
| ④ HTML 预览 | 面板接收 HTML 推送、逐页切换、缩放控制、配色实时切换、溢出标注 |
| ⑤ 微调 | 改文字/换版式/换图片——聊天下指令 + 面板即时反馈 |
| ⑥ 校验 | 自动校验，面板视觉标注 + 聊天摘要提醒 |
| ⑦ 渲染 | 用户确认后 PNG 批量渲染、字体等待、配色注入 |
| ⑧ 导出 | 批量导出、命名规则、导出前校验、导出路径 |
| — 组件体系 | 内置 5 个风格（各含 Contract + 12 种页面类型版式 + 4-6 配色）、页面类型系统、输出规格系统 |
| — 项目 | 新建/列表/打开/删除（项目数据模型见附录 D） |

> **MVP 完成定义：** 用户在 Agent 聊天窗口 @ 一篇 1000 字的 Markdown 文章 → 编排 Skill 询问平台和风格 → 用户选择 → 编排 Skill 调用排版 Skill 生成 → HTML 推送到右侧面板（< 30 秒）→ 用户在面板中看到 5-9 页排版 → 用户说「第 3 页标题改成 XXX」→ 面板即时更新 → 用户确认 → PNG 渲染 → 一键导出到桌面。全过程在聊天 + 面板双通道中完成。
>
> **v1.0 仅支持单项目操作。** 多项目并发切换在架构上预留 `projectId` 字段，但不实现。

### 18.3 P1（v1.0 尽量包含）

风格克隆与导入导出、版式自定义与导入导出、收藏/禁用、手动编排模式、配色自定义与冲突提示、图片备注与图库搜索、历史版本、项目复制、校验规则开关与阈值、设置默认值、并排对比模式（PNL-04）、高亮修改区域（PNL-05）、命名空间与依赖检查。

### 18.4 P2（v1.x / v2.0 可延后）

| 砍的顺序 | 模块 | 理由 |

## 20. 安全模型

### 20.1 组件导入安全

三方 `.style`、`.recipe` 文件可能包含恶意代码。导入时执行：

| 检查项 | 策略 |
|--------|------|
| **HTML sanitize** | 移除所有 `<script>` 标签、`onerror`/`onload` 等事件处理器、`javascript:` 协议 URL |
| **CSS sanitize** | 渲染时通过 CSP 限制：禁止 `url()` 引用外部资源、禁止 `behavior` 属性、禁止 `expression()` |
| **文件大小限制** | 拒绝 > 10MB 的组件文件（防止解压炸弹） |
| **HTML 嵌套深度限制** | 拒绝深度 > 50 层的 DOM 嵌套 |
| **外部资源阻止** | 渲染时阻止所有外部网络请求 |

### 20.2 身份验证

| 场景 | 策略 |
|------|------|
| **官方组件** | 使用数字签名。导入时验证签名，未通过则标记「来源不可信」 |
| **社区组件** | `author` 字段为 `@username` 格式。`official` 前缀仅限签名验证通过的组件 |
| **本地组件** | 用户自己创建/修改的组件，无签名，标记「本地创建」 |

### 20.3 渲染沙箱

- 渲染引擎运行在隔离的浏览器上下文中
- 禁止访问 `file://` 协议
- 禁止弹出窗口和导航
- 渲染完成后销毁上下文

## 21. 技术约束与性能指标

### 21.1 渲染引擎

| 约束 | 值 |
|------|-----|
| 浏览器引擎 | Playwright 或等价方案，内置免安装 |
| HTML 推送延迟 | < 100ms |
| 渲染 5 张 1080×1440 PNG | ≤ 15 秒 |
| 并发渲染数上限 | 默认 2（用户可在设置中调整） |
| 单页重渲（同版式） | < 1 秒 |
| 字体 | 内置常用中文字体（思源黑体/宋体），不依赖 Google Fonts 等外部 CDN |
| 离线能力 | 核心功能完全离线可用。图库搜索、AI 图片生成需要网络——离线时灰显入口 |

### 21.2 网络依赖降级

| 功能 | 依赖网络 | 离线行为 |
|------|----------|----------|
| 内容输入 | 否 | 正常 |
| 风格 / 版式 / 配色管理 | 否 | 正常 |
| HTML 预览面板 | 否 | 正常 |
| 渲染与导出 | 否 | 正常 |
| 图库搜索 | **是** | 入口灰显 + 提示「需要网络连接」 |
| AI 图片生成 | **是** | 入口灰显 + 提示「需要网络连接」 |

### 21.3 性能预算

| 操作 | 目标 | 测量条件 |
|------|------|----------|
| 应用冷启动 | < 3 秒 | 进程启动到主窗口可交互 |
| HTML 推送到面板 | < 100ms | 编排 Skill 发起推送到面板渲染完成（面板与产品层同进程，IPC 延迟不计） |
| 配色切换 | < 100ms | CSS 变量注入到全部页面重绘（不含复杂 CSS 滤镜/混合模式的风格） |
| 排版 Skill 生成 7 页 | < 30 秒 | 云端推理（GPT-4o 级别模型）；本地模型（如 7B 参数量化版）参考目标 < 60 秒，待实测后调整 |
| 7 页批量 PNG 渲染 | < 15 秒 | 不含字体首次加载（首次加载另计 ≤ 5 秒）；字体已缓存时测量 |
| 单页 PNG 重渲 | < 3 秒 | 触发到新 PNG 完成（含字体回退场景 ≤ 5 秒） |
| 项目打开（含 7 页历史） | < 2 秒 | 点击项目到预览展示（本地读取，不含网络） |

### 21.4 项目可移植性

| 约束 | 说明 |
|------|------|
| 所有路径为相对路径 | `project.json` 中 `assets/`、`output/` 路径均为项目根目录的相对路径 |
| 项目目录可移动 | 用户可自由移动/重命名项目目录 |
| 跨平台兼容 | 项目目录在 Windows/macOS 间可直接移动（路径分隔符自动适配） |

## 附录 D：项目数据模型

> 本节定义 Project（项目）的数据结构、生命周期和存储格式。这是工程团队设计存储层的前置输入。

### D.1 项目生命周期

```
创建 → 排版（可多次生成/微调） → 导出 → [可选：归档/继续编辑]
```

- 项目创建于用户首次 @ 文档触发图文生成时
- 导出后项目仍然可编辑——用户可以回到项目继续微调并再次导出
- 项目可被用户手动归档（标记为「已完成」但保留在项目列表中）

### D.2 项目目录结构

```
{project-name}/
  ├── project.json          # F1-F8 数据 + 当前 Style/Theme + 版本号
  ├── posters.json          # Skill 输出的 Poster 数组（含用户手动修改的增量 diff）
  ├── assets/               # 用户上传的图片（原始文件）
  └── output/               # 导出的 PNG
```

> **v1.0 不实现 `history/` 目录。** 历史版本是 P1 功能，v1.0 架构上预留但仅保存当前版本。

### D.3 project.json 结构

```jsonc
{
  "projectVersion": "1.0",
  "projectId": "uuid",
  "name": "马尔代夫攻略",
  "createdAt": "2026-06-03T10:00:00Z",
  "updatedAt": "2026-06-03T10:30:00Z",
  "data": {
    "targetSpecs": ["xhs"],           // F1
    "body": "# 马尔代夫完全攻略\n\n...", // F2（可能含用户修改后的版本，见 §9.7.1）
    "images": [ImageAsset],           // F3
    "tone": "travel",                 // F4
    "stylePreference": "editorial-magazine", // F5
    "themePreference": "midnight-ink", // F6
    "metadata": {},                   // F7
    "constraints": ""                 // F8
  },
  "activeStyleId": "editorial-magazine",
  "activeThemeId": "midnight-ink"
}
```

### D.4 posters.json 结构

```jsonc
{
  "posters": [
    {
      "index": 0,
      "pageType": "cover",
      "recipeId": "M01",
      "spec": "xhs",
      "html": "<section>...</section>",    // 当前 HTML（含用户手动修改）
      "originalHtml": "<section>...</section>", // Skill 原始输出（用于 diff 和回退）
      "imageAssignments": [
        { "slotId": "HERO_IMG", "assetId": "img-003" }
      ],
      "userModifications": [            // 用户手动修改的增量记录
        { "type": "text", "selector": ".h-xl", "oldValue": "...", "newValue": "...", "timestamp": "..." },
        { "type": "recipe", "oldRecipeId": "M01", "newRecipeId": "M07", "timestamp": "..." }
      ],
      "status": "confirmed"             // "draft" | "confirmed" | "rendered"
    }
  ]
}
```

### D.5 项目的可移植性

| 约束 | 说明 |
|------|------|
| 所有路径为相对路径 | `project.json` 和 `posters.json` 中 `assets/`、`output/` 路径均为项目根目录的相对路径 |
| 项目目录可移动 | 用户可自由移动/重命名项目目录，内部引用不失效 |
| 跨平台兼容 | 路径分隔符在 Windows/macOS 间自动适配 |
