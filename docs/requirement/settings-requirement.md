# 设置界面需求文档

## 1. 设置界面整体结构

设置界面采用 Modal 弹窗 + Tab 分页的形式，宽度 560px。大模型相关配置已拆分为两个 Tab，详见 [llm-service-profiles-requirement.md](./llm-service-profiles-requirement.md)。

| Tab | 名称 | 包含内容 |
|-----|------|---------|
| 通用 | 通用 | 工作目录、界面主题、默认聊天模式、并行会话上限 |
| 大模型服务 | 大模型服务 | 多套服务卡片（名称 + Key + URL 一体）、切换当前使用、卡片内测试连接 |
| 默认大模型设置 | 默认大模型设置 | 模型列表、Temperature、最大输出 tokens、默认开启 Thinking |
| 工具 | 工具 | 内置工具、Python 路径等 |
| Skill | Skill | Skill 配置 |

### 1.1 界面样式

- Modal body 内正文字体统一为 12px（Modal 标题、Tab 标签、区块小标题、按钮除外）
- 模型列表标题行右侧放置操作按钮，与表单控件右边缘对齐

## 2. 通用 Tab

### 2.1 工作目录

| 需求点 | 描述 |
|--------|------|
| 输入框 | 显示当前工作目录路径，支持手动输入 |
| 目录选择 | 输入框右侧提供按钮，点击后弹出系统文件夹选择器，支持新建文件夹 |
| 写入权限校验 | 用户设置完工作目录后，主动检查应用是否有写入权限；若无权限，在输入框下方显示红色错误提示，说明该目录不可用（无法写入），需更换 |
| 校验时机 | (1) 通过选择器选择目录后立即校验；(2) 手动输入路径后失焦时校验；(3) 保存时再次校验，不可写则阻止保存 |
| 必填 | 工作目录为必填项 |

## 3. 大模型相关 Tab

> **说明：** 自 v1.1 起，原「大模型」Tab 拆为「大模型服务」与「默认大模型设置」。API Key、Base URL、多套服务切换、测试连接等详见 [llm-service-profiles-requirement.md](./llm-service-profiles-requirement.md)。本节保留 **默认大模型设置** 中的模型列表与生成参数规格。

### 3.1 模型列表

模型列表替代了原来的单个"模型"和"默认模型"文本输入，改为可视化的列表管理界面。

#### 3.1.1 模型数据结构

每个模型条目包含以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 唯一标识 |
| name | string | 模型名称（不可重复） |
| maximumContext | number | 最大上下文长度 |
| maxTokens | number | 最大输出长度 |
| isDefault | boolean | 是否为默认模型（列表中只能有一个默认，且列表非空时必须有一个） |
| isFast | boolean | 是否为快速模型 |
| enabled | boolean | 是否启用（禁用后该模型不可用于聊天，半透明显示） |

#### 3.1.3 列表显示

- 每个模型占两行显示：
  - **第一行**：启用开关（Switch）+ 模型名称 + "默认"标签（蓝色）+ "快速"标签（橙色）+ 删除按钮
  - **第二行**（缩进）：上下文长度 + 输出长度 + "默认"复选框 + "快速"复选框
- 禁用的模型整行半透明（opacity: 0.45），复选框不可操作
- 大数字友好显示：1000000 → 1M，262144 → 262.1K，8192 → 8.2K

#### 3.1.4 操作按钮

位于"模型列表"标题行的右侧，与表单控件右边缘对齐：

| 按钮 | 图标 | 行为 |
|------|------|------|
| 恢复默认 | mingcute refresh_2_line | 将模型列表重置为内置默认列表 |
| 添加模型 | mingcute add_line | 点击弹出 Popover 浮层，供用户输入新模型信息 |

#### 3.1.5 添加模型 Popover

包含以下输入项：

| 字段 | 控件 | 说明 |
|------|------|------|
| 模型名称 | Input | 必填，不可与已有模型重名 |
| 最大上下文 | InputNumber | 默认值 200000 |
| 最大输出 | InputNumber | 默认值 8192 |
| 默认 | Checkbox | 勾选后，原有默认模型自动取消默认 |
| 快速 | Checkbox | 标记为快速模型 |
| 确认 | Button | 验证通过后添加到列表 |

#### 3.1.6 默认模型逻辑

- 列表中只能有一个默认模型
- 列表非空时，必须有一个默认模型
- 勾选某个模型为默认时，自动取消其他模型的默认
- 删除默认模型后，自动将列表中第一个模型设为默认
- 禁用默认模型时，自动将默认角色转移给下一个启用的模型
- 保存时校验：至少需启用一个模型

### 3.2 Temperature

| 需求点 | 描述 |
|--------|------|
| 控件 | InputNumber |
| 范围 | 0 ~ 2 |
| 步长 | 0.1 |

### 3.3 默认开启 Thinking

| 需求点 | 描述 |
|--------|------|
| 控件 | Switch |
| 默认值 | 开启（true） |

| 按钮 | 行为 |
|------|------|

## 5. 数据存储与兼容性

### 5.1 存储方式

- 模型列表以 JSON 字符串存储在 `config.models` 键中
- 保存模型列表时，自动从默认模型条目同步 `config.model` 和 `config.defaultModel` 字段（向后兼容）
- `thinkingEnabled` 存储在 `config.thinkingEnabled` 键中

### 5.2 迁移策略

- 首次加载时若 `config.models` 为空，从旧的 `config.model` / `config.defaultModel` 字段或使用内置默认列表自动生成
- `AppConfig` 类型中保留 `model` / `defaultModel` 字段，供 `ChatView` 等模块直接使用
