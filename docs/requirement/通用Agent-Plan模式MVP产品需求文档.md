# 通用 Agent Plan 模式 MVP 产品需求文档

> **状态：已废弃 — 见 [remove-plan-mode-requirement.md](./remove-plan-mode-requirement.md)**

> 版本：v2.0（MVP 阶段）
> 目标：在执行前插入「用户主动触发的规划+审批」环节，解决 Agent 方向跑偏、返工成本高的问题。
> 设计原则：先跑通单线流程，再逐步叠加并行、自治、审查等高级能力。

---

## 一、MVP 要解决的核心问题

> **Agent 在未充分理解需求、未与用户对齐方案时，就直接修改文件/数据/系统，导致不可逆的错误和反复返工。**

Plan 模式 MVP 不做「自动化调度」「分层自治」「智能审查」，只做一件事：**提供一种用户主动选择的、有审批闸门的工作模式，让 Agent 先规划再执行。**

---

## 二、MVP 核心流程（四步走）

### 2.1 流程图

```text
用户输入需求 + 选择模式
    │
    ├─ 普通模式 → 直接执行（与现有行为一致，无审批）
    │
    └─ Plan 模式 →
        │
        ▼
    [Step 1: Plan Mode（探索期）]
        ├── Agent 进入只读模式（禁止写入）
        ├── 探索环境、理解需求、设计推荐方案
        ├── 生成结构化计划文件（Plan Doc）
        │   └── 探索后可能判断：无需计划 / 不可行 → 终止并报告
        │
        ▼
    [Step 2: 审批闸门] ──→ 用户拒绝 → Agent 迭代计划（生成新版本）
        │                      用户批准 → 进入执行
        │                      用户变更需求 → Agent 评估后决定迭代计划或就地调整
        ▼
    [Step 3: 执行与交付]
        ├── Agent 按已批准的计划逐步执行
        ├── 执行中若发现计划不可行 → 暂停并上报
        ├── 执行中用户可提出变更 → Agent 评估影响后决定是否需重新规划
        └── 完成后提交结果，退出 Plan 模式
```

### 2.2 模式选择机制

**核心决策：用户主动选择，Agent 不自动判断复杂度。**

- **为什么不用 Agent 自动判断**：
  1. Agent 存在利益冲突——可能倾向于将任务归类为"简单"以减少交互摩擦
  2. 用户往往不具备评估任务复杂度的能力（这正是他们需要 Agent 的原因）
  3. 一旦归类为"简单任务"，审批闸门形同虚设

- **如何实现**：在消息输入框的发送按钮旁放置模式选择器，提供「普通模式」和「Plan 模式」两个选项，默认为普通模式。

- **好处**：
  1. **零歧义**：不存在"谁来判断复杂度"的问题
  2. **零摩擦**：普通模式下 Agent 行为与现有完全一致，不增加任何额外步骤
  3. **渐进式采纳**：用户在不确定时用普通模式快速试错，在重要任务时切到 Plan 模式获得保障
  4. **MVP 实现简单**：只需在 UI 层加一个模式选择器，后端根据模式决定是否启用权限隔离和审批流程

### 2.3 两种模式下的行为对比

| 行为 | 普通模式 | Plan 模式 |
|------|---------|----------|
| Agent 权限 | 全部工具（含写入） | 探索期只读，获批后读写 |
| 审批闸门 | 无 | 计划必须获批才可执行 |
| 工具确认机制 | 现有 `WriteConfirmCard` 逐次确认 | 获批后仍保留逐次确认（安全网不变） |
| 会话恢复 | 现有行为 | 可恢复计划状态继续执行 |
| 执行方式 | 单 Agent 直接执行 | Coordinator + Worker 隔离执行 |

## 三、MVP 功能清单（5 项核心 + 5 项产品化支撑）

### 3.1 核心功能

| 序号 | 功能 | 说明 |
|:---:|------|------|
| 1 | **模式选择器** | 用户在发送消息时主动选择「普通模式」或「Plan 模式」。选择 Plan 模式后进入规划流程。 |
| 2 | **两阶段权限切换** | 探索期只读（Read/Search/Grep），执行期读写（Edit/Write/Run）。权限在主进程 IPC 层隔离，不靠 Prompt 约束。 |
| 3 | **计划文件生成与审批闸门** | Agent 将探索结果写入标准 Markdown 计划文件，系统向用户展示结构化审批卡片。计划未获批，Agent 不获得写入权限。 |
| 4 | **Coordinator-Worker 隔离** | 协调者（Coordinator）负责与用户对话、拆解计划、策展上下文；执行者（Worker）在隔离上下文中执行任务。MVP 为单 Worker，核心价值是上下文隔离。 |
| 5 | **执行期异常上报与用户变更** | 执行中计划与实际不符时 Agent 暂停上报；用户也可在执行期主动提出变更，Agent 评估影响后决定迭代或调整。 |

### 3.2 产品化支撑

| 序号 | 功能 | 说明 |
|:---:|------|------|
| 6 | **计划存储与索引** | 计划文件存储在用户工作目录下，索引和状态纳入数据库管理 |
| 7 | **审批交互界面** | 聊天流中的嵌入式审批卡片（React + Ant Design） |
| 8 | **状态持久化与恢复** | 会话中断后，计划状态可从数据库恢复继续执行 |
| 9 | **权限隔离实现** | 基于实际工具集在主进程 IPC 层实现权限拦截 |
| 10 | **计划取消与终止** | 用户可中途取消计划，系统记录已执行进度 |

## 四、产品化支撑详细设计

### 4.1 SpaceAssistant 适配说明

> 文档 v1.x 按通用 SDK 设计撰写，v2.0 明确适配 SpaceAssistant 的 Electron 应用架构。

| 领域 | 适配方案 |
|------|---------|
| **审批界面** | React 组件（Ant Design），嵌入聊天流中，非终端卡片 |
| **权限控制** | 主进程 IPC 层（`electron/appIpc.ts`），在 IPC handler 中拦截被禁止的工具调用，非渲染进程层 |
| **状态存储** | 复用现有 JSON 数据库（`spaceassistant-data.json`），在 `Session.metadata` 中存储计划状态，而非独立的 `.agent/state/` 文件 |
| **计划文件** | 存储在用户工作目录下（`<workDir>/.spaceassistant/plans/`），随项目代码一起版本管理 |
| **模式选择** | 在 `MessageInput` 组件发送按钮旁的下拉选项，通过 IPC 通知主进程切换模式 |
| **"批准前修改"** | 两种方式：1) 在审批卡片中直接编辑关键字段；2) 在聊天中输入修改意见，Agent 据此更新计划 |

### 4.2 与现有功能的关系

| 现有功能 | Plan Mode 下的关系 |
|---------|-------------------|
| **会话管理** | 计划是会话级的——计划绑定到创建它的会话。同一时间每个会话最多一个活跃计划。不支持跨会话共享计划。 |
| **工具确认机制**（`WriteConfirmCard`） | 获批后的执行步骤仍需逐个确认。Plan 审批是「计划级」授权，`WriteConfirmCard` 是「操作级」安全网，两者互补。 |
| **引用文件列表**（`ReferencedFilesPanel`） | Plan Mode 下的文件操作同样计入引用列表，无需特殊处理。 |
| **配置管理** | 用户可在设置中配置默认模式（普通/Plan），也可在每次发送时覆盖。 |
| **多计划并行** | MVP 不支持。同一时间只能有一个活跃计划。在 MVP 边界中明确声明。 |

### 4.3 计划存储与索引

**存储位置**：用户工作目录下

```
<workDir>/                        # 用户配置的工作目录
└── .spaceassistant/                       # SpaceAssistant 项目级数据目录
    └── plans/                    # 计划文件目录
        ├── 2026-05-19-feature-a.md
        └── 2026-05-19-refactor-b.md
```

**为什么放在用户工作目录下**：
1. 计划文件与项目代码强关联，随项目一起版本管理，换电脑或团队共享时不丢失
2. 用户可以直接在 IDE 中查看和编辑计划文件，无需进入应用
3. `.spaceassistant/` 命名空间明确归属 SpaceAssistant，避免与其他工具冲突
4. 建议用户将 `.spaceassistant/` 加入 `.gitignore`（计划是过程文件，通常不需要入库）

**版本管理**：采用**单文件 + Frontmatter 版本号**，而非文件名编码版本号（`feature-a.v2.md`）。

- 每次迭代覆盖同一文件，Frontmatter `version` 递增
- 历史版本通过数据库中的 `plan_versions` 数组管理（保留版本号、创建时间、审批结果）
- 避免迭代次数多时目录混乱，也避免 `index.json` 的 `latest` 指针手动维护问题

**数据库扩展**：在 `Session.metadata` 中新增计划相关字段：

```typescript
// Session.metadata 中新增的字段
interface PlanMeta {
  planId: string              // 计划唯一 ID
  status: 'drafting' | 'awaiting_approval' | 'approved' | 'executing' | 'completed' | 'cancelled'
  planFilePath: string        // 计划文件路径
  currentStepIndex: number    // 当前执行步骤（0-based）
  stepsTotal: number          // 总步骤数
  version: number             // 当前版本号
  createdAt: number           // 创建时间
  approvedAt: number | null   // 审批时间
  cancelledAt: number | null  // 取消时间
  envSnapshot: {              // 审批时的环境快照（用于过期检测）
    gitHead: string | null    // 当前 git HEAD
    timestamp: number         // 快照时间
  }
}
```

**计划文件模板**：

```markdown
---
plan_id: plan-20260519-001
status: pending
version: 1
created_at: 2026-05-19T09:30:00Z
approved_at: null
steps_total: 5
steps_completed: 0
---
```

### 4.4 审批交互界面

**设计原则**：审批界面必须让用户在 **10 秒内**理解"做什么、影响多大、风险在哪"，不需要读完整篇 Markdown。

**界面形态**：聊天流中的嵌入式审批卡片（React + Ant Design 组件），非终端风格 ASCII 框。

**审批卡片内容**：

| 区域 | 内容 |
|------|------|
| 标题 | 任务名称（取自计划文件 `# 标题`） |
| 方案摘要 | 推荐方案的一句话概括、涉及文件数、步骤数 |
| 验收标准 | 关键验收标准列表（最多 3 条） |
| 风险提醒 | 风险点列表（如有） |
| 占位符警告 | 如果关键字段含占位符，高亮标注 |
| 操作按钮 | 「批准并执行」「修改计划」「拒绝并反馈」 |

**交互规则**：

| 用户操作 | 系统行为 |
|---------|---------|
| **批准并执行** | 标记 `status: approved`，记录环境快照，切换 Agent 到读写模式，开始执行 Step 1 |
| **修改计划** | 提供两种方式：1) 卡片内编辑关键字段；2) 在聊天中输入修改意见，Agent 据此更新计划。修改后生成新版本，再次提交审批 |
| **拒绝并反馈** | 保留当前版本，Agent 进入迭代模式。用户反馈写入计划文件 `## 8. 审批反馈` 章节，Agent 据此生成下一版本 |

**关键字段零占位符**：目标、方案、执行步骤这三个核心字段不允许出现「TODO」「TBD」「待确认」等模糊描述，否则审批卡片标注警告。其他字段（如风险点）可标注「需执行时确认」。

### 4.5 权限隔离实现

**基于实际工具集的权限表**：

SpaceAssistant 当前的 6 个内置工具（定义在 `src/shared/builtinToolDefinitions.ts`）：

| 工具 | 风险等级 | Plan Mode 探索期 | Plan Mode 执行期 | 普通模式 |
|------|---------|-----------------|-----------------|---------|
| `read_file` | 低 | ✅ 允许 | ✅ 允许 | ✅ 允许 |
| `list_directory` | 低 | ✅ 允许 | ✅ 允许 | ✅ 允许 |
| `grep` | 低 | ✅ 允许 | ✅ 允许 | ✅ 允许 |
| `edit_file` | 中 | ❌ 禁止 | ✅ 允许（需确认） | ✅ 允许（需确认） |
| `write_file` | 中 | ❌ 禁止 | ✅ 允许（需确认） | ✅ 允许（需确认） |
| `run_script` | 高 | ❌ 禁止 | ✅ 允许（需确认） | ✅ 允许（需确认） |

**`run_script` 在探索期禁止的原因**：
- `run_script` 可执行任意代码，理论上可间接写入文件，是权限隔离的潜在绕过点
- 探索期所需的环境信息（如依赖版本、项目结构）可通过 `read_file` + `list_directory` + `grep` 获取
- 执行期仍需逐次确认（复用现有 `WriteConfirmCard` 机制），作为操作级安全网

**实现方式**：在主进程 IPC 层拦截

```typescript
// electron/appIpc.ts 中新增
function isToolAllowedInPlanMode(toolName: string, phase: 'planning' | 'implementation'): boolean {
  const READONLY_TOOLS = new Set(['read_file', 'list_directory', 'grep'])
  if (phase === 'planning') {
    return READONLY_TOOLS.has(toolName)
  }
  return true // implementation 阶段允许全部工具
}

// 在工具执行前检查
function handleToolExecution(toolName: string, params: unknown, session: Session): ToolResult {
  const planMeta = session.metadata?.plan as PlanMeta | undefined
  if (planMeta && planMeta.status === 'awaiting_approval' || planMeta?.status === 'drafting') {
    if (!isToolAllowedInPlanMode(toolName, 'planning')) {
      return {
        error: 'BLOCKED_BY_PLAN_MODE',
        message: `工具 ${toolName} 在 Plan Mode 探索期不可用。请先完成计划并获得批准。`,
      }
    }
  }
  // 正常执行工具（含确认流程）
}
```

**工具注册安全**：新增工具时必须在 `READONLY_TOOLS` 或写入工具集中显式声明其分类，否则默认视为写入工具（白名单策略，漏加比误加更安全）。

**承诺级别**：使用「系统层强制执行」而非「物理上不可绕过」——Tool ACL 是软件层权限控制，依赖工具注册的完整性，新增工具时的白名单审查流程是其安全保证。

### 4.6 Coordinator-Worker 隔离

**MVP 核心价值：上下文隔离，而非多 Worker 并行。**

如果 MVP 不做隔离，让单个 Agent 同时承担对话+执行，执行过程中产生的大量工具输出会快速填满上下文窗口，导致 Coordinator 丢失早期的用户意图和计划上下文，后续决策质量下降。

**MVP 架构**：Coordinator + 单 Worker

| 角色 | 职责 | 上下文 |
|------|------|-------|
| Coordinator | 与用户对话、理解需求、拆解计划、策展上下文、决策 | 保持用户意图和计划全貌，不加载执行细节 |
| Worker | 按计划步骤执行任务、返回结果摘要 | 加载当前步骤所需的执行上下文 |

**上下文策展策略**：Worker 执行完成后，Coordinator 不接收全量工具输出，而是接收结构化摘要：

```typescript
interface WorkerStepResult {
  stepIndex: number
  status: 'completed' | 'failed' | 'blocked'
  summary: string          // 1-2 句话概括执行结果
  filesModified: string[]  // 修改的文件列表
  errors?: string[]        // 如有错误，错误信息
}
```

**V2 演进**：扩展为多 Worker 并行，引入 Task Board 和异步消息。

### 4.7 状态持久化与恢复

**存储**：计划状态存储在 `Session.metadata` 的 `PlanMeta` 字段中（见 4.3 节），随会话数据一起持久化到 JSON 数据库。

**恢复流程**：

```text
用户重新打开含活跃计划的会话
    │
    ▼
系统检测 Session.metadata 中的 PlanMeta
    │
    ├─ 无计划 → 正常开始对话
    └─ 有计划 → 根据状态处理：
        ├─ drafting / awaiting_approval → 重新展示审批卡片
        ├─ executing → 校验后提示恢复：
        │   "检测到未完成的计划：[xxx]，当前执行到第 N/M 步。
        │    [继续执行] [查看计划] [取消计划]"
        ├─ completed → 仅展示结果摘要
        └─ cancelled → 仅展示取消记录
```

**状态校验**（恢复执行前）：

| 检查项 | 失败处理 |
|--------|---------|
| `currentStepIndex` 在 `[0, stepsTotal)` 范围内 | 重置到最近的有效检查点 |
| 计划文件存在 | 标记计划为 `cancelled`，提示用户重新创建 |
| 计划文件 Frontmatter 与数据库状态一致 | 以计划文件为准，更新数据库状态 |

**环境变更检测**：恢复执行时，对比 `PlanMeta.envSnapshot` 与当前环境。如果 `git HEAD` 已变更或距审批时间超过 24 小时，提醒用户计划可能需要更新。

**步骤幂等性**：每个步骤完成后在 `PlanMeta` 中记录已完成步骤的索引和修改的文件列表。恢复时从下一个未完成步骤继续，不重复执行已完成步骤。

### 4.8 计划取消流程

**触发场景**：用户在执行期想放弃计划。

**流程**：

1. 用户点击「取消计划」按钮
2. 系统弹出确认对话框：「已执行 N/M 步，取消后已修改的文件将保留。确认取消？」
3. 用户确认后，标记 `status: cancelled`，记录 `cancelledAt` 和 `currentStepIndex`
4. Agent 退出 Plan 模式，会话恢复为普通对话

**关于回滚**：MVP 不提供自动回滚能力。取消后已修改的文件保留在磁盘上。理由：
- 自动回滚需要文件备份或 git 操作，复杂度高
- 用户可通过 git 手动回退（如项目在 git 管理下）
- V2 可考虑基于 git stash 的可选回滚

---

### 4.9 探索期终止出口

Agent 在探索后可能判断：

| 情况 | Agent 行为 |
|------|-----------|
| 任务不可行（如依赖缺失） | 终止探索，向用户报告发现和建议，不生成计划 |
| 任务过于简单，不需要计划 | 终止探索，建议用户切换到普通模式执行 |
| 信息不足，无法制定计划 | 终止探索，向用户请求补充信息，不生成计划 |

**实现**：Agent 在 Plan Mode 探索期可输出特殊标记（如 `<plan-abort>`），系统据此展示终止报告卡片而非审批卡片。

---

### 4.10 审批超时与计划过期

**审批超时**：计划提交审批后 24 小时未响应，标记为 `stale`。用户返回后提示「计划已等待审批较长时间，是否仍要审批或需要更新？」

**计划过期**：计划获批后，如果距审批时间超过 24 小时或代码库 `git HEAD` 已变更，恢复执行时提醒用户计划可能需要更新，提供「继续执行」「重新规划」两个选项。

## 五、明确不做（MVP 边界）

| 砍掉的功能 | 替代方案（MVP 内） | 什么时候做 |
|-----------|------------------|-----------|
| Agent 自动判断复杂度 | 用户主动选择模式（普通/Plan） | 不做——用户选择比 Agent 判断更可靠 |
| 多 Worker 并行 | MVP 单 Worker，核心价值是上下文隔离 | V2：任务复杂到需要并行时 |
| 场景化计划模板（3 个） | MVP 使用通用模板 | V2：积累足够使用数据后按场景优化 |
| 双层拆解（方案层+执行层） | Coordinator 直接拆步骤给单个 Worker | V2：任务复杂到需要 Worker 局部 Plan 时 |
| 双环审查（合规+质量） | Worker 执行后 Coordinator 做一致性检查 | V2：对质量要求极高的场景 |
| 信任账户 / L0-L4 分层自治 | MVP 默认所有计划都需审批 | V3：用户与 Agent 长期协作后 |
| 异步消息总线 / Task Board | MVP 用数据库状态+会话 metadata | V2：多 Worker 并行且需跨会话恢复时 |
| 模型分级调度（大小模型混用） | MVP 统一用标准模型 | V2：规模化后控制 Token 成本 |
| 概率审查 | MVP 100% 审批（Plan 模式下） | V2 |
| 探索性分支（多方案并行尝试） | MVP 单方案执行 | V3 |
| 多计划并行 | MVP 同一时间只能有一个活跃计划 | V2 |
| 计划自动回滚 | 取消后文件保留，用户手动 git 回退 | V2：可选 git stash 回滚 |
| 项目目录下的 `.agent/` 隐藏目录 | 计划文件存储在 `.spaceassistant/plans/` 下 | 不做——避免与其他 Agent 工具命名冲突 |

## 六、验收标准

1. **模式选择可用**：消息输入框发送按钮旁有模式选择器，可在「普通模式」和「Plan 模式」之间切换，默认为普通模式。
2. **权限隔离有效**：Plan Mode 探索期调用 `edit_file`/`write_file`/`run_script`，主进程返回 `BLOCKED_BY_PLAN_MODE` 错误，Agent 无法写入。
3. **审批不可绕过**：Plan 模式下，计划未获批前 Agent 不获得写入权限。使用「系统层强制执行」描述，不使用「物理上不可绕过」。
4. **审批可理解**：审批卡片展示后，用户能在不滚动的情况下看到「做什么、影响多大、风险在哪」三个关键信息。
5. **状态可恢复**：关闭并重新打开含活跃计划的会话后，系统能正确检测并恢复计划状态。
6. **关键字段零占位符**：生成的计划中，目标、方案、执行步骤三个核心字段不允许出现「TODO」「TBD」「待确认」等模糊描述。其他字段可标注「需执行时确认」。
7. **异常必上报**：执行中遇到计划未覆盖的阻塞，Agent 在 1 次尝试恢复失败后上报用户，不得无限重试或静默跳过。
8. **取消有记录**：用户取消计划后，`status` 标记为 `cancelled`，记录已执行步骤，不丢失进度信息。
9. **探索可终止**：Agent 可在探索后选择「终止并报告」而非「生成计划」，向用户展示发现而非审批卡片。
10. **环境变更有提醒**：恢复执行时，若 `git HEAD` 已变更或距审批超过 24 小时，系统提醒用户计划可能需要更新。
11. **工具注册安全**：新增工具默认视为写入工具（白名单策略），需显式加入只读白名单才可在探索期使用。

## 七、演进路线图

### Phase 1：MVP（当前）——「用户选择、先规划、后执行、有审批」
- 目标：解决「方向跑偏」问题。
- 特征：用户主动选择模式、单层拆解、单 Worker 隔离、全审批、数据库持久化、工作目录存储。

### Phase 2：效率优化——「能并行的并行，能自动的自动」
- 引入 Task Board 和异步消息，支持多 Worker 并行。
- 引入概率审查（低风险变更自动过审）。
- 引入 Worker 局部 Plan Mode（双层拆解）。
- 引入模型分级调度（机械任务用快模型）。
- 引入场景化计划模板。
- 引入可选的 git stash 回滚。

### Phase 3：自治升级——「人类定规则，Agent 自主跑」
- 引入信任账户，分领域累积自治级别。
- 引入 L1-L3 分层（批量前置、有限自治、高度自治）。
- 引入探索性分支（多方案自动竞争选优）。

## 八、一句话总结

> **MVP 的 Plan 模式 = 用户主动选择 + 只读探索期 + 计划文件契约 + 系统层审批闸门 + Coordinator-Worker 上下文隔离 + 数据库状态持久化。先让 Agent 学会「先问后做」，再教它「做得更快更聪明」。**
