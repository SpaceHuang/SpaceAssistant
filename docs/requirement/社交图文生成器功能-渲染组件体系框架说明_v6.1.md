# 第一部分：体系总览

---

## 1. 三个核心系统

社交图文生成器的架构建立在三个正交的核心系统之上。每个系统可以独立设计、独立升级、独立优化。

### 1.1 系统全景

| 系统 | 回答的问题 | 核心工作 | 执行者 |
|------|-----------|----------|--------|
| **视觉呈现系统** | 长什么样？ | 定义排版语言、颜色、版式 | 产品管理组件，设计师创作组件 |
| **智能排版引擎** | 每页放什么？ | 理解内容、拆页、选版式、填文案 | Skill（AI） |
| **渲染与输出系统** | 怎么拿出来？ | HTML → PNG、校验、导出 | 产品 |

### 1.2 正交性

三个系统通过**接口**而非**实现**耦合：

```
视觉呈现系统  ──Class Contract──→  智能排版引擎
                                      │
                              Skill 输出协议
                                      │
                                      ▼
                              渲染与输出系统
```

- **Class Contract**：视觉呈现系统声明「我提供了哪些排版零件（CSS class）」，智能排版引擎据此选择版式（Recipe），Recipe 引用 Contract 中声明的 class
- **Skill 输出协议**：智能排版引擎将排好的页面以结构化 JSON 交给渲染与输出系统

### 1.3 各系统的独立性证明

| 变更场景 | 影响的系统 | 不影响的系统 |
|----------|-----------|-------------|
| 新增一个风格（Style） | 视觉呈现系统 | 智能排版引擎、渲染引擎（只要 Contract 兼容） |
| 升级排版 Skill 的拆页算法 | 智能排版引擎 | 视觉呈现系统、渲染引擎 |
| 优化 PNG 渲染性能 | 渲染与输出系统 | 视觉呈现系统、智能排版引擎 |
| 新增一个版式（Recipe） | 视觉呈现系统 | 渲染引擎（Recipe 引用已有 Contract class） |
| 新增一个配色（Theme） | 视觉呈现系统 | 智能排版引擎（Theme 只改 CSS 变量，不影响版式选择） |

## 2. 核心概念与层级关系

### 2.1 一张图说清所有概念

```
Style（视觉风格）              ← 「电子杂志风」
  │
  ├── Template（模板 HTML）    ← 定义所有 CSS class、字体、背景系统
  │     └── 实现 → Contract    ← 声明「我提供了哪些排版零件」
  │
  ├── Themes（配色预设）       ←「午夜墨」「暖纸经典」……
  │     └── 覆盖 → CSS 变量
  │
  ├── Recipes（版式骨架）      ←「封面 M01」「清单 M05」「图文 M02」……
  │     └── 引用 → Contract 中声明的 class
  │
  └── StyleRules（风格专属校验）← 字号下限、字体一致性

Project（项目）
  ├── 绑定一个 Style
  ├── 用户的内容（文字 + 图片）
  ├── 排版 Skill 的输出（选了什么 Recipe、填了什么内容）
  └── 渲染结果（PNG）
```

### 2.2 概念层级

```
Style（最高层级：视觉风格的完整定义包）
  ├── Template（HTML 骨架 + CSS class 库）
  │     └── 声明 → Contract（class 接口）
  ├── Recipe（版式：class 的装配方案）
  │     └── 属于 → Page Type（页面类型）
  ├── Theme（配色：CSS 变量覆盖）
  └── StyleRules（校验规则）

Page Type（跨 Style 的页面语义分类）
  └── 每个 Style 下每种 Page Type 有 ≥ 1 个 Recipe

Output Spec（输出规格：画布尺寸）
  └── Recipe 可声明适用的 Output Spec 范围
```

### 2.3 关键关系一句话说明

| 关系 | 说明 |
|------|------|
| **Template → Contract** | Template 实现 Contract 中声明的 class。Contract 是接口，Template 是实现 |
| **Recipe → Contract** | Recipe 只引用 Contract 中的 class 名，不关心具体 Style |
| **Recipe → Theme** | 透明。Recipe 用 `var(--ink)` 引用颜色，不关心色值 |
| **Theme → Template** | Theme 覆盖 Template 中定义的 CSS 变量 |
| **Style → Project** | 一一绑定。一个项目用一种风格 |
| **Page Type → Recipe** | 一对多。一种页面类型在一个 Style 下有多个 Recipe 变体 |
| **Recipe → Output Spec** | 多对多。一个 Recipe 可适配多个输出规格（通过 `specVariants` 提供不同 HTML 变体） |

## 3. 体系设计原则

### 3.1 接口先于实现

> **Contract 是 Style 的「承诺书」，Recipe 是 Contract 的「消费者」。**

Template 作者通过 Contract 声明哪些 class 是开放的、稳定的排版零件。Recipe 作者只依赖这些声明，不关心 Template 的内部实现。这类似于 TypeScript 的类型声明文件——实现可以换，接口不变则消费者不受影响。

### 3.2 正交可替换

> **Style、Recipe、Theme 是三个独立可替换的维度。**

- 换 Style = 换整套视觉语言（但不一定换 Recipe 序列——如果 Contract 兼容）
- 换 Recipe = 换单页的布局方案（不影响其他页、不影响 Style、不影响 Theme）
- 换 Theme = 换全部页面的配色（不影响版式结构、不触发 AI 推理）

### 3.3 确定性归产品，语义推理归 AI

> **产品做一切确定性的工具活；Skill 只做「理解语义 + 选择版式 + 填入内容」。**

- 渲染截图、校验规则执行、配色切换、图片映射维护——这些都是确定性的，由产品层执行
- 理解文章结构、选择版式序列、填入文案、压缩文字——这些需要语义理解，由 Skill 执行

### 3.4 分层校验

> **校验规则按作用域分为全局层、风格层、版式层。**

规则属于谁，就由谁负责定义和分发。三方风格开发者不需要重新声明通用规则（如溢出检测）。

## 4. 风格组件包（Style Package）

### 4.1 什么是 Style

Style 是视觉呈现的**完整定义包**，它回答「这组图文长什么样」。一个 Style 包含五个独立可版本化的子组件：

```
Style "editorial-magazine"（电子杂志风）
  │
  ├── Template          ← 种子模板 HTML（CSS class 库 + 字体 + 背景系统）
  ├── Contract          ← Class 声明文件（模板暴露的稳定接口）
  ├── Themes            ← 配色预设集合（4-6 套内置 + 用户自定义）
  ├── Recipes           ← 版式骨架集合（每种页面类型至少 1 个 Recipe）
  └── StyleRules        ← 风格专属校验规则
```

### 4.2 五个子组件的关系

```
Contract  ←──  Template 实现 Contract 中声明的所有 class
    ↑
    ├──  Recipes 依赖 Contract 中声明的 class 名
    ├──  StyleRules 从 Contract 读取校验阈值（如字号下限）
    └──  校验引擎依据 Contract 做静态检查
Themes  ←──  依赖 Template 中定义的 CSS 变量名
             但不关心 Template 的 class 定义
```

### 4.3 组件版本化

Style 包整体有一个版本号（如 `2.3.0`），内部五个子组件各自独立版本化：

| 子组件 | 独立版本号 | 变更影响 |
|--------|-----------|----------|
| Template | 有 | 影响所有 Recipe 的渲染结果 |
| Contract | 有（semver major.minor） | 影响 Recipe 的兼容性——major 变更可能导致 Recipe 失效 |
| Themes | 有 | 仅影响颜色，不影响版式 |
| Recipes | 每个 Recipe 独立版本号 | 仅影响使用该 Recipe 的页面 |
| StyleRules | 有 | 影响校验结果 |

### 4.4 跨风格约束

| 编号 | 约束 | 说明 |
|------|------|------|
| C-01 | 风格隔离 | 项目绑定一个 Style。切换 Style 时已有 Recipe 映射全部失效，提示用户确认 |
| C-02 | 页面类型覆盖检查 | 切换 Style 后，检查新 Style 的 Recipe 是否覆盖了当前项目所需的所有页面类型。缺失的类型明确提示 |
| C-03 | Contract 兼容提示 | 若新旧 Style 实现了**同一版本**的 Contract，提示用户「原版式序列可自动迁移」（v1.x 实现） |

## 5. Class Contract：模板与骨架的正式接口

### 5.1 为什么需要 Contract

Template 定义了数百个 CSS class（`.h-xl`、`.body-text`、`.ledger-row`……），但 Recipe 只使用其中一部分。没有 Contract 时：

- Recipe 作者不知道哪些 class 是「稳定的排版零件」，哪些只是 Template 内部实现
- Recipe 无法跨 Style 复用——`.h-xl` 在 Style A 和 Style B 中语义完全不同
- 校验系统只能检查 class 是否存在，无法检查语义是否匹配

**Contract 是一份正式声明：Template 作者承诺「这些 class 是我对外开放的稳定排版零件，每个 class 的语义和约束如下」。** 它的角色类似于：
- TypeScript 的 `.d.ts` 类型声明文件
- REST API 的 OpenAPI Schema
- 操作系统内核的 syscall 表

### 5.2 Contract 的结构

```yaml
# contract.yaml — 随 Style 独立发布
contractVersion: "1.0"
styleId: "editorial-magazine"
styleName: "电子杂志风"

classes:

  # ── 标题层级 ──
  h-xl:
    role: "最大标题，封面主标题"
    minFontSize: 48px
    maxLines: 2
    maxCharsPerLine: 8
    allowedPageTypes: [cover, quote]

  h-lg:
    role: "次级标题，内页标题"
    minFontSize: 32px
    maxLines: 3
    allowedPageTypes: [essay, photo, checklist, data, closing]

  # ── 正文 ──
  body-text:
    role: "正文段落"
    minFontSize: 18px
    maxLines: 20
    allowedPageTypes: [essay, photo, closing]

  # ── 布局容器 ──
  ledger-row:
    role: "清单行容器，纵向排列"
    childClass: "ledger-item"

  two-col:
    role: "双列对比容器"
    childClass: "col-card"

  # ── 图片 ──
  img-hero:
    role: "全幅主图"
    objectFit: "cover"

  img-inline:
    role: "内联图片"
    minWidth: 200px

  # ── 装饰/引用 ──
  accent-tag:
    role: "高亮标签"

  pull-quote:
    role: "引用文字，占满画面制造节奏停顿"
    minFontSize: 36px
    maxLines: 4
    allowedPageTypes: [quote, closing]
```

完整的 Contract YAML 示例见附录 A。

### 5.3 Contract 的效力

| 对 Template 作者 | 对 Recipe 作者 | 对校验引擎 |
|------------------|----------------|------------|
| 承诺实现的 class 必须在 Template 中存在且满足声明约束 | 只能引用 Contract 中声明的 class | 基于 Contract 做静态校验：Recipe 引用的 class 是否在 Contract 中、使用是否合规 |
| 可以定义 Contract 之外的内部 class，Recipe 不可引用 | 不感知 Template 的内部实现 | 从 Contract 读取规则阈值（如字号下限） |
| Contract 版本变更需标记兼容性（major/minor） | Recipe 声明依赖的 Contract 版本 | — |

### 5.4 Contract 与 Theme 的关系

Contract **不声明颜色相关信息**。颜色完全由 Theme 通过 CSS 变量管理：

```
Contract 声明：h-xl 是「最大标题」，字号 ≥ 48px，最多 2 行
Theme 覆盖：   --ink: #1a1a1a  （h-xl 的颜色通过 var(--ink) 获取）
```

Recipe 引用 `h-xl` 时，字体大小由 Template + Contract 保证，颜色由当前 Theme 的 CSS 变量决定。两者独立变化，互不干扰。

## 6. Template 规范

> **本节定义 Template 的具体结构、CSS 变量体系、以及渲染引擎如何将 Recipe HTML 拼接为完整可渲染文档。这是渲染引擎工程化的前置输入。**

### 6.1 Template 的结构

一个 Template 是一个目录，包含以下文件：

```
{template-name}/
  ├── template.html       # HTML 骨架，定义 <!DOCTYPE>/<html>/<head>/<body> 结构
  ├── variables.css       # CSS 变量声明（:root {} 中定义所有可被 Theme 覆盖的变量）
  ├── styles.css          # CSS class 的具体实现（Contract 中声明的每个 class 的视觉定义）
  ├── fonts/              # 字体文件目录（.woff2 格式）
  └── preview.png         # 风格预览图（用于轮播卡片展示）
```

### 6.2 template.html 规范

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <link rel="stylesheet" href="variables.css">
  <link rel="stylesheet" href="styles.css">
  <!-- 字体声明由渲染引擎注入（见 §20 字体管理） -->
</head>
<body>
  <!-- POSTER_CONTENT -->
</body>
</html>
```

**关键约定：**

| 约定 | 说明 |
|------|------|
| `<!-- POSTER_CONTENT -->` 占位标记 | 渲染引擎在此位置插入 Recipe HTML（单个页面的 `<section>` 片段） |
| `<body>` 不设内联样式 | 画布尺寸（width/height）由 Recipe 的 `specVariants` 中的 `style` 属性控制 |
| 字体声明不写在 template.html 中 | 字体由渲染引擎统一注入 `@font-face`，保证所有 Template 共享字体缓存 |
| 禁止外部资源引用 | Template 中的 CSS `url()` 只能引用本地相对路径（`fonts/` 目录内的字体文件） |

### 6.3 CSS 变量体系

每个 Template 必须在 `variables.css` 中声明以下**必选 CSS 变量**（以及强烈建议的 `--color-muted`）。Theme 文件只覆盖这些变量：

```css
:root {
  /* ── 必选变量（所有 Template 必须声明）── */
  --color-bg: #FFFFFF;           /* 页面背景色 */
  --color-text: #1A1A1A;         /* 正文色 */
  --color-accent: #C41E3A;       /* 强调色（链接、高亮标签、CTA） */
  --font-body: "Source Han Serif SC", "Noto Serif SC", serif;    /* 正文字体族 */
  --font-heading: "Source Han Sans SC", "Noto Sans SC", sans-serif; /* 标题字体族 */

  /* ── 强烈建议声明（未声明时回退到 --color-text）── */
  --color-muted: #888888;        /* 次要文字色（脚注、期号、辅助信息） */

  /* ── 可选变量（Template 按需声明）── */
  --color-card-bg: #F8F8F8;      /* 卡片底色 */
  --color-rule: #DDDDDD;         /* 分割线色 */
  --space-section: 40px;         /* 段落间距 */
  /* Template 可声明任意数量的额外变量 */
}
```

> **`--color-muted` 的 fallback 规则：** Template 中引用次要文字色的 CSS 规则应使用 `var(--color-muted, var(--color-text))`。若某风格不需要独立的次要文字色（如极简风格只用 2 色），Template 可以不声明此变量，所有次要文字自动回退到正文色。

> **Theme 的行为约束：** Theme 只能覆盖 `variables.css` 中已声明的变量名。Theme 不能新增变量名（新增变量需要 Template 更新）。

### 6.4 Template 与 Recipe 的拼接流程

```
渲染引擎渲染单页时：
  1. 读取 template.html → 获得 HTML 骨架
  2. 在 <!-- POSTER_CONTENT --> 处插入 Recipe HTML
  3. 注入当前 Theme 的 CSS 变量覆盖值（覆盖 variables.css 中声明的变量）
  4. 注入字体 @font-face 声明
  5. 替换 Recipe HTML 中的 {{PLACEHOLDER}} 为实际内容
  6. 替换图片 src 为渲染沙箱可访问的路径（base64 data URI 或本地临时副本）
  7. 在目标视口尺寸下渲染
```

## 7. 版式骨架（Recipe）

### 7.1 什么是 Recipe

Recipe 是一段可复用的 HTML 片段模板。它定义了某一种页面类型（封面、清单、对比……）的 DOM 结构和 class 组合方式。它是 Template 定义的「零件」的**装配方案**。

Recipe 不包含任何 CSS 样式定义——它只引用 Template 中定义的 class 名。Recipe 也不关心颜色——颜色通过 CSS 变量由 Theme 注入。

### 7.2 Recipe 的样式依赖层级

Recipe 的完整视觉呈现依赖三个层级，按加载顺序：

```
L1（自包含）：width/height 由 specVariants 中的 style 属性确定（必须）
L2（Template 提供）：所有 class 的视觉呈现（颜色、字体、字号、间距）
L3（Theme 注入）：颜色通过 CSS 变量覆盖 Template 默认值
```

**降级行为：** 如果 Template 未加载（加载延迟、离线场景下 Template 资源缺失），Recipe HTML 的降级渲染为：
- 保留 DOM 结构和文字内容
- 使用浏览器默认样式（黑白、系统默认字体）
- 面板显示「样式加载中」的半透明覆盖层；2 秒后若 Template 仍未加载，以此降级视图展示

> **设计意图：** Recipe 自身不包含 CSS 样式，但这不意味着它在无 Template 时完全不可渲染。降级视图保证了用户在任何情况下都能看到内容，只是不美观。

### 7.3 specVariants 中的 style 属性说明

§10.3（原 §9.3）的 `specVariants` 示例中包含了 `style="width:1080px;height:1440px"` 内联样式。这是 Recipe 中**唯一允许的内联样式**——它仅声明画布尺寸，不涉及视觉样式（颜色、字体、间距）。这与「Recipe 不含样式」的原则不冲突：

| 内联属性 | 是否允许 | 理由 |
|----------|----------|------|
| `style="width:...;height:..."` | ✅ 允许 | 画布尺寸是 Recipe 的结构性约束，非视觉样式 |
| `style="color:...;font-size:..."` | ❌ 禁止 | 颜色/字体属于 Template/Theme 职责 |
| `style="padding:...;margin:..."` | ❌ 禁止 | 间距应由 Template 的 class 定义 |

### 7.4 Recipe 元数据

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `id` | `string` | 唯一标识，Style 内不重复 | `M01` |
| `name` | `string` | 中文名称 | 「杂志封面」 |
| `pageType` | `enum` | 页面类型（见第 9 章） | `cover` |
| `description` | `string` | 适用场景说明 | 「适合小红书首页封面，大标题 + 主图 + 底部要点条」 |
| `applicableSpecs` | `enum[]` | 适用的输出规格 | `["xhs", "square"]` |
| `specVariants` | `object?` | 不同输出规格的 HTML 变体（key 为规格 ID，value 为该规格的 HTML） | `{"xhs": "<section...>", "square": "<section...>"}` |
| `contractVersion` | `string` | 依赖的 Contract 最低版本 | `"1.0"` |
| `minContentDensity` | `number` | 最低内容覆盖率（用于校验 R6） | `0.75` |
| `contentLimits` | `object` | 各项内容的硬限制 | `{ title: { maxLines: 2, maxCharsPerLine: 8 }, steps: { max: 5 } }` |
| `slotSchema` | `object` | 占位符定义（Skill 据此填入内容） | `{ TITLE: "string", HERO_IMG: "image", TAGS: "string[]" }` |
| `html` | `string` | 带占位符的 HTML 片段 | 见 §7.5 |
| `thumbnail` | `image` | 预览缩略图 | PNG |

### 7.5 Recipe HTML 示例

```html
<section class="poster" data-page-type="cover" data-recipe="M01">
  <div class="cover-header">
    <span class="accent-tag">{{TAG}}</span>
    <span class="issue-info">{{ISSUE}}</span>
  </div>
  <h1 class="h-xl">{{TITLE}}</h1>
  <div class="hero-image">
    <img class="img-hero" src="{{HERO_IMG}}" alt="">
  </div>
  <div class="cover-footer">
    <p class="body-sm">{{SUBTITLE}}</p>
  </div>
</section>
```

关键观察：
- HTML 中使用的 class（`accent-tag`、`h-xl`、`img-hero`、`body-sm`）全部来自 Contract 声明
- 占位符 `{{TITLE}}`、`{{HERO_IMG}}` 对应 slotSchema 中的字段
- `data-page-type` 和 `data-recipe` 属性供校验引擎和预览面板使用
- 没有任何内联样式或颜色——全部依赖 CSS class 和 CSS 变量

### 7.6 Recipe 管理功能

| 编号 | 功能 | 优先级 | 说明 |
|------|------|--------|------|
| L-01 | Recipe 浏览 | P0 | 按页面类型分组浏览当前 Style 的所有 Recipe |
| L-02 | Recipe 收藏 | P1 | 标记为收藏，排版 Skill 选 Recipe 时优先 |
| L-03 | Recipe 禁用 | P1 | 不需要的 Recipe 可禁用，排版 Skill 不选 |
| L-04 | Recipe 排序 | P2 | 同类型内调整优先级 |
| L-05 | 自定义 Recipe | P1 | 基于现有 Recipe 克隆修改，或从空白 HTML 创建 |
| L-06 | Recipe 校验 | P0 | 保存时自动检查：见 §7.7 交叉验证规则 |
| L-07 | Recipe 导入/导出 | P1 | 单个 Recipe 或整组 Recipe 的 `.recipe` 文件导入导出 |
| L-08 | 手动编排模式 | P1 | 高级用户拖拽 Recipe 序列 |
| L-09 | 序列模板 | P2 | 保存和复用常用 Recipe 序列 |
| L-10 | Recipe 推荐 | P2 | 同类型下，Contract 兼容的跨 Style Recipe 推荐（v1.x） |

### 7.7 Recipe 保存时的交叉验证规则

Recipe 保存时，校验引擎执行以下交叉验证（基于 Contract + Recipe 元数据）：

| 编号 | 验证内容 | 违反时 |
|------|----------|--------|
| **XV-01** | Recipe 引用的所有 class 必须在 Contract.classes 中声明 | ✗ **FAIL** — 拒绝保存。Recipe 不能引用未声明的 class |
| **XV-02** | Recipe 的 `pageType` 必须在每个引用 class 的 `allowedPageTypes` 中（若 class 声明了此字段） | ⚠ **WARN** — 允许保存但标记。例如 `body-text` 的 `allowedPageTypes: [essay, photo, closing]` 不包含 `cover`，若 `cover` 类型 Recipe 引用了 `body-text` 则警告 |
| **XV-03** | Recipe `contentLimits` 的约束可以比 Contract 更严格，但不可更宽松 | ⚠ **WARN**（更宽松时）。例如 Contract 中 `h-xl.maxLines: 2`，Recipe 中 `contentLimits.title.maxLines: 3` 则警告 |
| **XV-04** | Recipe 的 `slotSchema` 与实际 HTML 中的 `{{PLACEHOLDER}}` 必须完全一致 | ✗ **FAIL** — 拒绝保存。`slotSchema` 中有 `TITLE` 但 HTML 中没有 `{{TITLE}}`（或反之）则报错 |
| **XV-05** | Recipe HTML 中无 `<script>` 标签、无事件处理器（`onerror`/`onload` 等）、无 `javascript:` 协议 URL | ✗ **FAIL** — 拒绝保存（安全策略） |

> **XV-02 的例外：** 若 class 未声明 `allowedPageTypes` 字段（即对所有页面类型开放），则不触发 XV-02 检查。

## 8. 主题色（Theme）

### 8.1 什么是 Theme

Theme 是当前 Style 下的一组 CSS 变量覆盖值。它**只改变颜色，不改排版结构**。

```
同一段 Recipe HTML：
  在「暖纸经典」下 → 暖纸底色 + 深墨文字
  在「午夜墨」下   → 暗底色 + 金辉文字
  排版结构完全相同
```

### 8.2 Theme 的作用域

Theme 依赖 Template 中定义的 CSS 变量名，**不可跨 Style 共享**。

不同 Style 的颜色语义不同：「电子杂志风」的色板语义是「纸色 + 墨色」，「瑞士网格」是「底色 + 单一高饱和锚点色」。同一套 hex 值在不同 Style 中扮演的角色完全不同。

```
Style A 的 CSS 变量：
  --paper: #FFF8F0     ← 纸张底色
  --ink: #1A1A1A       ← 正文墨色
  --accent: #C41E3A    ← 强调色

Style B 的 CSS 变量：
  --bg: #FFFFFF        ← 全局背景
  --text: #111111      ← 正文色
  --anchor: #0066FF    ← 锚点色（唯一高饱和色）
```

### 8.3 Theme 的运行时行为

配色切换在 HTML 预览面板中的执行路径：

```
用户选择 Theme "midnight-ink"
  → 面板获取该 Theme 的 CSS 变量覆盖值
    → 注入到 Template 的 :root {}
      → 浏览器即时重绘全部页面（< 100ms）
        → 完全不触发排版 Skill
        → 完全不改变 Recipe 序列
        → 完全不重新渲染 HTML
```

这是纯 CSS 级别的操作——类似于在浏览器 DevTools 中切换颜色变量。速度极快，无需任何等待。

### 8.4 Theme 功能

| 编号 | 功能 | 优先级 | 说明 |
|------|------|--------|------|
| TH-01 | 预设选择 | P0 | 每个 Style 内置 4-6 套 Theme 预设，面板顶部下拉或色块选择 |
| TH-02 | 实时预览 | P0 | 切换 Theme 时面板即时反映变化（< 100ms） |
| TH-03 | 自定义色板 | P1 | 高级模式下覆盖预设的 CSS 变量值 |
| TH-04 | 色板命名与保存 | P1 | 自定义色板命名、保存、删除 |
| TH-05 | 色板冲突提示 | P1 | 低对比度或接近色值的变量组合给出警告 |
| TH-06 | 色板重置 | P1 | 将修改过的预设恢复到默认值 |
| TH-07 | 色板导出 | P2 | 单独导出 `.theme` 文件 |

## 9. 页面类型（Page Type）

### 9.1 什么是页面类型

页面类型是**跨 Style 的页面语义分类**。它定义了「这一页做什么」，不定义「这一页长什么样」。

页面类型是排版 Skill 选择 Recipe 的基础：排版 Skill 决定「第 3 页需要一个清单页」，然后从当前 Style 的 `checklist` 类型下挑选一个具体 Recipe。

### 9.2 页面类型清单

| 页面类型 ID | 名称 | 说明 | 典型内容 |
|---------|------|------|----------|
| `cover` | 封面页 | 文章的第一张图，含标题钩子和主视觉 | 大标题、主图、标签、期号 |
| `photo` | 图文/现场笔记 | 图片是主角，配合简短说明 | 图片 + 说明文字 |
| `essay` | 文字论述页 | 中等文字量，标题 + 段落 + 边栏备注 | 标题 + 正文 + 备注 |
| `quote` | 引语/宣言页 | 一句核心观点占满画面，制造节奏停顿 | 引用文字 |
| `checklist` | 清单页 | 编号条目，每项有标题和说明 | 3-8 条带编号的条目 |
| `compare` | 对比页 | 两列对比（前后对比 / 优劣对比） | 左右两列对比卡片 |
| `data` | 数据展示页 | 数字、柱状图、KPI 等量化信息 | 数据可视化 |
| `pipeline` | 步骤流 | 纵向 3-5 步流程 | 步骤 + 说明 |
| `evidence` | 截图/证据墙 | 多张截图或小图网格排列 | 多图网格 |
| `closing` | 收束页 | 最后一页，总结 + 行动号召 | 总结 + CTA |
| `divider` | 章节分隔页 | 无实质内容，纯视觉节奏打断 | 章节标题 |
| `matrix` | 能力矩阵 | 多单元格网格展示能力/功能清单 | 网格卡片 |

### 9.3 页面类型覆盖要求

每种页面类型在每个 Style 中应至少提供 ≥ 1 个 Recipe。类型缺失时，该 Style 无法生成对应类型的页面。

**MVP 底线：** 内置的 5 个 Style 必须覆盖全部 12 种页面类型，即至少提供 5 × 12 = 60 个 Recipe。

## 10. 输出规格（Output Spec）

### 10.1 什么是输出规格

输出规格定义了最终图片的画布尺寸和比例。它是「图文发布到哪个平台」的物理约束。

### 10.2 内置规格

| 规格 | 尺寸 | 比例 | 用途 |
|------|------|------|------|
| 小红书图文 | 1080 × 1440 | 3:4 | 小红书/Rednote 图文内页 |
| 公众号主封面 | 2100 × 900 | 21:9 | 微信公众号文章主封面 |
| 公众号分享卡 | 1080 × 1080 | 1:1 | 公众号分享卡片 |
| 自定义 | 用户指定 | — | — |

### 10.3 与 Recipe 的关系

Recipe 通过 `applicableSpecs` 声明自己适配哪些输出规格。对于同一 Recipe 需适配多个规格的情况，通过 `specVariants` 提供不同规格的 HTML 变体：

```jsonc
{
  "id": "M01",
  "applicableSpecs": ["xhs", "square"],
  "specVariants": {
    "xhs": "<section class=\"poster\" style=\"width:1080px;height:1440px\">...</section>",
    "square": "<section class=\"poster\" style=\"width:1080px;height:1080px\">...</section>"
  }
}
```

如果 `specVariants` 为空或未包含某规格，则该规格使用默认 `html` 字段。`html` 字段始终存在——它是 Recipe 的默认 HTML（通常用于最常见的输出规格如 `xhs`）。`specVariants` 中的条目以键值对形式覆盖默认 HTML 中对应规格的版本：当渲染引擎需要规格 `S` 的 HTML 时，优先取 `specVariants[S]`；若不存在则 fallback 到 `html` 字段。

## 11. Contract 版本演进与兼容性

### 11.1 版本号语义

Contract 版本号遵循 `major.minor` 语义（不设 patch）。版本兼容性规则如下：

| 变更类型 | 版本变化 | 对 Recipe 影响 | 示例 |
|----------|----------|----------------|------|
| **新增 class** | minor ↑ | 旧 Recipe 不受影响；新 Recipe 可使用新 class | 新增 `img-stack` class |
| **放宽约束**（如 `maxLines: 2 → 3`） | minor ↑ | 旧 Recipe 兼容 | `h-xl` 每行字数从 8 放宽到 10 |
| **新增 allowedPageTypes** | minor ↑ | 旧 Recipe 兼容 | `h-xl` 新增 `allowedPageTypes: [closing]` |
| **收紧约束**（如 `maxLines: 3 → 2`） | **major ↑** | 旧 Recipe 可能不合规——原先填 3 行的标题现在触发 R7 校验 | `h-lg` 行数从 3 收紧到 2 |
| **class 重命名** | **major ↑** | 旧 Recipe 引用失效，需迁移 | `accent-tag` → `badge` |
| **class 删除** | **major ↑** | 旧 Recipe 引用失效 | 移除 `pull-quote` |
| **移除 allowedPageTypes** | **major ↑** | 使用了被移除页面类型的旧 Recipe 不合规 | `h-xl` 移除 `allowedPageTypes: [quote]` |

### 11.2 产品层行为

- 导入 Recipe 时检查 `contractVersion` 是否 ≤ 当前 Style 的 Contract 版本
- 若 Recipe 依赖 `major` > 当前 Style 的 Contract `major` → **拒绝导入**，提示「此 Recipe 需要更新的 Contract」
- 若 Recipe 依赖 `minor` > 当前 Style 的 Contract `minor`（major 相同）→ **警告但不阻止**，提示「此 Recipe 使用了部分当前 Style 可能不支持的新 class」
- Style 升级 major 版本时，弹出「受影响 Recipe 清单」（引用已删除/重命名 class 的 Recipe），用户可选择逐个迁移或保留旧版本 Style

### 11.3 设计意图：鼓励 minor 演进，审慎 major 变更

这套规则的设计意图是引导 Style 作者**向后兼容地演进**：
- 新增 class 和放宽约束对现有 Recipe 无影响 → 用 minor
- 删除 class、重命名 class、收紧约束会破坏现有 Recipe → 用 major
- 这意味着 major 版本变更应该是一个 **有意识的、附迁移指南的决策**，而非意外发生

## 12. 跨风格版式复用机制

### 12.1 复用的前提

Recipe 可以跨 Style 复用的条件是：**目标 Style 实现了与 Recipe 依赖相同（或更高）major 版本的 Contract，且 Contract 中包含 Recipe 引用的所有 class。**

```
Recipe「极简清单 M05」
  ├── 依赖 Contract v2.0 中的：ledger-row, ledger-item, h-md, body-sm, rule-line
  │
  ├── 在 Style A（editorial-magazine）下 ✓ 可用
  │     └── Style A 实现了 Contract v2.0
  │
  ├── 在 Style B（swiss-grid）下 ✓ 可用
  │     └── Style B 实现了 Contract v2.0（用瑞士风格的排版零件实现了同样语义的 class）
  │
  └── 在 Style C（brutalist）下 ✗ 不可用
        └── Style C 实现了 Contract v1.0，缺少 ledger-row 等 class
```

### 12.2 复用的视觉差异

同一个 Recipe 在不同 Style 下渲染时，**结构完全相同，但视觉呈现不同**：

```
「极简清单 M05」：
  在 editorial-magazine 下 → 纸色底 + 衬线字体 + 手绘风格分割线
  在 swiss-grid 下         → 纯白底 + 无衬线字体 + 几何分割线
  排版结构完全相同（条目数量、间距、层级关系一致）
```

这是因为 Recipe 只引用 class 名（`ledger-row`、`h-md`），而不同 Style 的 Template 对同一个 class 有不同的 CSS 实现。

### 12.3 v1.0 的复用边界

v1.0 只内置 5 个 Style，各 Style 的 Contract 独立定义。跨 Style 复用是 v1.x 的功能——届时多个 Style 可声明实现同一 Contract 版本，Recipe 即可在它们之间自由迁移。

## 13. 校验规则分层体系

### 13.1 设计原则

校验规则按作用域分为三层。规则属于谁，就由谁负责定义和分发：

```
全局规则（引擎内置，不随 Style 分发）
 ├── R1 内容溢出：scrollHeight > clientHeight
 ├── R2 Footer 碰撞：绝对定位页脚与正文重叠
 └── R3 margin 漂移：浏览器默认 margin 导致偏移

风格规则（随 Style 分发）
 ├── R4 字号下限：阈值来自 Contract 中的 minFontSize
 └── R5 字体一致性：检测未被 Template 声明的字体家族

Recipe 规则（随 Recipe 元数据）
 ├── R6 画布密度：阈值来自 Recipe 的 minContentDensity
 └── R7 标题超限：阈值来自 Recipe 的 contentLimits
```

### 13.2 收益

- 三方 Style 开发者不需要重新声明通用规则（R1-R3）
- 规则的归属与其触发条件一致——哪里定义了约束，哪里就负责校验
- 新增 Style 时自动获得全局规则的覆盖

### 13.3 内置规则清单

| ID | 名称 | 检测内容 | 作用域 | 默认级别 | 执行阶段 | 前置条件 |
|----|------|----------|--------|----------|----------|----------|
| R1 | 内容溢出 | `scrollHeight > clientHeight` | 全局 | FAIL | **PNG 渲染阶段**（截图前检测） | 需要在目标画布尺寸下以实际字体渲染后测量；Recipe 自身若 `overflow: hidden`，需临时移除该属性后测量 |
| R2 | Footer 碰撞 | 绝对定位页脚与正文重叠 | 全局 | FAIL | **PNG 渲染阶段**（截图前检测） | 需要解析 computed style 中 `position: absolute` / `position: fixed` 元素的 bounding rect |
| R3 | margin 漂移 | 浏览器默认 `<figure>` 边距 | 全局 | WARN | **HTML 面板渲染阶段**（即时检测） | 遍历所有 `<figure>` 元素的 computed margin 值 |
| R4 | 字号下限 | 正文/标题低于 Contract 声明的最低尺寸 | 风格 | WARN | **HTML 面板渲染阶段**（即时检测） | 需要已加载 Contract 中的 `minFontSize` 阈值 |
| R5 | 字体一致性 | 检测到未被 Template 声明的字体家族 | 风格 | WARN | **HTML 面板渲染阶段**（即时检测） | 需要已加载 Template 中声明的字体列表 |
| R6 | 画布密度 | 内容覆盖率不足 Recipe 声明的 `minContentDensity` | Recipe | WARN | **HTML 面板渲染阶段**（即时检测） | 需要计算所有非空白元素的面积总和 / 画布总面积 |
| R7 | 标题超限 | 标题行数/字数超过 Recipe 声明的 `contentLimits` | Recipe | WARN | **HTML 面板渲染阶段**（即时检测） | 需要在目标字号和容器宽度下计算标题元素的实际行数 |

> **关键区分：** R1 和 R2 必须在 PNG 渲染管线中执行，因为它们依赖真实的字体渲染结果和视口尺寸（HTML 面板的渲染环境可能与截图环境有差异——字体渲染、deviceScaleFactor 不同）。R3-R7 可以在 HTML 面板中即时检测，因为它们不依赖截图管线的特定环境。

### 13.4 校验结果的双通道呈现

| 通道 | 呈现方式 |
|------|----------|
| **HTML 预览面板** | 溢出区域红色虚线边框标注（PNL-07）、字号过小区高亮、密度不足区域半透明覆盖 |
| **聊天消息** | 编排 Skill 摘要消息（「⚠️ 检测到 2 个问题：① 第 3 页标题超限，② 第 1 页密度不足」）+ 快捷选项 |

### 13.5 规则引擎功能

| 编号 | 功能 | 优先级 | 说明 |
|------|------|--------|------|
| V-01 | 自动校验 | P0 | 渲染完成后自动跑校验 |
| V-02 | 规则状态 | P0 | 每条规则返回 ✓ PASS / ⚠ WARN / ✗ FAIL |
| V-03 | FAIL 处理 | P0 | 显著提示，但不阻止导出（用户可自行判断） |
| V-04 | 规则开关 | P1 | 逐条开关 |
| V-05 | 规则阈值调整 | P1 | 部分规则阈值可调 |
| V-06 | 规则集自动切换 | P0 | 切换 Style 时自动加载新 Style 的 StyleRules 和 RecipeRules |

## 14. Skill 输出协议

### 14.1 协议概览

排版 Skill 输出的不是裸 HTML 字符串，而是结构化的 Poster 数组。协议包含 **success、partial、failed** 三种状态。

### 14.2 协议结构

```jsonc
{
  "protocolVersion": "1.0",
  "status": "success",          // "success" | "partial" | "failed"
  "styleId": "editorial-magazine",
  "themeId": "midnight-ink",
  "posters": [
    {
      "pageType": "cover",
      "recipeId": "M01",
      "spec": "xhs",
      "width": 1080,
      "height": 1440,
      "html": "<section class=\"poster\" data-page-type=\"cover\" data-recipe=\"M01\">...</section>",
      "imageAssignments": [
        { "slotId": "HERO_IMG", "assetId": "img-003" }
      ]
    }
  ],
  "warnings": [
    {
      "level": "warn",
      "posterIndex": 0,
      "code": "TITLE_TRUNCATED",
      "message": "标题被压缩：原文「2026年最值得关注的10个AI趋势与工具」→「2026最值得关注的AI趋势」"
    }
  ],
  "errors": [                    // status = "partial" 或 "failed" 时存在
    {
      "level": "error",
      "posterIndex": 3,
      "code": "NO_SUITABLE_RECIPE",
      "message": "当前 Style 的 data 页面类型下无可用 Recipe",
      "suggestion": "建议切换 Style 或手动添加 data 类型 Recipe"
    }
  ],
  "analysis": {                  // status = "failed" 时存在
    "contentStructureType": "对比型",
    "suggestedPageTypes": ["cover", "compare", "data", "closing"],
    "estimatedPageCount": 5,
    "missingRequirements": ["当前 Style 缺少 compare 和 data 类型的 Recipe"]
  }
}
```

完整的协议示例见附录 C。

### 14.3 三种状态的产品层处理

| status | 含义 | 产品层处理 |
|--------|------|-----------|
| `success` | 全部页面生成成功 | HTML 推送面板、展示 warnings（如有） |
| `partial` | 部分页面生成失败 | 成功页 HTML 推送面板 + 失败页显示占位标注（无按钮）；聊天中呈现两条路径（跳过 vs 换风格），各自说明代价 |
| `failed` | 全部生成失败 | 面板保持空置；编排 Skill 根据 `analysis` 字段生成精准诊断消息；聊天中提供下一步建议 |

### 14.4 `analysis` 字段的用途收敛

`analysis` 字段仅在 `status: failed` 时存在。它的价值是**帮助编排 Skill 生成更精准的聊天消息**，而非驱动面板渲染。

```
编排 Skill 拿到 analysis 后的精准消息：
  「电子杂志风」缺两种版式（对比和数据），所以这次没能生成。
  你的文章是「对比型」结构，换瑞士网格就能完整覆盖。要试试吗？

对比不依赖 analysis 的泛泛消息：
  这篇文章我没能生成出合适的排版。可能是内容太短或风格缺少版式。
  你要调整一下再试吗？
```

`analysis` 字段不应被用于在面板上构建骨架视图、✓/⚠️ 标注或修复按钮——这些交互在 `failed` 状态下属于聊天通道而非面板通道。

## 15. 图片与版式的绑定机制

### 15.1 图片分配映射

排版 Skill 输出的 `imageAssignments` 字段显式声明了每张图片被分配到哪个 poster 的哪个 slot：

```jsonc
"imageAssignments": [
  { "slotId": "HERO_IMG", "assetId": "img-003" },
  { "slotId": "PHOTO_01", "assetId": "img-001" }
]
```

### 15.2 图片到 slot 的映射信号优先级

排版 Skill 按以下优先级链分配图片到 slot：

| 优先级 | 信号 | 来源 | 说明 |
|--------|------|------|------|
| 1 | **文件名关键词匹配** | 编排 Skill 自动提取 | 零成本。文件名含「封面」「hero」「cover」→ 封面 slot |
| 2 | **AI 推理** | 排版 Skill | 看图理解内容 × 文章结构上下文。最智能但最慢 |
| 3 | **`uploadIndex` 上传顺序** | 编排 Skill 自动记录 | 零成本降级锚点。第 1 张大概率最重要（封面候选） |

> 同优先级内的冲突解决规则详见 PRD §8.3。

### 15.3 各种场景下的产品层行为

| 场景 | 行为 |
|------|------|
| **排版 Skill 生成时** | 排版 Skill 输出 `imageAssignments`，产品层据此将图片填入对应 slot，渲染出图 |
| **用户手动替换图片** | 用户以自然语言指令替换某页的图片 → 更新该 slot 的 `assetId` 映射 → 单页重渲。**仅影响当前 poster** |
| **换 Recipe（同 slot schema）** | 产品层将旧 Recipe 各 slot 的图片映射到新 Recipe 同名 slot（`HERO_IMG → HERO_IMG`），多余 slot 留空，缺失 slot 的图片退回未分配池 |
| **换 Recipe（不同 slot schema）** | 调用排版 Skill 重新填入，但**保留用户手动指定过的图片映射**（不覆盖用户的显式选择），其余由 Skill 重新分配 |
| **用户上传了新图片** | 新图片进入未分配池。排版 Skill 重新生成时可使用；用户也可手动指定 |
| **图片删除** | 若图片被任何 poster 引用，编排 Skill 以对话确认后执行 |

### 15.4 ImageAsset 的 uploadIndex 字段

v5.0 新增 `uploadIndex` 字段（int），由编排 Skill 在收到每张图片时自动填入序号。这是零成本信号，为图片映射提供了一个比随机分配强一个数量级的降级锚点。

## 17. 风格切换的连锁反应

当用户切换项目的 Style 时，发生以下连锁反应：

```
用户将 Style A → Style B
  │
  ├── 1. 新的 Template 提供了不同的 class 实现
  │     （可能实现了与旧 Style 相同或不同的 Contract）
  │
  ├── 2. 新的 Themes 覆盖 CSS 变量
  │     （旧 Theme 不可迁移到新 Style——颜色语义不同）
  │
  ├── 3. 已有的 Recipe 映射全部失效
  │     （因为 Recipe 引用的是旧 Template 的 class）
  │
  ├── 4. [检查] 如果新旧 Style 实现了同一个 Contract
  │     → 原本的 Recipe 序列理论可迁移
  │     → 编排 Skill 提示用户确认
  │
  └── 5. [默认] 否则用户需要重新由排版 Skill 为新 Style 选择 Recipe 序列
        → 编排 Skill 提示：「切换风格会清除当前 7 页的版式选择。确认吗？」
```

### 产品层行为

- 切换 Style 前编排 Skill 确认（聊天场景）或弹出确认对话框（产品 UI 场景）
- 确认后：清除所有 Recipe 映射 → 调用排版 Skill 重新生成 → 新 HTML 推送面板
- 若用户取消：保持当前状态不变

## 18. 配色切换的运行时行为

配色切换与风格切换有本质区别——它是**纯视图操作**，不触发任何数据变更：

```
用户切换 Theme（在面板右上角下拉选择）
  │
  ├── 1. 面板获取该 Theme 的 CSS 变量覆盖值
  │
  ├── 2. 注入到当前 Template 的 :root {}
  │
  ├── 3. 浏览器即时重绘全部页面（< 100ms）
  │
  └── 不触发：排版 Skill、Recipe 重新选择、HTML 重新生成、PNG 渲染
```

### 为什么配色不经过聊天

配色切换是「感知」而非「决策」——用户看到配色变化，自己判断是否满意。编排 Skill 不需要就配色切换发任何消息，除非用户主动在聊天中讨论配色。

## 19. HTML 预览面板与渲染引擎的关系

### 19.1 两条独立的技术路径

```
HTML 预览面板路径：
  排版 Skill 输出 Poster 数组
    → 产品层推送 HTML 到面板
      → 面板浏览器视口渲染 HTML（< 100ms）
        → 用户即时查看

PNG 渲染路径（仅在用户确认后触发）：
  排版 Skill 输出 Poster 数组
    → 用户确认「就这样」「渲染吧」
      → 产品层启动 Playwright 截图管线
        → 字体加载 → 等待渲染完成 → 截图（1-3 秒/页）
          → PNG 缩略图发送到聊天
```

### 19.2 解耦的价值

两条路径独立意味着：
- HTML 面板可以一直可用，即使 PNG 管线故障（字体超时、引擎崩溃）
- 用户可以连续探索性调整（5 次、10 次），每次 < 100ms，完全不触发 PNG 管线
- PNG 渲染是「盖章确认」而非「预览查看」——只在用户满意后才触发

## 20. 字体管理

> **字体是渲染管线中最脆弱的环节。** 中文字体体积大（单个字重约 15MB 完整版）、渲染慢，必须在架构层面提前设计。

### 20.1 字体归属与存储

| 决策 | 说明 |
|------|------|
| **字体是全局资源，非 Style 独有** | 如果 5 个 Style 都使用思源黑体，字体文件存储 1 份而非 5 份 |
| **内置字体集** | 思源黑体 Regular + Bold、思源宋体 Regular + Bold（子集化为常用汉字 6763 个 + 标点符号） |
| **存储位置** | 字体文件存储在引擎共享目录（`fonts/`）。引擎启动时读取 `.woff2` 文件内容为 base64 data URI，注入 `@font-face` 声明，不依赖 `file://` 协议（保证沙箱严格隔离时字体仍然可用，且 Windows/macOS 路径格式一致） |
| **字体格式** | `.woff2`（浏览器原生支持，体积最小） |

### 20.2 字体子集化

| 参数 | 值 |
|------|-----|
| 子集化范围 | 常用汉字 6763 个（覆盖 99.9% 日常中文使用场景）+ ASCII 可打印字符 + 常用中文标点 |
| 子集化后体积 | 4 字重（Sans Regular + Sans Bold + Serif Regular + Serif Bold）合计约 **5.4MB** |
| 完整版体积 | 4 字重完整 woff2 约 19MB |

> ⓘ **字体版权备注：** 思源黑体（Source Han Sans）与思源宋体（Source Han Serif）使用 **SIL Open Font License 1.1**（OFL 1.1），明确允许子集化、嵌入应用随产品分发、商业用途、渲染为图片输出。唯一的合规要求：(1) 子集化后须使用自定义名称（如 `app-text-sans-regular.woff2`），不得保留「Source Han Sans」「思源黑体」等原名；(2) 在产品「关于/开源许可」页面附带 OFL 1.1 许可证全文（约 4KB）。无需付费、无需联系版权方、无法律风险。

### 20.3 字体加载策略

| 场景 | 策略 |
|------|------|
| **@font-face 声明** | 由渲染引擎统一注入（而非 Template 各自声明），`font-display: block`（渲染前等待字体加载，避免 FOUT） |
| **PNG 渲染时的字体等待** | 单页字体等待 ≤ 5 秒。超时则用 fallback 字体（系统默认中文字体）继续渲染，不阻塞 PNG 管线 |
| **HTML 预览面板的字体** | 若字体尚未加载完成，面板使用系统默认字体 + 半透明覆盖层提示「字体加载中」；加载完成后自动切换 |
| **字体回退链** | 所有 Template 的 `font-family` 声明必须包含 fallback 到内置字体：`--font-body` 默认值必须以思源宋体开头，以 `serif` 结尾 |

### 20.4 跨 Style 字体一致性问题

Recipe 跨 Style 复用时，字体变化可能影响排版：

```
Recipe「杂志封面 M01」引用 h-xl（标题 class）：
  Style A 的 h-xl → font-family: "Source Han Serif SC", serif（衬线体）
  Style B 的 h-xl → font-family: "Source Han Sans SC", sans-serif（无衬线体）
  同样的文字，在两种字体下可能产生不同的行数和溢出情况
```

**缓解措施：**
- 排版 Skill 填入标题时应使用 Contract 中的 `maxCharsPerLine` 约束（该约束应在字体最宽松的情况下设定）
- 校验规则 R1（溢出检测）和 R7（标题超限）在切换 Style 后自动重新执行
- Recipe 的 `contentLimits` 约束应比 Contract 更严格（而非更宽松），为跨 Style 字体差异预留缓冲

### 20.5 样式分发时的字体依赖

`.style` 包**不包含字体文件**。字体依赖在 `manifest.json` 中声明：

```jsonc
{
  "fonts": {
    "required": ["source-han-sans-regular", "source-han-sans-bold"],
    "optional": ["source-han-serif-regular"]
  }
}
```

用户安装 Style 时，若系统缺少必需字体，产品层提示下载（内置字体集已预装，此场景仅在三方 Style 引用内置集之外的特殊字体时出现）。

## 21. 图片处理管线

### 21.1 管线总览

```
用户上传图片
  → 校验（尺寸、格式、文件大小）
    → 存储至项目 assets/ 目录
      → 排版 Skill 分配 slot
        → 渲染引擎加载图片
          → 适配 slot 尺寸（缩放/裁剪）
            → 渲染
```

### 21.2 上传校验

| 检查项 | 规则 |
|--------|------|
| 格式 | 仅接受 JPEG、PNG、WebP |
| 最小尺寸 | 宽 ≥ 200px 且 高 ≥ 200px（低于此值警告但允许使用） |
| 最大尺寸 | 宽 ≤ 8000px 且 高 ≤ 8000px（超出则自动缩放至上限） |
| 文件大小 | 单张 ≤ 20MB |
| 颜色配置文件 | 自动转换为 sRGB（避免 CMYK 图片在浏览器中色差） |

### 21.3 适配策略（由渲染引擎执行）

| slot 声明的 objectFit | 行为 |
|-----------------------|------|
| `cover`（如 `img-hero`） | 等比例缩放填满 slot 区域，居中裁剪。宽高比与 slot 不一致时，超出部分被裁剪 |
| `contain`（如 `img-inline`） | 等比例缩放适配 slot 区域，留白不裁剪 |
| 无声明（默认） | 使用 `cover` |

**裁剪锚点：** 默认居中裁剪。v1.x 支持用户为每张图片单独调整裁剪区域（拖拽图片在 slot 内的位置）。

### 21.4 渲染时的图片路径处理

Recipe HTML 中的图片引用使用相对路径（`src="assets/img-003.jpg"`），但渲染沙箱**不依赖 `file://` 协议**（见 §22.4）。渲染引擎在渲染前：

1. 解析 Recipe HTML 中所有 `src` 属性
2. 将图片文件读取为 base64 data URI，注入 `<img src="data:image/jpeg;base64,...">`
3. 此策略保证沙箱严格隔离（无需 `file://` 权限），同时消除 Windows/macOS 路径格式差异

### 21.5 多输出规格下的图片适配

同一张图片在不同画布比例下（如小红书 3:4 vs 公众号 21:9），`objectFit: cover` 的裁剪结果不同。这是预期行为——不同平台的视觉需求不同。用户若对裁剪不满意，可在微调阶段手动替换该 slot 的图片。

## 22. 渲染引擎技术规格

### 22.1 推荐配置（v1.0）

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| **引擎** | Playwright + Chromium（headless-shell 模式） | headless-shell 对 CSS 渲染保真度与 headed 模式一致，字体渲染、抗锯齿、颜色配置文件无差异 |
| **视口 (viewport)** | `width=1080, height=1440`（匹配画布尺寸） | 每个规格使用对应尺寸的视口 |
| **deviceScaleFactor** | `2`（输出 2160×2880 PNG，Retina 品质） | 小红书图片在 Retina 屏幕上展示时不会模糊 |
| **并发模型** | 1 个 Browser Context + N 个 Page（N = 并发数，默认 2） | Context 约 20MB 内存，Page 约 5MB；比启动多个 Browser 实例节省大量内存 |
| **Browser Context 生命周期** | 应用启动时创建，应用退出时销毁 | 避免每次渲染都启动 Chromium（冷启动约 1-2 秒） |
| **Page 生命周期** | 每次渲染任务后关闭并新建 | 避免跨渲染任务的 CSS/JS 状态污染 |
| **内存上限** | 渲染进程总内存 ≤ 1.5GB | 超出时排队等待，不创建新 Page |

### 22.2 内存估算

| 场景 | 估算内存 | 说明 |
|------|----------|------|
| 1 个 Browser Context（无 Page） | ~300MB | Chromium headless-shell 基础开销 |
| 2 个 Page 并发渲染 | ~300MB + 2 × 5MB ≈ 310MB | 最小并发 |
| 7 页批量渲染（2 并发 × 4 轮） | 峰值 ~310MB | 分页渲染，不会同时打开 7 个 Page |
| 含 5.4MB 字体缓存 | +10MB（字体在 GPU 纹理中） | — |
| **总计** | **≤ 500MB**（远低于 1.5GB 上限） | — |

### 22.3 颜色管理

- 渲染输出使用 sRGB 颜色空间
- CSS 中的颜色值（hex、rgb()、hsl()）原样传递，不做色彩空间转换
- 若用户上传图片含嵌入的 ICC Profile（如 Adobe RGB），渲染引擎在加载时自动转换为 sRGB

### 22.4 沙箱文件访问策略

> **核心原则：渲染沙箱不依赖 `file://` 协议。** 这不仅是为了安全隔离，也为了跨平台一致性——Windows（`file:///C:/...`）与 macOS（`file:///Users/...`）的路径格式不同，且 Chromium 沙箱可能在不同平台上对 `file://` 有不同的权限策略。

| 资源类型 | 访问方式 | 说明 |
|----------|----------|------|
| **字体** | 引擎启动时将 `.woff2` 文件读取为 base64 data URI，注入 `@font-face` | 字体体积小（子集化后 5.4MB），base64 编码开销可接受；一次性注入，所有 Page 共享 |
| **图片** | 渲染前将图片文件读取为 base64 data URI，替换 Recipe HTML 中的 `src` | 单次渲染的图片通常 ≤ 5 张，base64 编码后总大小 ≤ 20MB（原始图片 ≤ 20MB × 4/3 ≈ 27MB base64），在内存中可接受 |
| **Template CSS/HTML** | 引擎从本地文件系统读取，以字符串形式传入 Page 的 `setContent()` | 不经过 URL 加载，直接注入 DOM |

> **不使用 `file://` 的性能影响：** base64 data URI 比 `file://` 路径多约 33% 体积（Base64 编码开销），但消除了跨平台路径兼容问题和沙箱权限配置。对于 5.4MB 字体 → 7.2MB base64 和 ≤ 20MB 图片 → ≤ 27MB base64，内存开销在桌面应用中完全可接受。字体 base64 在 Browser Context 生命周期内仅注入一次，所有 Page 共享。

## 23. 资源加载与离线策略

### 23.1 组件资源打包

5 个 Style × 12 种页面类型 × 4-6 套配色 × 60+ Recipe = 大量 HTML/CSS/JSON 资源。加载策略如下：

| 时机 | 加载内容 | 说明 |
|------|----------|------|
| **应用启动时** | 5 个 Style 的 Contract 元数据（YAML） | Contract 极小（每个约 2-5KB），解析后即可展示风格列表和 Recipe 元数据摘要 |
| **用户选择 Style 后** | 该 Style 的 Template（HTML + CSS）+ 全部 Recipe HTML + 全部 Theme | 首次使用某 Style 时懒加载完整资源 |
| **切换 Style 时** | 新 Style 的完整资源（若未缓存则从磁盘加载） | 目标：加载时间 ≤ 500ms |
| **缓存在内存中** | 当前活跃 Style 的全部 Recipe 和 Theme | 切换版式、切换配色时无需磁盘 I/O |

### 23.2 资源打包格式

| 资源 | 打包格式 | 说明 |
|------|----------|------|
| 内置 5 个 Style | 编译进应用包（不作为独立 `.style` 文件） | 所有内置资源随应用安装，不依赖网络 |
| 三方 Style | `.style` 文件（zip 压缩） | 导入时解压至应用数据目录 |
| 字体 | 独立于 Style，编译进应用包 | 所有 Style 共享 |

### 23.3 离线能力

| 功能 | 离线可用 | 说明 |
|------|----------|------|
| 核心排版生成（Skill） | 取决于 Skill 部署方式 | 若 Skill 为云端 API，离线不可用；若为本地模型，完全离线可用 |
| HTML 预览面板 | ✅ | 不依赖网络 |
| 配色切换 | ✅ | 纯 CSS 操作 |
| PNG 渲染 | ✅ | Playwright 内置 Chromium，完全本地 |
| 导出 | ✅ | 写入本地磁盘 |
| 风格/版式浏览 | ✅ | 所有内置资源本地存储 |
| 图库搜索 | ❌ | 需要网络；离线时入口灰显 |
| AI 图片生成 | ❌ | 需要网络；离线时入口灰显 |

## 24. 多项目与状态隔离

> **v1.0 仅支持单项目操作。** 本节描述架构预留，不影响 v1.0 工程实现。

### 24.1 需要隔离的状态

若未来支持多项目并发（如用户同时编辑两篇不同的文章），以下状态必须 per-project 隔离：

| 状态 | 隔离方式 |
|------|----------|
| HTML 预览面板展示内容 | 切换项目时面板整体替换为新项目的 Poster 数组 |
| 编排 Skill 对话上下文 | 每个项目拥有独立的对话历史（聊天上下文按 `projectId` 分组） |
| CSS 变量（当前 Theme） | per-project：项目 A 使用「午夜墨」，项目 B 使用「暖纸经典」 |
| 图片未分配池 | per-project：项目 A 上传的图片不可被项目 B 的排版 Skill 使用 |
| 校验结果 | per-project：切换项目时重新执行校验 |

### 24.2 架构预留

所有协议（Skill 输入/输出、面板推送消息、项目文件）的顶层均预留 `projectId` 字段。v1.0 中该字段为固定值（单项目），不实现多项目切换 UI。

```jsonc
{
  "projectId": "uuid",  // v1.0 固定值；v2.0 多项目时动态切换
  // ... 其余协议字段
}
```
