# SpaceAssistant 可用性审计报告

**日期**：2026-05-29 ~ 2026-05-30
**审计范围**：SpaceAssistant 全功能模块，聚焦普通用户与准专业用户的使用体验
**方法**：代码审查 + 界面分析 + 角色扮演式旅程地图

---

## 审计一：完全不懂 AI 的普通办公用户

### 用户画像

普通办公人员，会用 Word/Excel、浏览器、微信/飞书，但不知道什么是"大模型""Token""上下文窗口""API Key"。

---

### 第一部分：可用性问题清单

#### 🔴 致命障碍 —— 用户完全无法开始使用

##### 1. 首次启动需要配置 API Key，但用户不知道那是什么

**场景**：用户打开应用，在输入框输入"帮我写个周报"，提示「请先在设置中配置 API Key」。

**困惑**：
- "API Key 是什么？哪里来的？"
- 设置页面有「大模型服务」标签，里面有 baseUrl、API Key 等字段，但没有引导说明去哪里获取
- 用户不知道这东西是花钱的（按 token 计费），可能用了很久才发现账单惊人

**影响**：**100% 的新用户在此处卡住**，除非有人帮他们配置好

##### 2. 模型列表令人恐惧

**场景**：用户进入「默认大模型设置」，看到 11 个模型：`kimi-k2.6`、`glm-5.1`、`minimax-m2.7`、`deepseek-v4-pro`、`deepseek-v4-flash`、`claude-sonnet-4-6`、`claude-opus-4-7`、`claude-haiku-4-5`、`gpt-5.5`、`gemini-3.1-pro`、`gemini-3.1-flash-lite`

**困惑**：
- "这些名字是什么意思？有什么区别？"
- "我应该选哪个？选错了会怎样？"
- "默认/快速是什么意思？"
- "上下文 200K、输出 64K 是什么？"
- "Temperature 是什么？为什么有个滑杆从 0 到 2？"

**影响**：用户要么随便选（可能导致效果差），要么不敢动（但默认值可能不合适）

##### 3. 工作目录概念不清

**场景**：设置页第一个必填项就是「工作目录（会话明文备份等）」，带红色星号必填标记。

**困惑**：
- "为什么要选工作目录？它用来干什么？"
- "我应该选哪个文件夹？选桌面行不行？"
- "会话明文备份是什么意思？会泄露我的隐私吗？"

**影响**：这是保存设置的前置条件，卡住就会导致所有配置无法保存

---

#### 🟠 主要困惑 —— 用户不知道功能是干什么的

##### 4. "Plan 模式"是一个黑盒概念

**场景**：输入框旁边有「普通模式」/「Plan 模式」的下拉选择器，设置页有专门的「Plan 模式」标签页。

**困惑**：
- "普通模式和 Plan 模式有什么区别？"
- "我什么时候应该用 Plan 模式？"
- 设置中的选项：「执行方式（自动连续执行/每步完成后手动继续）」「工具确认策略（计划级信任/逐步逐工具确认/全部自动批准）」「自动批准 Agent 生成的脚本」——这些选项的名字对普通用户来说完全是天书
- "Agent 生成的脚本是什么？为什么需要批准？"

**影响**：用户永远只用普通模式，Plan 模式的强大功能完全被浪费

##### 5. "工具调用"是什么？为什么 AI 要操作我的文件？

**场景**：用户发了一条消息，突然弹出一个 diff 预览卡片，显示 AI 想修改某个文件，底部有「确认」和「拒绝」按钮。

**困惑**：
- "AI 为什么要改我的文件？我没让它改啊！"
- "这个 diff 是什么？绿色和红色是什么意思？"（diff 是程序员概念）
- "我该点确认还是拒绝？拒绝会不会让 AI 生气/出错？"
- 看到代码预览卡片（`run_script` 工具），里面是 Python 代码——"这什么？为什么要我看代码？"

**影响**：用户对 AI 产生不信任感，可能全部拒绝导致功能不可用，或全部确认导致安全风险

##### 6. Skill 系统完全不可见

**场景**：聊天框提示「输入问题，或尝试 /skill、/wiki 命令」。设置页有一个「Skill」标签页。

**困惑**：
- "Skill 是什么？和普通聊天有什么区别？"
- "/skill 命令怎么写？有什么可用的 skill？"
- "我应该主动用 skill 还是 AI 会自动用？"
- "为什么有时候消息旁边会出现一些系统提示文字（skill hint）？"

**影响**：Skill 系统是产品的核心竞争力之一，但普通用户完全不会用

##### 7. LLM Wiki 概念陌生

**场景**：设置页有「LLM Wiki」标签，里面有 `rootPath`、`hideWikiFromFileTree`、`interactiveIngest`、`maxBatchIngest` 等选项。

**困惑**：
- "LLM Wiki 是什么？和维基百科有什么关系？"
- "interactiveIngest（交互式摄入）是什么意思？"
- "为什么有些助手回复旁边有个「归档到 Wiki」按钮？点了会怎样？"
- 默认是关闭的——用户根本不知道这个功能存在

---

#### 🟡 次要摩擦 —— 能用但体验不佳

##### 8. "Thinking（思考过程）"的展示令人困惑

**场景**：发送消息后，在 AI 正式回复之前，出现一个可折叠的「思考中…」区块，里面是 AI 的内部推理文字。

**困惑**：
- "这是 AI 在自言自语吗？"
- "我需要看这些吗？还是可以忽略？"
- "为什么有时候有，有时候没有？"

##### 9. 左侧边栏的功能分区不够直观

**场景**：左侧边栏有三个图标（会话、文件、搜索），没有文字标签。

**困惑**：
- 只有图标没有文字，需要悬停才能看到 tooltip
- "文件"标签里的内容和工作目录是什么关系？
- 三个标签之间的切换逻辑是什么？

##### 10. 飞书集成的配置门槛极高

**场景**：设置页「飞书」标签页包含 CLI 自动安装、账号认证、事件监听等配置。

**困惑**：
- "我没有飞书账号，这个能用吗？"
- "CLI 是什么？为什么要安装 CLI？"
- 配置项目繁多（App ID、App Secret、事件订阅等），即使是飞书用户也难以独立完成

##### 11. 浏览器自动化功能普通人不会用

**场景**：设置页「浏览器」标签页有 `Stagehand Model`、`actionTimeoutSec`、`maxInferencesPerRequest`、`navigateRequiresConfirm`、`actRequiresConfirm` 等选项。

**困惑**：
- "浏览器自动化是什么意思？"
- "Stagehand 是什么？"
- 这些选项即使看了说明也很难理解实际效果

##### 12. 工具设置页的技术术语堆砌

**场景**：设置页「工具」标签页包含 `Python 路径`、`脚本默认超时（秒）`、`grep 超时（秒）`、`文件历史备份`、`每文件最多快照数`、`白名单模式` 等。

**困惑**：
- "grep 是什么？"
- "文件快照是什么意思？"
- "白名单模式和黑名单模式有什么区别？我该选哪个？"

### 第二部分：新手用户旅程地图

**角色**：小王，30 岁，市场部职员。每天用 Word 写方案、Excel 做报表、飞书沟通。听说过 ChatGPT 但没用过。同事推荐了 SpaceAssistant 说"能帮你干活"。

---

#### 🕐 阶段 0：下载安装（✅ 相对顺利）

小王下载了安装包，一路"下一步"装好。桌面出现 SpaceAssistant 图标。

**卡点**：无。安装流程符合预期。

---

#### 🕑 阶段 1：首次启动（🔴 卡住）

双击图标，应用启动。小王看到三栏布局：左侧有一列图标，中间是空白聊天区，写着「选择或创建一个会话」。

小王点击左侧第一个图标，看到会话列表为空。他点"新建会话"，然后在输入框输入：

> "帮我写一个市场推广方案"

按下 Enter。

❌ **弹窗提示**：「请先在设置中配置 API Key」

**小王的心声**："API Key？这是什么东西？我没听说过啊。"

小王打开设置（齿轮图标），看到 8 个标签页：通用、Plan 模式、大模型服务、默认大模型设置、工具、LLM Wiki、浏览器、飞书、Skill。

**小王的心声**："这也太复杂了吧……我就想让它帮我写个方案而已。"

他点进「大模型服务」，看到需要填 `Base URL`、`API Key`、`API Key ID`。没有任何说明告诉这些从哪里获取。

**小王的心声**："算了，我去问同事吧。"

**👉 第一个放弃点**

---

#### 🕒 阶段 2：同事帮忙配好（🟠 持续困惑）

同事帮小王配好了 API Key 和模型（选了 `claude-sonnet-4-6`）。小王又回来输入：

> "帮我写一个市场推广方案"

这次 AI 开始回复了！文字一个字一个字地出现（流式输出），小王觉得还挺酷的。

但回复过程中，出现了奇怪的东西：

1. **Thinking 区块**：一个可折叠的灰色区域，里面写着 "让我分析一下用户的需求。用户要求写市场推广方案，我需要了解……"

   **小王的心声**："这什么东西？AI 在自言自语？我需要管它吗？"

2. **Tool Call 卡片**：突然弹出一个卡片，显示 `write_file`，内容是一段 diff 预览，绿色和红色的文字对比。

   **小王的心声**："？？？我让它写方案，它为什么要在我的电脑上创建文件？它在改什么东西？"

   卡片底部有「确认」和「拒绝」按钮。

   **小王的心声**："我该点哪个？点了确认会不会把我电脑上的文件弄坏？点了拒绝是不是就没方案了？"

   👉 小王犹豫了 30 秒，点了「确认」。方案文件被写入了工作目录。

---

#### 🕓 阶段 3：尝试进阶功能（🟠 迷失）

用了几天后，小王已经习惯了基本对话。但他注意到了一些他不懂的东西：

1. **输入框旁边的下拉菜单**：「普通模式」vs「Plan 模式」

   **小王的心声**："Plan 模式是什么？和普通模式有什么区别？我试试看。"

   切换到 Plan 模式后，AI 不再直接回答，而是先输出一个"计划"——步骤列表。然后小王需要点「批准」才能继续。

   **小王的心声**："为什么要先计划再执行？我之前的普通模式不是也能出方案吗？Plan 模式到底好在哪？"

2. **设置页的 Skill 标签**：小王好奇点进去，看到一堆 skill 名称和开关。

   **小王的心声**："lark-doc、lark-sheets……这些和飞书有什么关系？我需要手动开吗？它们什么时候会自动用？"

3. **「归档到 Wiki」按钮**：有些 AI 回复下方出现了一个小按钮「归档到 Wiki」。

   **小王的心声**："Wiki 是什么？点了之后去哪儿了？我能在哪找到归档的内容？"

---

#### 🕔 阶段 4：飞书集成（🔴 放弃）

小王的团队用飞书。他看到设置页有「飞书」标签，想试试。

点进去看到：CLI 自动安装、App ID、App Secret、事件订阅 URL、加密 Key……

**小王的心声**："算了，这完全不是给我用的。"

**👉 第二个放弃点**

---

#### 🕕 阶段 5：日常使用中的摩擦（🟡）

小王已经习惯了基本用法，但日常仍有不适：

1. **模型选择焦虑**：偶尔 AI 回复质量不好，同事说"换个模型试试"。小王打开设置，面对 11 个模型名称，不知道选哪个。

2. **不理解计费**：小王不知道每次对话会消耗多少 tokens，设置页没有任何用量统计或费用估算。

3. **工具审批疲劳**：每次 AI 想写文件都弹确认框。小王已经养成肌肉记忆——直接点「确认」，不再看 diff 内容。安全机制形同虚设。

4. **无法判断 AI 是否在"胡说"**：AI 有时会给错误信息，小王无法分辨。没有引用来源、没有置信度提示。

---

#### 总结：普通用户 TOP 5 最严重问题

| 排名 | 问题 | 严重程度 | 影响用户比例 |
|------|------|----------|-------------|
| 1 | **API Key 配置无引导** | 🔴 致命 | 100% 新用户 |
| 2 | **模型选择无说明** | 🔴 致命 | 90% 用户 |
| 3 | **工具调用/Diff 确认不可理解** | 🟠 主要 | 80% 用户 |
| 4 | **Plan/Skill/Wiki 概念无入门引导** | 🟠 主要 | 70% 用户 |
| 5 | **飞书/浏览器等高级功能配置门槛过高** | 🟡 次要 | 50% 用户 |

**核心结论**：SpaceAssistant 目前是为**懂 AI 的开发者**设计的产品。它的功能强大、架构灵活，但每个功能模块都假设用户已经理解"大模型""Token""工具调用""Diff""Skill"等概念。对于一个普通办公人员来说，从"我想让 AI 帮我干活"到真正用起来，中间隔着至少 5 个不可逾越的认知鸿沟。

最关键的改进方向是：**首次使用引导（Onboarding）**——在用户首次打开应用时，用通俗的语言解释核心概念，帮助完成最小可用配置，而不是把一整套开发者级设置面板直接扔给用户。

## 审计二：了解 AI 但技术底子弱的准专业用户

### 用户画像

产品经理、运营、市场等非技术岗位，了解 AI/大模型基础概念，使用过 Claude Code 和 Cursor（小龙虾），知道 API Key、Token 是什么，有强烈学习动机，但缺乏编程/运维技术底子。怀着「这个工具比 Claude Code 更好上手」的期望来尝试。

---

### 第一部分：可用性问题清单

#### 🔴 致命障碍 —— "我以为会比 Claude Code 简单，结果更懵了"

##### 1. "黑盒感"比 Claude Code 更严重 —— 看不到 AI 在做什么

**背景**：Claude Code 的终端界面让用户能看到完整的执行过程——每一步工具调用、每一条系统消息，全都打印在屏幕上。用户虽然不一定全懂，但至少「看得见」。

**在 SpaceAssistant 中**：
- 工具调用折叠在可展开的卡片中，默认收起
- Skill 注入到 System Prompt 的过程完全不可见
- LLM 路由（自动匹配 Skill）在后台默默发生，用户只看到一条 `[Skill] 已加载: xxx（项目级，AI 匹配）` 的灰色提示
- Thinking 区块默认折叠，用户不知道 AI 的推理过程

**用户的认知断裂**：
> "在 Claude Code 里我至少看到它一行行在跑，SpaceAssistant 里我只看到它在'想'，然后突然弹出一个文件修改确认——它怎么知道要改哪个文件的？它什么时候读了文件？"

##### 2. Skill 系统：知道它很强，但不知道怎么"驾驭"它

**背景**：这批用户从 Claude Code 知道了「自定义指令」「hooks」的概念。他们听说 SpaceAssistant 有更强大的 Skill 系统，充满期待。

**实际体验**：
- 打开设置 → Skill 标签页，看到一张表：名称、作用域、描述、版本，以及三个配置项：「自动检测」「始终加载」「全局禁用」
- 他们有 50+ 个 lark-* 开头的飞书 Skill，但不知道每一个具体能做什么
- 「自动检测」开了，但不知道 AI 什么时候会匹配到哪个 Skill
- 聊天时有时会看到 `[Skill] 已加载: lark-doc（AI 匹配）`，有时没有——不知道为什么
- 「始终加载」vs「手动激活」vs「自动检测」vs「全局禁用」——四种状态矩阵，组合爆炸

**用户的认知断裂**：
> "我知道 Skill 是强大的，但我感觉是它在用我，不是我在用它。Claude Code 的 /command 至少是我主动打的，这里我不知道什么触发了什么。"

##### 3. Plan 模式 vs 普通模式：两个世界的割裂

**背景**：Claude Code 有 `/plan` 命令，Cursor 有 Agent 模式。用户理解「先规划再执行」的价值。

**实际体验**：
- 输入框旁有个下拉菜单：「普通模式」「Plan 模式」
- 切换到 Plan 模式发消息 → AI 生成一个计划 → 显示「等待批准」
- 批准后进入执行阶段——但此时用户**不知道这和普通模式到底有什么本质区别**
- Plan 执行中也有工具调用确认，和普通模式一样弹卡片
- Plan 有 7 种状态（drafting → awaiting_approval → approved → executing → completed/cancelled），3 种执行模式（auto/step_manual/step_confirm），4 种工具确认策略（trust_plan/trust_plan_all/always_confirm/confirm_high_risk）
- 设置页 Plan 模式标签还有「自动批准 Agent 生成的脚本」「步骤完成进度消息」等选项

**用户的认知断裂**：
> "我在 Claude Code 里用 /plan，它给我一个计划，我确认，然后它就执行了。这里有一堆设置选项，我都不知道选哪个对。而且为什么 Plan 模式下还有工具确认？不是应该自动执行吗？"

##### 4. 工具确认：从"理解风险"到"麻木点确认"

**背景**：Claude Code 的工具确认是「Allow/Deny/Always allow」。用户习惯了这种节奏。

**实际体验**：
- 每个写文件操作弹出一张卡片，里面是 diff 预览（绿行/红行）
- 执行 Python 脚本弹出一张卡片，里面是代码
- 浏览器的 navigate 和 act 也需要确认
- 风险等级标签（low/medium/high）和「需确认」标签并存
- 设置页有「文件写入确认模式」：diff 预览 vs 直接确认
- 还有白名单/黑名单模式切换

**用户的认知断裂**：
> "Claude Code 的确认至少我能看懂——它告诉我它要做什么。这里的 diff 我看不懂，Python 代码我也看不懂，但我又不能不确认，因为确认了任务才能继续。我就是在闭着眼点确认。"

##### 5. 模型选择：11 个模型，0 个决策依据

**背景**：用户在 Claude Code 中用的是 Claude 模型。他们知道不同模型有不同能力。

**实际体验**：
- 11 个预置模型：kimi-k2.6、glm-5.1、minimax-m2.7、deepseek-v4-pro/flash、claude-sonnet-4-6/opus-4-7/haiku-4-5、gpt-5.5、gemini-3.1-pro/flash-lite
- 每个模型只有两个数字：上下文 X M、输出 X K
- 有「默认」和「快速」标签但没有解释
- Temperature 滑杆 0-2，默认值未知
- 用户不知道该为「写周报」选哪个、为「数据分析」选哪个、为「飞书操作」选哪个

**用户的认知断裂**：
> "我知道 Claude Opus 比 Haiku 强，但 kimi-k2.6 和 glm-5.1 谁更强？deepseek 的 1M 上下文是真的能用吗？我该不该为了省钱用 flash 模型？"

---

#### 🟠 主要困惑 —— "功能都看到了，但用不起来"

##### 6. 知识库（Wiki）：我知道 RAG 是什么，但不知道怎么让它"吃"我的文档

**背景**：用户听说过「AI 可以读你的文档回答问题」，这是他们想用 SpaceAssistant 的核心场景之一。

**实际体验**：
- 设置页「LLM Wiki」标签默认关闭
- 开启后需要设置 rootPath、hideWikiFromFileTree、interactiveIngest、maxBatchIngest
- 「interactiveIngest」（交互式摄入）——用户猜这是「让 AI 读文档」的意思，但不确定
- 需要在聊天框输入 `/wiki ingest <文件路径>` 才能导入文档——用户不知道这个命令
- 聊天框提示「输入问题，或尝试 /skill、/wiki 命令」——但没有告诉用户有哪些 wiki 子命令
- 「归档到 Wiki」按钮出现在助手回复下方——但用户不知道归档后去哪了、怎么查看

**用户的认知断裂**：
> "我想让 AI 读我的项目文档然后回答我的问题。但在 Claude Code 里我直接把文件拖进去就行。这里我要先开 Wiki，然后不知道怎么导入文档，然后还要用特殊命令。太复杂了。"

##### 7. 工作目录的"虚拟边界"不透明

**背景**：Claude Code 在终端中运行，工作目录就是当前目录——用户看得见、摸得着。

**实际体验**：
- 设置页第一个必填项是「工作目录」
- AI 读写的所有文件都在这个目录下
- 左侧文件浏览器也显示这个目录
- 但用户可能选了 `C:\Users\小王\Documents`，然后让 AI「帮我分析这个 Excel」——AI 说找不到文件，因为文件在 `Desktop`
- AI 创建的文件不知道去哪了——用户需要去文件浏览器里找

**用户的认知断裂**：
> "Claude Code 的 pwd 让我知道我在哪。这里的'工作目录'是个抽象的设置项，我选了之后就没再想过它。AI 创建的文件到底放在哪了？"

##### 8. 飞书集成：最期待的功能，最高的门槛

**背景**：用户团队用飞书，最期待的就是「让 AI 帮我处理飞书消息、写文档、做报表」。

**实际体验**：
- 设置页「飞书」标签需要：CLI 安装、App ID、App Secret、事件订阅
- 「CLI 自动安装」按钮点了之后不知道在干什么
- 看到 `lark-doc`、`lark-sheets`、`lark-im` 等 Skill 在列表中，但不知道如何让它们工作
- 不知道「远程代理执行（飞书 Bot 调用）」是什么场景
- 飞书消息能触发 AI 回复——但不知道怎样让这个功能跑起来

**用户的认知断裂**：
> "50 多个飞书 Skill 摆在那里，一个都用不了。我不知道该先配什么后配什么。Claude Code 里至少没有这种'看得到吃不到'的情况。"

##### 9. 浏览器自动化：概念听过，实操抓瞎

**背景**：Claude Code 有 browser 工具，用户见过它打开网页抓内容。觉得酷。

**实际体验**：
- 设置页「浏览器」标签有：env（LOCAL/BROWSERBASE）、allowedDomains、trustedDomains、headless、stagehandModel、actionTimeoutSec、idleTimeoutSec、maxInferencesPerRequest、navigateRequiresConfirm、actRequiresConfirm、deniedActions、allowRemoteSessions、captureSubdir
- 大部分选项对用户来说完全陌生
- 「Stagehand Model」——用户甚至不知道 Stagehand 是什么
- 「headless」默认 true——用户看不到浏览器窗口，不知道 AI 到底有没有在访问网页

**用户的认知断裂**：
> "我就想让 AI 帮我查一个网页上的信息。Claude Code 里它直接就打开浏览器了。这里我要先配域名白名单，还要选本地还是远程浏览器，还要决定是否 headless……我全用默认值，但默认值对不对我也不知道。"

---

#### 🟡 次要摩擦 —— "能跑通，但不顺手"

##### 10. 看不到"完整对话记录"——Claude Code 用户的信息焦虑

Claude Code 的终端输出是完整的、线性的、可回滚的。SpaceAssistant 的聊天界面中，Thinking 折叠、工具调用折叠、Skill 提示一闪而过。用户无法完整回顾「AI 到底做了什么」，缺乏安全感。

##### 11. Python 依赖是隐形的炸弹

`run_script` 工具需要 Python。用户可能没装 Python，或者装了但路径不对。设置页有个「Python 路径」配置项和「测试」按钮——但用户第一次知道需要 Python 时，可能是在 AI 报错的时候。

##### 12. 设置页信息过载——8 个标签页，每个都有 5-15 个配置项

Claude Code 的配置在 `.claude/settings.json` 里，用户可以不看。SpaceAssistant 把一切都摊在 GUI 里，总共有 50+ 个配置项。用户不知道哪些是「必须配的」、哪些是「默认就好的」。

##### 13. 会话管理的"断点续传"心智模型缺失

用户关闭应用后重新打开，会话还在——但正在执行的任务会被标记为 `paused_blocked`，提示「执行已中断，是否继续？」。用户不知道点了「继续」之后 AI 能不能接上之前的上下文。

**他实际得到的东西**：
- ✅ 一个界面更友好的 AI 聊天工具

#### 总结：准专业用户 TOP 5 阻碍

| 排名 | 阻碍 | 严重程度 | 根因 |
|------|------|----------|------|
| 1 | **Skill 系统可见性差，无法主动驾驭** | 🔴 致命 | 有 60+ Skill 但用户不知道每个做什么、何时触发、如何手动控制 |
| 2 | **飞书/浏览器等高级功能配置门槛过高** | 🔴 致命 | 需要开发者知识（App ID、Secret、域名白名单），准专业用户不具备 |
| 3 | **工具确认缺少可理解的决策信息** | 🟠 主要 | Diff/代码预览对非开发者无意义，导致"麻木点确认" |
| 4 | **Plan 模式与普通模式的差异化价值不清晰** | 🟠 主要 | 用户感受不到 Plan 模式的实际优势，配置选项增加了焦虑 |
| 5 | **配置项总数过载 + 缺少分层引导** | 🟡 次要 | 50+ 配置项平铺在 8 个标签页，没有「基础/进阶」分层 |

**核心结论**：这批用户和完全不懂 AI 的用户相比，**他们的问题不再是「不知道这是什么」，而是「知道这是什么但不知道该怎么用好」**。他们对 Claude Code/Cursor 的体验形成了参照系，但这个参照系既是助力（让他们能过 API Key 这一关）也是负担（他们期望 SpaceAssistant 比 Claude Code 更简单，结果发现它只是换了一种复杂的方式）。

最关键的改进方向：**从「暴露所有能力」转向「引导式的能力发现」**——不是把 60 个 Skill、50 个配置项、9 个工具平铺给用户，而是根据用户的实际场景（「我想用飞书」「我想让 AI 读我的文档」「我想做一个复杂任务」），一步一步引导他们完成配置和使用。

## 附录：跨用户群体的共性发现

### 两类用户的共同痛点

| 问题 | 普通用户 | 准专业用户 |
|------|----------|-----------|
| 设置页信息过载 | 完全看不懂 | 能看懂但不会选 |
| 工具确认 | 不知道 AI 在干什么 | 知道在干什么但不懂 diff/代码 |
| Plan 模式 | 不知道是什么 | 知道概念但感受不到差异化价值 |
| 飞书集成 | 完全放弃 | 想用但配不起来 |
| Skill 系统 | 完全不知道存在 | 知道存在但无法驾驭 |

### 设计原则建议

1. **渐进式信息披露**：设置页应分为「基础配置」（3-5 项）和「高级配置」（其余），默认只展示基础配置
2. **场景化引导**：根据用户意图（"我想处理飞书消息""我想让 AI 读文档"）引导式配置，而非平铺所有选项
3. **可理解的决策信息**：工具确认不应展示原始 diff/代码，而应用自然语言解释"AI 要做什么、为什么、有什么风险"
4. **首次使用向导**：覆盖 API Key 配置、模型选择、工作目录设置三个最关键步骤
5. **Skill 发现机制**：在聊天界面提供 Skill 浏览/搜索/预览功能，让用户能主动探索而非被动等待 AI 匹配
