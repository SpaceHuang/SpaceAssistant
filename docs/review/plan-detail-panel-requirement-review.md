# Plan 详情面板需求规格 — 评审意见

**评审对象：** [plan-detail-panel-requirement.md](../../requirement/plan-detail-panel-requirement.md) v1.0
**评审日期：** 2026-05-20
**评审状态：** 有条件通过（需解决下方问题后方可进入实现）

---

## 一、界面逻辑疏漏

### 1.1 文件预览互斥时的 Plan 状态丢失风险

**问题：** 第 3.2 节规定 `selectedFile` 非空时 PlanPanel 整体不展示。但未说明——用户在 PlanPanel 展示审批视图（状态 C）期间，如果点击了引用文件列表中的文件或文件树中的文件，PlanPanel 被隐藏，用户将**完全看不到**待审批提示。

**风险：**
- 用户打开文件后忘了回来审批，计划无限等待。
- 没有任何视觉提示告诉用户"还有一个待审批的计划需要处理"。

**建议：**
- 增加一个**全局提示机制**：当存在 `awaiting_approval` 计划且 PlanPanel 被隐藏时，在右侧栏顶部（FileOverlay 之上或文件预览工具栏旁）显示一个不可忽略的角标/横幅，如"1 个计划待审批"。
- 或者在 FileOverlay 关闭按钮旁显示提示气泡。
- 需求文档应明确此场景的交互规格。

### 1.2 状态 C 审批视图中缺少旧计划上下文

**问题：** 第 5.3.3 节推荐方案为状态 C 时仅显示审批视图，不并排展示旧计划全文，底部仅一行灰色说明。但用户审批新计划时，可能需要对照旧计划的执行进度来判断"新计划是否会覆盖当前进度"。

**具体场景：**
- 旧计划执行到 3/5 步，新计划待审批。用户想知道"批准新计划后，旧计划那 3 步的结果还在不在？"
- 一行灰色文案无法回答这个问题。

**建议：**
- 审批区底部的那行灰色说明应当更具体，至少包含旧计划的**状态摘要**（如"当前有 1 个执行中计划，已完成 3/5 步"），而非笼统的合并规则说明。
- 或者增加一个"查看当前计划"的轻量展开入口，点击后在审批视图下方折叠展示旧计划卡片（只读、不可操作），实现复杂度可控。

### 1.3 "修改计划"按钮的行为歧义

**问题：** 第 5.3.2 节"修改计划"按钮描述为"聊天输入修改意见或跳转计划文件编辑"。这两个行为差异巨大，用户无法预期点击按钮后会发生什么。

**风险：**
- 跳转编辑计划文件（`.spaceassistant/plans/*.md`）是**高危操作**——用户可能手动破坏 frontmatter 或 Markdown 结构，导致解析失败。
- MVP 主文档 4.4 节的原始设计是"卡片内编辑关键字段"或"聊天补充意见"，需求文档将其简化为"跳转计划文件编辑"，降低了安全性。

**建议：**
- "修改计划"按钮应**明确行为**：点击后将焦点移至聊天输入框，并自动插入一段引导文字如"请描述你对计划的修改意见："，引导用户通过对话迭代。
- 跳转文件编辑应作为**次要操作**（如"在编辑器中打开"链接），并标注风险提示。
- 不建议 MVP 阶段支持直接在审批视图内编辑字段（与主文档一致），但也不应引导用户直接编辑文件。

### 1.4 拒绝并反馈的交互未明确

**问题：** 第 6.2 节提到"用户填写拒绝原因（弹窗或面板内输入框）"，但未确定 MVP 采用哪种方案。两种方案的用户体验和实现复杂度差异显著：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 弹窗 | 不打乱面板布局 | 弹窗内容有限，不适合长文本反馈 |
| 面板内输入框 | 无遮挡，可写较长反馈 | 占用审批视图空间，审批区布局需调整 |

**建议：**
- MVP 应明确选择一种方案。推荐**面板内输入框**：在"拒绝并反馈"按钮点击后，按钮区替换为一个 textarea + "提交反馈"按钮，交互在 PlanPanel 内闭环，无需弹窗。
- 或者更简化的方案：点击"拒绝"后直接在聊天输入框聚焦，附带引导文字，反馈自然成为对话的一部分（与"修改计划"一致）。

### 1.5 plan_abort 终止报告的"关闭"操作缺少清除条件

**问题：** 第 5.4 节提到 `plan_abort` 存在时展示终止报告卡片，可提供"关闭"清除 abort 元数据。但未说明：

- 关闭后，面板进入什么状态？（A 空态？B 计划列表？）
- 如果终止前有已展示的计划（displayPlans 非空），关闭后应恢复状态 B 吗？
- abort 元数据被清除后，如果用户切换会话再切回来，abort 还能恢复吗？

**建议：**
- 明确"关闭"后面板状态判定逻辑：遵循第 4 节的状态优先级规则（检查是否有待审批 → 有无 displayPlans → 空态）。
- 建议不清除 `metadata.plan_abort`，仅标记为已读（如 `plan_abort_dismissed: true`），避免审计信息丢失。

### 1.6 探索期（drafting）面板状态未独立定义

**问题：** 第 4 节声明三种互斥状态 A/B/C，但第 5.4 节提到 `plan_drafting === true` 时在状态 A 或 B 上叠加 Spin + 文案。这导致：

- 面板实际上有 **3×2 = 6 种**视觉组合（A+drafting、A+非drafting、B+drafting、B+非drafting、C、C+drafting），但需求只定义了 3 种。
- `plan_drafting` 与 `status: drafting` 的关系未说明：如果 `metadata.plan.status === 'drafting'`，`plan_drafting` 是否一定为 `true`？两者是否可能不同步？

**建议：**
- 将 `drafting` 明确定义为**叠加态**（overlay），在第 4 节状态图中补充叠加规则：
  - 叠加态仅存在于状态 A 和 B 上，状态 C 不叠加（因为 C 本身就表示"有内容要展示"）。
  - 若 `plan_drafting === true` 且存在 `awaiting_approval`，应优先展示 C（审批优先原则已覆盖此场景，但应显式声明）。
- 明确 `plan_drafting` 和 `PlanMeta.status === 'drafting'` 的数据同步关系。

## 二、逻辑不合理之处

### 2.1 已完成计划替换规则可能导致信息丢失

**问题：** 第 7.3 节规定旧计划 `completed` 时批准新计划，**整表替换为仅含新计划**。这意味着旧计划的完成记录从面板中消失。

**风险：**
- 用户可能想回顾"上次做了什么"，替换后找不到。
- 主文档 4.7 节提到 `completed` 状态恢复时"仅展示结果摘要"，但本需求中替换后连摘要都没有了。

**建议：**
- 替换不应**删除**已完成计划，而应**折叠/归档**：已完成计划以折叠形式保留在列表底部（标题+完成状态徽标），新计划仍为主要内容。
- 或者至少在替换时提供"查看历史计划"入口，链接到 `plan_versions`。
- MVP 可折中：保留最近 1 条已完成计划在列表底部（折叠态），不无限累积。

### 2.2 执行中计划批准新计划的冲突处理过于模糊

**问题：** 第 7.3 节"执行绑定"部分提到：

> 若旧计划仍为 `executing` 又批准了新计划，应提示用户：「存在执行中的计划，是否先取消或完成后再执行新计划？」（MVP 可简化为：新计划批准后不自动开跑，仅入列表；用户手动点「继续执行」）

这个"简化方案"等于**将冲突决策推给用户**，但面板上没有足够的上下文让用户做决策。用户看到新计划出现在列表下方，可能不理解为什么它没有自动开始执行。

**建议：**
- MVP 应采用明确方案而非模糊的"可简化为"：
  - **方案 A（推荐）：** 批准时检测到旧计划 `executing`，弹出一个**二次确认对话框**："当前有计划正在执行（第 N/M 步），批准新计划后旧计划将暂停。是否继续？"确认后旧计划状态变为 `cancelled`，新计划入列表并等待用户手动触发执行。
  - **方案 B：** 禁止在旧计划 `executing` 时批准新计划，按钮灰化并提示"请先完成或取消当前计划"。
- 无论选哪种，都应在需求中明确。

### 2.3 `pending_plan` 与 `plan` 的职责划分不清

**问题：** 第 8.1 节数据模型中同时定义了 `plan`（当前活跃计划指针）和 `pending_plan`（待审批草稿），但存在以下模糊：

- `pending_plan` 的生命周期是什么？批准后它变成了 `plan`？还是变成 `display_plans` 的一条？
- 如果 `plan` 指向"当前应执行的计划"，那 `plan` 和 `display_plans` 最后一条的关系是什么？是否可能不同步？
- 拒绝后 `pending_plan` 是清空还是保留？需求说"不保留被拒稿在展示列表"，但未说明 `pending_plan` 字段本身。

**建议：**
- 明确数据流转：
  - `awaiting_approval` 期间：`pending_plan` 指向待审稿，`plan` 保持审批前状态（指向当前活跃计划或 null），`display_plans` 不变。
  - 批准后：`pending_plan` 清空，待审稿合并入 `display_plans`，`plan` 指向新计划。
  - 拒绝后：`pending_plan` 清空，`plan` 和 `display_plans` 保持不变。
- 此流转应在需求中以状态机形式呈现，而非散落在多个小节中。

### 2.4 验收标准缺少边界场景

**问题：** 第 11 节验收标准缺少以下重要边界场景：

| 缺失场景 | 为什么重要 |
|----------|-----------|
| 文件预览期间有新计划进入 `awaiting_approval` | 用户可能长时间查看文件而错过审批（见 1.1） |
| 同一会话内多次拒绝后再次提交 | 验证 `pending_plan` 正确替换，旧被拒稿不残留 |
| `drafting` → 直接 `awaiting_approval` 的过渡 | 验证 Spin 消失后审批视图即时出现 |
| 会话恢复时 `display_plans` 为空但 `plan.status === 'completed'` | 验证迁移逻辑（8.2）正确填充 `display_plans` |
| 旧计划 `executing` 时批准新计划 | 验证冲突处理行为（见 2.2） |
| `plan_abort` 与 `awaiting_approval` 同时存在 | 两个特殊状态优先级未定义 |

**建议：**
- 补充以上边界场景的验收标准。

## 三、与现有实现的一致性问题

### 3.1 与引用文件需求的布局冲突

**问题：** `referenced-files-requirement.md` 第 3.3.1 节将上半区定义为"仅占位符，不放其他内容"：

> 上半区域：占位区（仅显示占位符，不放其他内容）

而本需求将上半区的占位符替换为 PlanPanel，两者描述直接冲突。

**建议：**
- 需要更新 `referenced-files-requirement.md`，将上半区描述改为"PlanPanel 或占位符"。
- 或在本需求的附录 A 中列出对关联文档的修订项（目前只列了对主文档的修订，未列对引用文件需求的修订）。

### 3.2 缺少与 ChatView 的具体集成规格

**问题：** 第 9 节说明要移除 `PlanApprovalCard` / `PlanAbortCard` / `PlanResumeBanner` 在聊天区的渲染，但未给出具体的迁移清单：

- `PlanApprovalCard` 移除后，聊天区是否保留一条系统消息提示"计划已提交审批，请在右侧面板查看"？
- 聊天流中与计划相关的工具调用消息（如 `plan:write`、`plan:approve`）如何展示？是正常显示工具调用卡片，还是隐藏？
- Agent 在探索期的只读工具调用（`read_file`、`grep` 等）是否在聊天区正常展示？

**建议：**
- 补充聊天区迁移的具体行为清单，至少包括：
  - 哪些组件从 ChatView 移除
  - 是否需要替代的系统消息/链接
  - 工具调用消息的展示策略

### 3.3 PlanPanel 的高度分配与引用文件面板的关系未说明

**问题：** 当前 `referenced-files-requirement.md` 定义上下区默认 50%/50%，拖动可调整。PlanPanel 占据上半区后，默认比例是否需要调整？

- 计划审批视图（状态 C）内容较多，50% 高度可能不够展示完整的摘要+步骤+按钮。
- 空态（状态 A）只需一行文案，50% 高度浪费空间。

**建议：**
- 定义**状态相关的默认高度建议**：
  - 状态 A（空态）：上半区最小，下半区最大化（如 20%/80%）。
  - 状态 B（计划展示）：上下均分（50%/50%）。
  - 状态 C（待审批）：上半区最大化（如 70%/30%）。
- 或者采用**自适应高度**：PlanPanel 根据内容自动调整，不需要固定比例（仅设置最小/最大值）。
- MVP 也可以保持 50%/50% 不变，但应在需求中说明这是初始值还是最终设计。

## 四、其他建议

### 4.1 新计划通知机制

当前仅第 9 节提到"可选"的高亮/角标通知。但对于从聊天流迁移到侧面板的核心交互，**通知机制不应是可选的**——否则用户可能完全意识不到有计划待审批。

**建议：** 将通知机制从"可选"提升为"必须"，至少包括：
- PlanPanel 区域出现短暂高亮动画（如边框闪烁 2-3 次后恢复）。
- 如果 PlanPanel 被文件预览隐藏，则在右侧栏边缘或文件预览工具栏显示角标。

### 4.2 多条计划卡片的视觉区分

第 5.2.2 节提到多条计划旧在上、新在下排列，但未定义不同状态卡片的视觉区分策略。当列表中同时存在 `executing`、`completed`、`cancelled` 的计划卡片时，用户需要快速区分。

**建议：**
- 为不同状态定义视觉样式：
  - `executing`：正常样式 + 进度条/步骤指示器
  - `completed`：灰色/淡化样式，标题行带绿色完成图标
  - `cancelled`：灰色/删除线样式，标题行带灰色取消图标

### 4.3 数据模型中 `PlanDisplayEntry` 缺少标题字段

第 8.1 节的 `PlanDisplayEntry` 包含 `planId`、`planFilePath`、`status` 等字段，但缺少**计划标题**。面板渲染时需要解析 Markdown 文件来获取标题，增加了渲染复杂度。

**建议：**
- 在 `PlanDisplayEntry` 中增加 `title: string` 字段，在计划批准/入列表时从文件解析一次并缓存，避免每次渲染面板都解析 Markdown。
