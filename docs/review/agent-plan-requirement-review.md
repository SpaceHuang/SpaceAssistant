## 一、需求合理性评估

### 1.1 核心问题定义合理（符合要求）

"Agent 未充分理解需求就直接修改文件导致不可逆错误"——这是真实痛点，定位准确。

### 1.2 "简单任务"绕过机制不合理，应改为用户主动选择 Plan 模式（需修改）

**问题**：Step 1 复杂度判断中，"简单任务 → 快速执行（不走 Plan）"是整个审批闸门的最大漏洞。

- 如果由 Agent 判断复杂度，存在利益冲突——Agent 可能倾向于将任务归类为"简单"以减少交互摩擦
- 如果由用户判断，用户往往不具备评估任务复杂度的能力（这正是他们需要 Agent 的原因）
- 一旦归类为"简单任务"，整个 Plan Mode 的安全保障形同虚设

**进一步分析**："写操作必经审批，只读操作可直接执行"看似合理，但同样存在问题——普通模式下 Agent 也会写文件（如修 bug、改配置），如果所有写操作都强制走 Plan 流程，日常使用会非常繁琐，用户最终会倾向于全程使用普通模式，Plan Mode 沦为摆设。

**建议**：MVP 取消 Agent 自动判断复杂度，改为**用户主动选择是否进入 Plan 模式**。在消息输入框的发送按钮旁放置一个下拉选项，提供"普通模式"和"Plan 模式"两个选项，默认为普通模式。

这样做的好处：
1. **零歧义**：不存在"谁来判断复杂度"的问题，用户自己决定
2. **零摩擦**：普通模式下 Agent 行为与现有完全一致，不增加任何额外步骤
3. **渐进式采纳**：用户在不确定时可以用普通模式快速试错，在重要任务时主动切到 Plan 模式获得保障
4. **MVP 实现简单**：只需在 UI 层加一个模式选择器，后端根据模式决定是否启用权限隔离和审批流程

### 1.3 Coordinator-Worker 隔离在 MVP 中必要，但文档需明确其核心价值是上下文隔离（需补充）

**评估**：Coordinator-Worker 隔离在 MVP 阶段是必要的，其核心价值不在于多 Worker 并行（那是 V2），而在于**上下文隔离**——执行者（Worker）的执行细节（文件内容、中间结果、工具调用输出等）不占用协调者（Coordinator）的上下文窗口，保证 Coordinator 有足够的可用上下文空间来维持与用户的对话、管理计划状态和做出决策。

如果 MVP 不做隔离，让单个 Agent 同时承担对话+执行，执行过程中产生的大量工具输出会快速填满上下文窗口，导致 Coordinator 丢失早期的用户意图和计划上下文，后续决策质量下降。

**问题**：文档将 Coordinator-Worker 隔离与多 Worker 并行混在一起描述，未突出上下文隔离这个 MVP 核心价值，容易让人误以为这是 V2 功能。

**建议**：
1. 在文档中明确区分：**MVP 保留 Coordinator + 单 Worker 的隔离架构**（核心目标是上下文隔离），**V2 再扩展为多 Worker 并行**
2. 补充上下文隔离的具体实现策略——Coordinator 如何策展上下文（从 Worker 执行结果中提取摘要而非全量传递给 Coordinator）
3. 2.3 节的状态恢复安全隐患仍然存在（Worker 重新初始化后上下文丢失），需补充幂等标记机制

### 1.4 "100% 审批"与"简单任务绕过"自相矛盾（需修改）

**问题**：文档在 3.1 功能 3 要求"不可绕过的审批闸门"，第五节又说"MVP 100% 审批"不做概率审查，但 Step 1 又开了"简单任务绕过"的口子。这三个声明互相矛盾。

采用用户主动选择模式的方案后，矛盾自然消解：
- **普通模式**：无审批，Agent 直接执行（等同于"简单任务快速执行"）
- **Plan 模式**：100% 审批，计划未获批不能执行（等同于"不可绕过的闸门"）
- 模式选择权在用户手中，不存在 Agent 自行判断的漏洞

## 二、逻辑正确性评估

### 2.1 "物理上不可绕过"过度声明（需修改）

**问题**：文档多处声称审批闸门是"物理上不可绕过的"，但 4.4 节的实际实现是 Tool ACL（工具访问控制列表），这是**软件层**的权限控制，不是物理隔离。

- Tool ACL 依赖工具注册的完整性——如果新增工具时忘记加入 ACL，就会产生权限漏洞

**建议**：

- 将"物理上不可绕过"改为"系统层强制执行"，降低承诺
- 补充工具注册时的 ACL 审查流程

### 2.2 权限表引用了不存在的工具（需修改）

**问题**：4.4 节权限表中列出了 `bash`（写操作）和 `execute_code` 作为 Plan Mode 下禁止的工具，但 SpaceAssistant 当前版本**不存在这两个工具**。实际工具集为：`read_file`、`list_directory`、`edit_file`、`write_file`、`grep`、`run_script`。

权限表应基于实际工具集重新编写，而非假设一个通用 SDK 的工具集。特别需要明确的是 `run_script`——它在 Plan Mode 下应如何处理：

- `run_script` 可执行任意代码，理论上可以间接写入文件，是权限隔离的潜在绕过点
- 但 `run_script` 也常用于环境探测（如运行测试、检查依赖版本），探索期完全禁用会影响计划质量

**建议**：权限表按实际工具集重新编写，并明确 `run_script` 在 Plan Mode 下的策略（建议：Plan Mode 下禁止，如需环境信息通过 `read_file` + `list_directory` + `grep` 获取）。

### 2.3 状态恢复逻辑存在安全隐患（需修改）

**问题**：4.3 节恢复流程中，"Coordinator 重新加载计划文件和 `current-session.json`，重建 Worker 上下文"存在以下问题：

- `current-session.json` 是明文 JSON，用户或外部程序可能修改它
- 恢复时没有校验状态文件与计划文件的一致性（如 `current_step_index` 超出 `steps_total`）
- Worker 重新初始化后，其执行上下文（如内存中的变量、已打开的文件句柄）会丢失，可能导致重复执行已完成的步骤

**建议**：

- 恢复时增加状态文件校验（检查 step_index 范围、文件存在性、状态一致性）
- 每个步骤完成后应记录**幂等标记**（如已修改的文件哈希），恢复时通过标记避免重复执行
- 补充状态文件被篡改时的处理策略（如重置到最近的安全检查点）

### 2.4 计划模板自动匹配逻辑缺失（需补充）

**问题**：4.5 节提到"Agent 在 Step 1 根据用户意图自动匹配模板"，但未定义匹配规则。

- 匹配错误会导致计划结构与任务不匹配（如将"代码重构"匹配为 `content-creation` 模板）
- 未提供模板匹配失败的降级策略

**建议**：补充模板匹配的规则说明（如基于用户输入的关键词/意图分类），并明确匹配失败时使用通用模板的降级策略。

## 三、业务流程完整性评估

### 3.1 缺失：用户中途变更需求（重要缺失）

**问题**：计划获批后、执行完成前，用户可能想修改需求（如"加一个额外功能"或"这个步骤不要了"）。当前流程只有"执行期异常上报"——Agent 主动暂停，没有"用户主动变更"的入口。

**影响**：用户要么等当前计划执行完再提新需求（浪费已执行的工作），要么直接放弃当前计划（丢失进度），要么在聊天中提出但 Agent 无视（体验断裂）。

**建议**：增加"计划变更"流程——用户可在执行期提出变更，Agent 评估变更影响并决定是否需要重新进入 Plan Mode（生成新版本计划）还是可就地调整当前计划。

### 3.2 缺失：计划取消与部分回滚（重要缺失）

**问题**：执行到第 3/5 步时，用户想取消。当前文档只在恢复流程中提到"放弃并重新开始"，但未定义：

- 已完成的步骤产生的文件修改如何处理（保留？回滚？）
- 取消后 `.agent/` 下的状态文件如何清理
- 取消操作本身是否需要确认（防止误触）

**建议**：定义取消流程，至少包含：
1. 确认对话框（"已执行 3/5 步，取消后已修改的文件将保留"）
2. 取消后的状态标记（`status: cancelled`，记录已完成步骤）
3. 可选的回滚能力（基于文件备份或 git stash）

### 3.3 缺失：计划过期检测（需补充）

**问题**：计划批准后，如果用户长时间未执行（如隔了几天），或代码库在外部被修改（如同事合并了 PR），计划可能已过时。当前文档没有过期检测机制。

**建议**：在恢复执行时增加环境校验——对比计划批准时的环境快照（如 git HEAD、关键文件哈希）与当前环境，如果差异显著则提醒用户计划可能需要更新。

### 3.4 缺失：多计划并行（需明确立场）

**问题**：用户可能想同时推进多个独立任务（如"修 bug A"和"加功能 B"）。文档未说明是否支持多计划并行，`current-session.json` 的结构也只支持单个 `active_plan`。

**建议**：在 MVP 边界中明确声明"不支持多计划并行，同一时间只能有一个活跃计划"，避免用户期望与实现不一致。如果后续需要支持，`current-session.json` 的结构需要重新设计。

### 3.5 缺失：审批超时处理（需补充）

**问题**：计划提交审批后，用户可能长时间不响应（离开电脑、忘记审批）。当前没有超时处理机制。

**建议**：定义审批超时策略（如 24 小时未审批则标记为 `stale`），以及用户返回后的提示逻辑。

## 四、使用场景覆盖评估

### 4.1 已覆盖场景

| 场景 | 覆盖情况 |
|------|----------|
| 首次制定计划 → 审批 → 执行 | 覆盖 |
| 计划被拒绝 → 迭代修改 → 重新审批 | 覆盖 |
| 执行中遇到异常 → 暂停上报 | 覆盖 |
| 会话中断 → 恢复执行 | 覆盖 |

### 4.2 未覆盖场景

| 遗漏场景 | 描述 | 严重程度 |
|----------|------|----------|
| 用户中途变更需求 | 执行期用户想修改计划内容 | 高 |
| 计划取消与部分回滚 | 用户想中途放弃计划 | 高 |
| 计划过期/环境变更 | 代码库在计划等待期间被外部修改 | 中 |
| 审批超时 | 用户长时间未审批 | 中 |
| 探索期发现无需计划 | Agent 探索后认为任务过于简单或不可行，不生成计划 | 中 |
| 计划执行结果与预期不符 | 所有步骤执行完成但结果不满足验收标准 | 中 |
| 并发操作冲突 | 用户在 Agent 执行期间手动修改了计划涉及的文件 | 中 |
| 多计划管理 | 用户想查看历史计划或切换活跃计划 | 低 |
| 计划文件被外部删除 | `.agent/plans/` 下的文件被用户或工具意外删除 | 低 |

### 4.3 "探索期发现无需计划"场景需补充

**问题**：当前流程假设 Step 2 一定会生成计划文件。但 Agent 探索后可能发现：
- 任务不可行（如依赖的服务不存在）
- 任务实际上非常简单，不需要计划
- 信息不足，无法制定计划

当前流程没有"探索期终止"的出口。

**建议**：增加 Step 2 的退出条件——Agent 可在探索后选择"终止并报告"而非"生成计划"。此时向用户展示探索结果和建议，而非进入审批流程。

## 五、产品上下文适配度评估

### 5.1 整体偏 SDK 通用设计，未适配 Electron 应用（需修改）

**问题**：文档整体风格是"通用 Agent SDK 设计"——伪代码用 Python、审批界面用终端卡片、存储规范假设文件系统直接操作。但 SpaceAssistant 是一个 **Electron 桌面应用**，有 GUI 和 IPC 架构。

具体不匹配：

| 文档描述 | SpaceAssistant 实际场景 |
|----------|----------------------|
| 审批界面用终端渲染卡片 | 应为 Electron 窗口内的 React 组件（Ant Design Modal/Drawer） |
| 伪代码用 Python | 应为 TypeScript，权限控制在主进程 IPC 层 |
| 状态存储用文件系统 | SpaceAssistant 已有 JSON 数据库（`spaceassistant-data.json`），应复用 |
| "打开计划文件供用户编辑" | Electron 应用中应提供内置编辑器或调用外部编辑器 |
| 权限控制在 SDK Tool ACL 层 | 应在主进程的 IPC handler 层实现权限拦截 |

**建议**：增加一节"SpaceAssistant 适配说明"，明确：
1. 审批界面为 React 组件（Ant Design），嵌入聊天界面或详情面板
2. 权限控制在主进程 IPC 层（`electron/appIpc.ts`），而非渲染进程
3. 状态存储复用现有 JSON 数据库，而非独立的 `.agent/state/` 文件系统
4. 计划文件可存储在 `.agent/plans/`（隔离性好），但索引和状态应纳入数据库管理

### 5.2 与现有功能的关系未定义（需补充）

**问题**：文档未说明 Plan Mode 与 SpaceAssistant 现有功能的关系：

- Plan Mode 与**会话管理**的关系：Plan 是会话级还是跨会话的？
- Plan Mode 与**工具调用确认机制**的关系：获批后的执行步骤是否仍需逐个确认？
- Plan Mode 与**引用文件列表**的关系：Plan Mode 下的文件操作是否计入引用列表？
- Plan Mode 与**配置管理**的关系：用户是否可以全局关闭 Plan Mode？

**建议**：补充"与现有功能的关系"章节，参考已有的 Skills/Tools 需求文档中的做法。

### 5.3 计划文件存储位置与项目污染问题（需重新评估）

**问题**：文档将 `.agent/` 放在"工作目录"下，理由是"对用户透明，默认隐藏"。但：

- SpaceAssistant 的"工作目录"是用户在配置中指定的项目目录，不是 SpaceAssistant 自身的数据目录
- 在用户项目中创建 `.agent/` 目录**本身就是一种污染**——即使加入 `.gitignore`，也会影响项目结构
- 不同 Agent 工具（Claude Code、Cursor、Copilot）可能各自创建自己的隐藏目录，造成混乱

**建议**：

- 方案 A：将计划文件存储在 Electron 的 `userData` 目录下（如 `%APPDATA%/SpaceAssistant/plans/<project-hash>/`），与项目目录完全隔离
- 方案 B：如果确实需要在项目目录下存储（便于团队共享），则使用 `.spaceassistant/` 而非 `.agent/`，与产品品牌一致

## 六、需求细节问题

### 6.1 计划文件版本管理方式脆弱

**问题**：使用 `feature-a.v2.md`、`feature-a.v3.md` 的文件命名做版本管理，迭代次数多时目录会变得混乱，且 `index.json` 的 `latest` 指针需要手动维护。

**建议**：改用单文件 + 内部版本号（Frontmatter `version: N`）的方式，每次迭代覆盖同一文件（或保留历史版本但通过 `index.json` 的 `versions` 数组管理，而非文件名编码版本号）。

### 6.2 "计划零占位符"验收标准难以执行

**问题**：验收标准第 6 条"计划零占位符——不允许出现 TODO、TBD、待确认"意图很好，但执行上有问题：

- 谁来检查？Agent 自己检查自己生成的计划，等于"靠 Prompt 约束"，与文档"不靠 Prompt 约束"的原则矛盾
- 如果用正则扫描，Agent 可能用同义替换绕过（如"待定"→"后续确定"）
- 某些场景下确实存在需要用户确认的未知信息（如 API 返回格式），强制不允许占位符可能导致 Agent 编造信息

**建议**：

- 改为"关键字段零占位符"——目标、方案、执行步骤这三个核心字段不允许模糊，其他字段（如风险点）可以标注"需执行时确认"
- 在审批界面中高亮标注含占位符的字段，提醒用户注意

### 6.3 审批卡片的信息设计需适配桌面应用

**问题**：4.2 节的审批卡片设计基于终端场景，信息密度过高。在 SpaceAssistant 的聊天界面中，需要考虑：

- 卡片在聊天流中的位置和尺寸
- 与其他消息（如 Agent 的探索过程文字）的视觉关系
- "批准前修改"在桌面应用中的具体交互形式

**建议**：审批界面应设计为聊天流中的嵌入式卡片或侧边面板，而非终端风格的 ASCII 框。"批准前修改"可提供两种方式：1) 在卡片中直接编辑关键字段；2) 在聊天中输入修改意见，Agent 据此更新计划。

## 七、评审结论与建议

### 7.1 总体结论

文档定义的核心问题准确、MVP 边界意识强、结构化程度高。但在以下方面存在显著问题：

1. **MVP 范围 creep**：3 个场景化模板等功能的复杂度超出"单线流程"的 MVP 定位；Coordinator-Worker 隔离需保留但应明确其核心价值是上下文隔离而非多 Worker 并行
2. **流程完整性不足**：缺失用户变更需求、计划取消、过期检测等关键分支场景
3. **产品适配度低**：文档按通用 SDK 设计撰写，未适配 SpaceAssistant 的 Electron 应用架构
4. **"简单任务绕过"与"100%审批"自相矛盾**：应改为用户主动选择模式（普通/Plan），消除矛盾

### 7.2 需修改的内容

| 优先级 | 内容 | 说明 |
|--------|------|------|
| **高** | 取消 Agent 自动判断复杂度，改为用户主动选择模式 | 在发送按钮旁加下拉选项（普通模式/Plan 模式），消除自动判断的安全漏洞和体验摩擦 |
| **高** | 明确 Coordinator-Worker 隔离的 MVP 核心价值是上下文隔离 | 区分于 V2 多 Worker 并行，补充上下文策展策略 |
| **高** | 补充用户中途变更需求的流程 | 避免执行期用户与 Agent 的交互断裂 |
| **高** | 补充计划取消与部分回滚流程 | 用户放弃计划时不应丢失进度信息 |
| **中** | 增加 SpaceAssistant 适配说明 | 审批界面、权限控制、状态存储的 Electron/React 实现方案 |
| **中** | 补充与现有功能的关系定义 | 会话、工具确认、引用文件、配置管理 |
| **中** | 权限表按实际工具集重新编写 | 当前无 bash/execute_code，需明确 run_script 在 Plan Mode 下的策略 |
| **中** | 补充探索期终止出口 | Agent 可选择"报告发现"而非"生成计划" |
| **低** | 计划文件版本管理优化 | 单文件 + 内部版本号，而非文件名编码版本 |
| **低** | 补充审批超时和计划过期检测 | 提升长时间场景下的鲁棒性 |
| **低** | 明确"多计划并行"不在 MVP 范围 | 避免用户期望与实现不一致 |

### 7.3 核心建议

**MVP 应该更小、更紧**。当前文档虽然定义了清晰的 MVP 边界（第五节），但功能清单中仍混入了 V2 级别的复杂度（Coordinator-Worker、场景化模板、文件系统状态机）。建议回归文档自身的设计原则——"先跑通单线流程"：

> **精简版 MVP = 用户主动选择模式 + Coordinator-Worker 隔离（单 Worker） + Plan 模式下审批闸门 + 计划文件 + 审批 UI + 状态持久化**

具体调整：
- 去掉 Agent 自动判断复杂度（Step 1），改为发送按钮旁的下拉选项让用户选择普通模式或 Plan 模式
- 保留 Coordinator-Worker 隔离（上下文隔离是 MVP 核心需求），但文档需明确 MVP 是单 Worker，V2 再扩展多 Worker 并行
- 补充上下文策展策略（Coordinator 如何从 Worker 结果中提取摘要）
- 去掉 3 个场景化模板，MVP 用一个通用模板
- 先把核心闸门跑通，再叠加高级能力
