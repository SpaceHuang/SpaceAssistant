# Claude Code 文件写入安全机制产品需求总结

> 本文档从产品需求视角，总结 Claude Code 文件写入工具的安全机制设计。

---

## 一、背景与目标

大模型存在"幻觉"问题，可能在文件编辑时产生误操作导致文件损坏。Claude Code 设计了多层次的文件修改安全机制，核心目标是：

- **防止文件损坏**：确保任何文件修改都可追溯、可恢复
- **避免数据丢失**：在误操作或模型幻觉时，用户能快速回滚到正确状态
- **并发安全**：检测外部修改与内部修改的冲突

---

## 二、核心功能特性

### 1. 前置读取验证

**产品需求**：模型必须先理解文件内容，才能进行修改。

**实现方案**：
- Edit/Write 工具强制要求文件在被修改前已通过 Read 工具读取
- 使用 `FileStateCache` 缓存已读取文件的状态（内容、时间戳、读取范围）
- 部分读取（partial view）会标记状态，要求完整读取后才允许写入

**业务价值**：防止模型在未理解文件上下文的情况下进行幻觉式编辑。

---

### 2. 并发修改检测

**产品需求**：检测并防止外部程序（如 IDE、云同步）在模型编辑期间修改文件。

**实现方案**：
- 写入前比较文件修改时间戳（mtime）
- 若时间戳在读取后发生变化，触发内容比较回退机制
- Windows 系统因云同步、杀毒软件可能导致时间戳不稳定，直接比较文件内容

**错误处理**：抛出 `FILE_UNEXPECTEDLY_MODIFIED_ERROR`，要求用户重新读取后再操作。

**业务价值**：避免外部修改被静默覆盖，保护用户并行工作流。

---

### 3. 文件历史备份

**产品需求**：每次修改前自动创建文件快照，支持历史追溯。

**实现方案**：
- 修改前自动创建备份文件，存储在 `~/.claude/file-history/{sessionId}/`
- 备份文件使用 SHA256 哈希命名，避免路径冲突
- 智能策略：仅当文件实际发生变化时才创建新备份
- 最多保留 100 个快照，自动清理旧版本

**备份结构**：
```
backupDir/
  ├── abc123@v1    # 第一次修改的备份
  ├── abc123@v2    # 第二次修改的备份
  └── def456@v1    # 另一个文件的备份
```

**业务价值**：提供细粒度的版本历史，用户可回滚到任意修改节点。

---

### 4. 原子性写入

**产品需求**：确保文件写入过程不会因中断导致文件损坏。

**实现方案**：
- 写入临时文件（`{targetPath}.tmp.{pid}.{timestamp}`）
- 对临时文件执行 flush 操作
- 使用 `fs.renameSync` 原子性重命名（POSIX 系统保证原子性）
- 失败时回退到普通写入并清理临时文件
- 保留符号链接目标
- 恢复原始文件权限

**业务价值**：即使写入过程中断，也不会产生不完整的临时文件。

---

### 5. 版本回滚能力

**产品需求**：用户可撤销任意历史修改，恢复到之前的文件状态。

**实现方案**：
- `/rewind` 命令按消息 ID 回滚到指定节点
- `applySnapshot` 函数将快照应用到文件系统
- `restoreBackup` 函数恢复单个文件
- `fileHistoryHasAnyChanges` 轻量级检查回滚是否会是空操作

**业务价值**：提供"后悔药"，模型操作失误时用户可一键恢复。

---

### 6. 会话恢复能力

**产品需求**：跨会话保留文件历史，重启程序后仍可回滚。

**实现方案**：
- 快照持久化到会话存储（`sessionStorage.ts`）
- 新会话启动时，从旧会话复制备份文件
- 优先使用硬链接，失败则复制文件
- 状态重建恢复完整的文件历史

**业务价值**：用户关闭终端后仍可继续之前的回滚操作。

---

### 7. 错误码体系

**产品需求**：提供明确的错误提示，帮助用户和模型快速定位问题。

| 错误码 | 场景 | 用户提示 |
|--------|------|---------|
| 0 | Secrets 检测 | 禁止写入包含密钥的内容 |
| 1 | 无内容修改 | old_string 与 new_string 相同 |
| 2 | 路径禁止 | 文件在禁止目录 |
| 4 | 文件不存在 | 必须先读取文件 |
| 6 | 文件未读取 | 必须先读取文件 |
| 7 | 文件已修改 | 重新读取后再写入 |
| 8 | 匹配失败 | old_string 未找到 |
| 9 | 多匹配冲突 | 使用 replace_all 或提供更多上下文 |
| 10 | 文件过大 | 超过 1GiB 限制 |

**业务价值**：结构化的错误反馈，便于模型自我纠正。

## 三、安全机制分层

### 预防层（阻止危险操作）

```
┌─────────────────────────────────────────────┐
│  1. 前置读取验证                             │
│     ├─ Edit/Write 必须先 Read               │
│     └─ readFileState 缓存验证                │
│                                              │
│  2. 并发修改检测                             │
│     ├─ 文件时间戳对比                        │
│     └─ Windows 内容比较回退                  │
│                                              │
│  3. 权限与路径检查                           │
│     ├─ Deny/Allow 规则                      │
│     └─ 危险路径保护（CON, PRN, AUX 等）      │
└─────────────────────────────────────────────┘
```

### 恢复层（提供回滚能力）

```
┌─────────────────────────────────────────────┐
│  1. 文件历史备份                             │
│     ├─ 修改前自动创建备份                     │
│     └─ 版本号管理（最多 100 个快照）          │
│                                              │
│  2. 原子性写入                               │
│     ├─ 临时文件 + rename                    │
│     └─ flush 保证                           │
│                                              │
│  3. 版本回滚                                 │
│     ├─ /rewind 命令                         │
│     └─ 按消息 ID 恢复                        │
│                                              │
│  4. 会话恢复                                 │
│     ├─ 快照持久化                           │
│     └─ 跨会话备份复制                        │
└─────────────────────────────────────────────┘
```

---

## 四、配置与控制

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `fileCheckpointingEnabled` | true | 全局开关 |
| `CLAUDE_CODE_DISABLE_FILE_CHECKPOINTING` | - | 环境变量禁用 |
| MAX_SNAPSHOTS | 100 | 最大快照数 |
| FileStateCache 大小 | 100 条 / 25MB | LRU 缓存限制 |

---

## 五、产品价值总结

| 能力 | 用户价值 |
|------|----------|
| 前置读取验证 | 减少模型编辑错误，提高成功率 |
| 并发检测 | 保护并行工作流，避免冲突 |
| 原子写入 | 消除写入损坏风险 |
| 版本备份 | 完整的修改历史，随时可回滚 |
| 会话恢复 | 跨会话连续性，不丢失工作 |
| 错误码体系 | 快速定位问题，提升体验 |

Claude Code 通过"预防 + 恢复"的双层架构，在利用大模型提升开发效率的同时，有效控制了文件操作风险，让用户能够放心使用 AI 辅助编辑。
