# FileWriteTool 产品功能文档

> 本文档从产品功能需求视角，介绍 FileWriteTool（Write 工具）的业务定位、核心流程、使用方法与约束限制。

---

## 一、产品定位

### 1.1 工具定位

FileWriteTool（简称 Write）是 Claude Code 中用于**创建新文件或完全覆盖现有文件**的核心工具。

**典型使用场景**：
- 创建全新的代码文件
- 对现有文件进行完整内容重写（非增量修改）

**与 Edit 工具的区别**：

| 维度 | Write | Edit |
|------|-------|------|
| 操作方式 | 完整替换 | 增量编辑 |
| 传输内容 | 完整文件内容 | 仅 diff 差异 |
| 适用场景 | 文件创建、完全重写 | 局部修改 |
| 性能影响 | 较大（全量传输） | 较小（增量传输） |

### 1.2 核心特性

| 特性 | 说明 |
|------|------|
| 原子性写入 | 通过时间戳+内容双重验证，防止并发冲突 |
| 多层安全验证 | Deny/Ask/Allow 三级权限体系，保护危险文件 |
| 强制 LF 换行 | 统一跨平台换行符标准，避免脚本损坏 |
| LSP 集成 | 自动通知语言服务器，保持代码诊断同步 |
| 历史备份 | 支持文件修改历史回溯（可配置） |

## 二、业务流程

### 2.1 完整执行流程

```
用户触发 Write
     │
     ▼
┌─────────────────────────────┐
│ 1. 路径处理与验证            │
│   - 路径展开（~、相对路径）   │
│   - 特殊路径安全检查         │
└─────────────┬───────────────┘
              ▼
┌─────────────────────────────┐
│ 2. 输入验证                  │
│   - Secrets 检测（密钥泄露） │
│   - Deny 规则匹配           │
│   - 文件读取状态验证         │
│   - 文件修改时间戳验证       │
└─────────────┬───────────────┘
              ▼
┌─────────────────────────────┐
│ 3. 权限检查                  │
│   - 内部可编辑路径检查       │
│   - .claude 文件夹特殊处理   │
│   - 危险文件安全检查         │
│   - Ask/Allow 规则匹配      │
└─────────────┬───────────────┘
              ▼
┌─────────────────────────────┐
│ 4. 前置处理                  │
│   - Skill 发现与激活         │
│   - 诊断跟踪记录             │
│   - 目录创建（如需要）       │
│   - 文件历史备份             │
└─────────────┬───────────────┘
              ▼
┌─────────────────────────────┐
│ 5. 原子性写入                │
│   - 读取当前文件状态         │
│   - 检测并发修改冲突         │
│   - 写入内容（强制 LF）      │
│   - 更新文件状态缓存         │
└─────────────┬───────────────┘
              ▼
┌─────────────────────────────┐
│ 6. 后置通知                  │
│   - LSP 服务器通知           │

              ▼

```

### 2.2 关键阶段说明

#### 阶段 1：路径处理与验证
- 将 `~` 和相对路径展开为绝对路径
- Windows UNC 路径特殊处理

#### 阶段 2：输入验证
- 检查文件内容是否包含敏感信息（Secrets）
- 验证文件是否在 Deny 列表中
- **重要**：验证文件是否已被读取过

#### 阶段 3：权限检查
- 检查文件是否属于内部可编辑路径（plan files、scratchpad）
- `.claude` 文件夹有特殊允许规则
- 危险文件（.env、配置文件、.git 等）需要显式确认
- 根据 Ask/Allow 规则决定是否需要用户授权

#### 阶段 4：前置处理
- 发现并激活相关技能目录
- 记录诊断状态
- 自动创建父目录
- 创建文件历史备份（如果启用）

#### 阶段 5：原子性写入
- 写入前再次验证文件未被修改
- 强制使用 LF 换行符写入
- 更新内存中的文件状态缓存

#### 阶段 6：后置通知
- 通知 LSP 服务器文件已变更/保存
- 通知 VSCode 文件已更新
- 清除旧诊断，显示新诊断

## 三、使用方法

### 3.1 接口定义

**输入参数**：

| 参数 | 类型 | 必需 | 说明 |
|------|------|------|------|
| `file_path` | string | 是 | 文件绝对路径（必须使用绝对路径） |
| `content` | string | 是 | 要写入的完整文件内容 |

**输出结果**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `type` | 'create' \| 'update' | 操作类型：新建或更新 |
| `filePath` | string | 实际写入的文件路径 |
| `content` | string | 写入的内容 |
| `structuredPatch` | array | 内容变更的 diff 补丁 |
| `originalFile` | string \| null | 原文件内容（新建时为 null） |
| `gitDiff` | object | Git diff 信息（可选） |

### 3.2 使用示例

**场景 1：创建新文件**

```
1. 直接使用 Write 工具写入新文件路径
2. 系统自动创建父目录
3. 返回 type: 'create'
```

**场景 2：更新现有文件**

```
1. 使用 Read 工具读取现有文件
2. 修改内容后使用 Write 工具重写
3. 系统验证读取时间戳
4. 返回 type: 'update' 和 originalFile
```

### 3.3 调用示例

```typescript
Write({
  file_path: "/Users/name/project/src/utils.ts",
  content: `export function helper() {
  return 'helper'
}
`
})
```

## 四、约束限制

### 4.1 硬性约束

| 约束 | 说明 | 错误码 |
|------|------|--------|
| **必须先读取** | 现有文件必须先用 Read 工具读取 | 2 |
| **必须使用绝对路径** | 不支持相对路径 | - |
| **文件未被修改** | 写入时文件不能自上次读取后被外部修改 | 3 |

### 4.2 安全约束

| 约束 | 说明 | 错误码 |
|------|------|--------|
| **Secrets 检测** | 禁止写入包含密钥、凭据等敏感信息 | 0 |
| **Deny 规则** | 文件在用户配置的拒绝列表中 | 1 |
| **危险文件保护** | .env、.git、配置文件等需要特殊权限 | - |

### 4.3 内容约束

| 约束 | 说明 |
|------|------|
| **不主动创建文档** | 不自动创建 .md 或 README 文件（除非用户明确要求） |
| **不主动使用 Emoji** | 不在文件中写入 Emoji（除非用户明确要求） |
| **强制 LF 换行** | 所有文件统一使用 LF 换行符（跨平台兼容） |

### 4.4 约束背后的产品逻辑

| 约束 | 产品目的 |
|------|----------|
| 必须先读取 | 确保 AI 操作基于最新文件状态，避免覆盖他人修改 |
| 绝对路径要求 | 明确操作目标，防止路径歧义 |
| 文件未修改验证 | 防止并发冲突，保护用户工作成果 |
| Secrets 检测 | 防止 AI 意外泄露密钥或凭据 |
| 危险文件保护 | 防止误操作破坏系统关键文件 |
| 文档/Emoji 限制 | 保持代码专业性，避免生成不必要的内容 |

## 五、权限体系

### 5.1 权限层级

```
┌─────────────────────────────────────────┐
│            权限检查优先级                 │
├─────────────────────────────────────────┤
│  1. Deny 规则    （最高优先级，直接拒绝）  │
│  2. 内部可编辑路径  （plan files 等）      │
│  3. .claude 文件夹  （特殊允许）           │
│  4. 安全检查       （危险文件需确认）      │
│  5. Ask 规则      （需用户授权）           │
│  6. Allow 规则    （明确允许）             │
│  7. acceptEdits 模式（开发目录自动允许）   │
│  8. 默认行为      （询问用户）             │
└─────────────────────────────────────────┘
```

### 5.2 危险文件类型

以下文件类型在写入前需要特殊安全验证：
- 敏感配置文件（.env、credentials、secrets）
- Git 目录（.git/）
- IDE 配置（.vscode/、.idea/）
- Claude 配置（settings.json、hooks/）
- Windows 系统特殊设备名（CON、PRN、AUX、NUL 等）

## 六、错误处理

### 6.1 常见错误

| 错误信息 | 原因 | 解决方案 |
|----------|------|----------|
| `File has not been read yet` | 现有文件未先读取 | 先使用 Read 工具 |
| `File has been modified since read` | 文件被外部程序修改 | 重新读取后再写入 |
| `Permission to edit has been denied` | 文件在拒绝列表中 | 检查权限配置 |
| `File contains secrets` | 内容包含敏感信息 | 检查并清理内容 |

### 6.2 并发冲突处理

系统通过双重验证机制防止并发冲突：
1. **时间戳验证**：比较文件修改时间
2. **内容验证**：比较文件实际内容（Windows 环境下时间戳可能不准确）

当检测到冲突时，写入操作会被拒绝，提示用户重新读取文件。
