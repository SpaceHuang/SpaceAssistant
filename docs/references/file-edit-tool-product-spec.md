# FileEditTool 产品功能规格说明

> 从产品需求视角，系统性梳理 FileEditTool 的功能定位、业务流程、使用方法与约束条件

---

## 一、功能定位

### 1.1 产品价值

**FileEditTool** 是 Claude Code 的核心增量文件编辑工具，用于执行精确的字符串替换操作。其核心价值在于：

| 价值维度 | 具体体现 |
|---------|---------|
| **高效性** | 增量编辑，仅传输变更数据，降低网络开销 |
| **精确性** | 基于字符串匹配的精准替换，避免误操作 |
| **安全性** | 多层验证机制，防止非法或危险操作 |
| **兼容性** | 智能处理引号规范化、换行符保留等细节 |

### 1.2 与 Write 工具的分工

| 场景 | 推荐工具 | 原因 |
|-----|---------|------|
| 修改现有文件的部分内容 | **Edit** | 增量编辑，保留文件原有特性 |
| 创建新文件或完全重写 | **Write** | 完整内容覆盖 |

### 2.2 关键决策点

| 决策点 | 判断条件 | 处理方式 |
|-------|---------|---------|
| 文件是否存在 | `fileContent === null` | 若 `old_string` 为空则创建，否则报错 |
| 字符串是否唯一 | 匹配次数 > 1 且 `replace_all=false` | 报错并提示增加上下文或使用 `replace_all` |
| 文件是否被修改 | 时间戳 > 读取时间戳 | 报错并提示重新读取 |
| 是否为 Notebook | 文件后缀为 `.ipynb` | 拦截并引导使用 NotebookEdit |

## 三、使用方法

### 3.1 基本调用格式

```typescript
Edit({
  file_path: string,      // 文件绝对路径（必需）
  old_string: string,     // 待替换文本（必需）
  new_string: string,     // 新文本（必需，需与 old_string 不同）
  replace_all: boolean    // 是否全局替换（可选，默认 false）
})
```

### 3.2 典型应用场景

| 场景 | 示例 |
|-----|------|
| **单次替换** | 修改函数实现、更新配置值 |
| **全局替换** | 重命名变量、批量修改常量 |
| **创建新文件** | `old_string: ""` + `new_string: "..."` |
| **删除内容** | `new_string: ""` |

### 3.3 输入规范

1. **路径要求**：必须提供绝对路径
2. **字符串要求**：
   - `old_string` 必须在文件中存在且唯一（除非使用 `replace_all`）
   - `old_string` 与 `new_string` 不能相同
3. **缩进保持**：从 Read 工具输出复制时，需保留精确缩进格式

## 四、约束条件

### 4.1 硬性约束

| 约束项 | 限制内容 | 原因 |
|-------|---------|------|
| **前置读取** | 必须先用 Read 工具读取文件 | 确保编辑基于最新内容，避免冲突 |
| **字符串唯一性** | `old_string` 在文件中必须唯一 | 防止误修改非目标内容 |
| **文件大小** | 最大 1 GiB | 防止大文件导致内存溢出 |
| **Notebook 拦截** | 禁止编辑 `.ipynb` 文件 | Notebook 需要特殊处理逻辑 |
| **Secrets 保护** | 禁止写入敏感信息 | 安全合规要求 |

### 4.2 行为约束

| 约束项 | 行为描述 |
|-------|---------|
| **引号规范化** | 自动处理弯引号/直引号差异，保留原文件风格 |
| **换行符保留** | 保持原文件换行符格式（LF/CRLF） |
| **原子性保障** | 写入前再次验证文件状态，检测并发修改 |
| **历史追踪** | 自动备份修改历史，支持撤销操作 |

### 4.3 错误码体系

| 错误码 | 错误类型 | 用户提示 |
|-------|---------|---------|
| 0 | 安全违规 | 检测到敏感信息 |
| 1 | 无效操作 | 新旧字符串相同，无需修改 |
| 2 | 权限不足 | 文件位于禁止目录 |
| 4 | 文件不存在 | 文件未找到，建议检查路径 |
| 5 | 类型不匹配 | 请使用 NotebookEdit 工具 |
| 6 | 前置条件缺失 | 文件尚未读取，请先使用 Read 工具 |
| 7 | 并发冲突 | 文件已被修改，请重新读取 |
| 8 | 匹配失败 | 未找到待替换的字符串 |
| 9 | 多匹配歧义 | 找到多个匹配，请提供更精确的上下文或使用 replace_all |
| 10 | 文件过大 | 文件超出大小限制（最大 1 GiB） |

## 五、核心特性

### 5.1 智能匹配机制

**引号自动规范化**：
- 支持弯引号（`"` `'`）与直引号（`"` `'`）的自动转换
- 匹配时进行规范化处理
- 替换时保留原文件的引号风格

**智能提示**：
- 文件不存在时，自动推荐相似文件名或工作目录下的路径
- 多匹配时，明确提示匹配数量和解决方案

### 5.2 安全保障体系

| 安全层级 | 保护内容 |
|---------|---------|
| **内容安全** | 敏感信息检测（Secrets） |
| **路径安全** | Deny 规则拦截、UNC 路径检查 |
| **并发安全** | 时间戳+内容双重验证 |
| **历史安全** | 自动备份、支持回滚 |

---

## 六、使用注意事项

1. **优先编辑原则**：除非明确需要创建新文件，否则应优先使用 Edit 工具修改现有文件
2. **上下文完整性**：确保 `old_string` 包含足够上下文以保证唯一性
3. **Emoji 使用**：除非用户明确要求，否则不要在文件中添加 Emoji
4. **批量操作**：全局替换前建议先预览变更，确认无误后再执行
